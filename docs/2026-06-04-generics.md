# Generics 設計 (2026-06-04)

bidirectional checker 上に generics を載せる。型変数・unification は無いままで、
**generics は明示適用 + 境界付き subtyping** で実現する。

## ゴール (wishlist #37)

agent/function の (param-sig P, return R, effect E) 多態。handler-providing
combinator (`with_session` / `retry` / `with_timeout`)、`call_agent` spreading、
map/filter/reduce、first-class handler provision の土台。data generics
(`Box[T]` 等) も自然な拡張として同じ仕組みに乗せる。

## 方針 (確定済み)

- **高階 generics なし**。generic agent の値はそのまま渡せない。
- **すべての使用箇所で明示適用が必須**(call でなくても)。
  - `agent foo[T](...)` に対し `let x = foo` は NG(foo は generics を要求)。
  - `let x = foo[int]` / `let x = foo[U]`(外側で導入した `U`)は OK。
  - 型チェック時に「未適用の generic 参照」を検出して reject。
- **kind は明示**(kind inference なし)。型パラメータ `T extends t`、effect
  パラメータ `effect R`。
- generics は**実行時に erase**される(IR は型を持たない)。monomorphize しない。
  Lowering は型引数を捨てて callee だけ lower する。

## 1. 表層構文

### 宣言

```
agent foo[T extends t, effect R](x: T) -> T with R { ... }
data Box[T](value: T)              // data generics (同じ仕組み)
```

- 型パラメータ: `T` または `T extends <upper>`。`extends` 省略時は上界 `unknown`。
- effect パラメータ: `effect R`(上界は全 request 集合 = 制約なし)。
- 宣言順がそのまま適用時の位置引数順。
- `[T extends int, U extends T]` のように後続パラメータの上界で先行を参照可。
- 内側 (local) agent で外側の generics を上界に使うのも可。
- extends の右辺に自分自身を含む再帰は不可(subtyping ループ停止性のため)。

### 適用

```
foo[int, req_1 | req_2](x = 0)
```

- 角括弧内は**位置引数の comma リスト**。各引数は `SyntacticType`(後述の理由で
  effect 引数も型式として parse する)。
- effect 引数は `with` 句と同じ bare-name union 構文: `req_1 | req_2` / `(a | b) | c`。
  (string literal ではなく bare name。`with` と一貫。)
- `foo[int, req_1 | req_2]` と続く `(...)` は別々の postfix。`foo[int]` 単独
  (未 call)も構文上は可。ただし「未適用 generic 参照」チェックは別。

### parser の曖昧さ回避

`arr[i]` index 構文は廃止済み(`get(array=, index=)` に一本化)。よって値の後ろの
postfix `[...]` は **generic 適用で一意**。type 位置の `[T1,T2]`(tuple)/`array[T]`
/`record[T]` とは構文位置が違うので衝突しない。

## 2. Identifier: GenericsId + 型解決の一般化

### GenericsId

```haskell
newtype GenericsId = GenericsId Int   -- module-local counter (LocalVarId と同様)
```

宣言の `[T extends t, effect R]` を見たら、各パラメータに GenericsId を発行し、
signature(parameters / returnType / withRequests / body)を解決する間だけ scope
に積む。

### 型参照解決の一般化 (タスク1の調査で判明したギャップ)

現状 `NameRefResolution _ TypeRef = Maybe QualifiedName`。generic param と request
(effect 引数として型位置に現れる)を表せるよう sum へ:

```haskell
data TypeResolution
  = ResolvedNamedType QualifiedName    -- data / 型シノニム
  | ResolvedGenericParam GenericsId    -- 型 or effect の generic パラメータ
  | ResolvedRequestName QualifiedName  -- req 名 (effect 引数位置でのみ意味を持つ)
-- NameRefResolution _ TypeRef = Maybe TypeResolution
```

`SymbolEntry.typeSymbol :: Maybe QualifiedName` も `Maybe TypeResolution` に拡張し、
scope frame に generic param を `ResolvedGenericParam gid` として積む。`resolveTypeRef`
は **generic param → 型 → request → prim** の順で解決。

> bracket 引数の名前を「型として」一様に parse・解決し、**kind(型 vs effect)の
> 振り分けは checker が宣言の generic パラメータ kind を見て行う**。これにより parser
> は kind を知らなくてよい。`req_1 | req_2` は `TypeUnion [TypeName req_1, TypeName
> req_2]` として parse され、effect パラメータ位置で checker が request 集合へ読み替える。

## 3. SemanticType

```haskell
-- 新コンストラクタ
SemanticTypeGeneric :: GenericsId -> SemanticType phase
-- effect tree の新リーフ
SemanticEffectGeneric :: GenericsId -> SemanticEffect phase
```

### Generic scheme (forall)

generic callable の signature は「スキーム」:

```haskell
data GenericParameter = GenericParameter
  { genericId :: GenericsId
  , kind      :: GenericKind          -- TypeKind | EffectKind
  , upperBound :: SemanticType Resolved  -- 型param=extends上界 / effectparam=全effect(=Unknown相当)
  }

SemanticTypeScheme :: [GenericParameter] -> SemanticType phase -> SemanticType phase
```

- bare `foo` の型 = `SemanticTypeScheme params (SemanticTypeFunction ...)`。
- **未適用検出**: `synthExpr` が変数参照の型として `SemanticTypeScheme` を得て、かつ
  それが generic 適用 (`ExpressionTypeApplication`) の直下でない場合 → `K02xx` エラー
  「generic agent は型引数の明示適用が必要」。
- `foo[args]` 適用 = スキームの params に args を**代入**して
  `SemanticTypeFunction`(具体)を得る。続く `(...)` は通常の call。

## 4. NormalizedType: generics レイヤ + 境界付き subtyping

### genericsLayer

`LayeredType` にもう 1 層:

```haskell
genericsLayer :: Set GenericsId
```

union = 集合和、intersection = 集合積(**completeness を捨て soundness を取る**:
`T & int -> never` は範囲が狭まるだけなので健全。intersection は agent 同士 union の
引数部=反変位置でしか使われないため安全)。

union で generics が nest していても従来通り union: `array[T] | array[U] -> array[T|U]`。

### 境界環境

```haskell
type BoundEnv = Map GenericsId NormalizedType   -- extends 上界(正規化済み)
```

`subtypeNormalizedType` / `normaliseSemantic` 等に読み取り専用で渡す。

### subtyping(境界展開ループ)

`subtype lhs rhs`:

1. lhs の generics のうち **rhs にも含まれるもの**を両者から除外(自由に動いてよい)。
2. lhs にしか無い generics それぞれについて、その**上界を正規化**し、lhs の他レイヤと
   union して新しい lhs とする(健全: `g <: bound(g)`)。
3. 上界に generics が含まれていれば、最外 genericsLayer に generics がある限り
   1–2 を繰り返す。extends は非再帰なので必ず停止。
   (上界が `array[U]` のように nest 位置に入った場合はその回で終了。)
4. generics が 0 個になったら、lhs/rhs の generics 以外のレイヤで従来 subtyping。

### match (scrutinee の generics 展開)

`e : T` を `[x, p] ->` 等の構造化パターンに当てる時、scrutinee 正規型の
tuple/array レイヤ + genericsLayer を見る。generics それぞれの上界を取り、subtyping と
同様に generics が消えるまで上界 union を繰り返す。消えたら最終 tuple/array レイヤと
パターンを当て、nest 位置で再帰(関数に切り出して抽象化)。object パターンも同様。

bare パターン `x` は generics の有無に関わらず scrutinee をそのまま採用。

(subtract: `int|string` に `int(x) ->; x ->` で後者を `string` に絞る件は現状の
exhaustive と整合する範囲で。completeness より soundness。)

### exhaustive

soundness 優先。generics は上界展開して既存アルゴリズムに載せる。

## 5. Lowering / Schema

- **Lowering**: 型引数は erase。`ExpressionTypeApplication { callee }` → callee のみ
  lower。generic agent 本体は単一ブロック(monomorphize なし)。
- **Schema**: generic agent をそのまま tool として出すケースは別途。当面は上界で
  近似 or 除外。discord example の tool-call で確認。

## 実装サブタスク

- 3a. Parser + AST: `[T extends t, effect R]` 宣言、`foo[args]` 適用 postfix。
- 3b. Identifier: GenericsId 発行、`TypeResolution` sum、scope への generic 積み。
- 3c. SemanticType: `SemanticTypeGeneric` / `SemanticEffectGeneric` / `SemanticTypeScheme`。
- 3d. Check: scheme elaborate、未適用検出、適用時の代入、bound seed。
- 3e. NormalizedType: genericsLayer + BoundEnv + 境界展開 subtyping + union/intersect。
- 3f. match/exhaustive の generics 展開。
- 3g. Lowering erase + Schema。
- 3h. テスト + e2e(discord tool-call)。
- data generics は agent generics 完成後に同枠で追加。
