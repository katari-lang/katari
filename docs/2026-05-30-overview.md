## 1. 動機

v0.1.0 (first publish) を AI オーケストレーターとして成立させるために、 以下が
言語仕様・runtime の根幹に関わる未解決問題として浮上した:

1. **大きい値の効率**: AI 会話履歴・画像・音声などを CORE state に inline で持つと、
   毎 tick で全 state を DB から reload / persist する現状では破綻する。
2. **streaming**: LLM のトークン列・音声 chunk など、 時間軸で materialize する値を
   扱えない。 callback per event (現状の event streaming) では UX が悪い。
3. **state load 効率**: 1 snapshot に複数 agent が並走するとき、 単一 State に全部
   入るので無関係な agent の state も毎回読み込む。
4. **module / snapshot の階層が不正確**: CORE / FFI が per-snapshot instance になって
   いるが、 これは Option A (後述) と矛盾。 ENV が project すら持たず global。

これらは「後から直すのが非常に大変」 なので、 v0.1.0 で完全解決する大マイルストーン
と位置づける。

## 2. 階層概念 (最重要)

すべての設計はこの階層に従う。 **project が全ての親**。

**snapshot だけが project の直接の子**。 それ以外 (instance / run / env var / file /
value) はすべて **どこかの module の子**。 module は per-project。

```
System (host process、 multi-tenant)
  └─ Project                          ← 全ての親、 isolation boundary、 永続
       ├─ Snapshot                    ← project の唯一の直接の子 (= CORE/FFI 共通の code)
       └─ Project runtime (= module 群、 すべて per-project):
            ├─ Bus (6 events の pure router)
            ├─ CORE module
            │    ├─ agent_defs   (snapshot, qname) → IR
            │    ├─ instances    (shard: threads / scopes / closures) → engine_shards
            │    └─ values       (owner=core)
            ├─ FFI module
            │    ├─ agent_defs   (snapshot, qname) → ext dispatch + (snapshot → sidecar bundle)
            │    ├─ instances    (in-flight ext call + relay)
            │    └─ values       (owner=ffi)
            ├─ API module        (= user の代理、 agent_def なし)
            │    ├─ instances    (= runs: user が起動した root delegation) → runs_audit に永続
            │    ├─ files        (api_files: persistent record + 共有 blob)
            │    └─ values       (owner=api、 ephemeral)
            └─ ENV module
                 ├─ agent_defs   qname → builtin op (snapshot 非依存)
                 ├─ env vars     (project-scoped)
                 └─ instances    (in-flight env lookup、 transient)
```

3 つの軸を明確に分離する:

| 軸           | 単位                  | lifecycle             |
| ------------ | --------------------- | --------------------- |
| **project**  | isolation boundary    | 明示削除まで          |
| **snapshot** | code version          | project 内に複数版     |
| **instance** | 実行中の agent         | 起動 → 完了            |

### module ごとの所有物

| module | agent_defs (key)         | instances              | 永続 resource              | values (owner) |
| ------ | ------------------------ | ---------------------- | -------------------------- | -------------- |
| CORE   | (snapshot, qname) → IR   | shard (engine state)    | —                          | owner=core     |
| FFI    | (snapshot, qname) → dispatch + sidecar bundle | in-flight ext call | — | owner=ffi |
| API    | なし (= user の代理)      | runs (= root delegation) | files (persistent)        | owner=api      |
| ENV    | qname → builtin (snapshot 非依存) | in-flight env lookup | env vars               | owner=env      |

ポイント:

- **instance は各 module の子**。 各 module が delegate を受けて自分の instance を spawn
  する (CORE は shard、 FFI は ext 呼び出し、 ENV は env lookup)。 live instance の
  routing/lifecycle registry が `delegations` table (= owner module で partition)
- **run は API の子**。 = API が自分の発行した root delegation の履歴を永続化したもの
  (`runs_audit`)。 他 module は live-only (= terminal で delete)、 audit history は API のみ
- **env var は ENV の子**
- **value = ref (module の handle) + blob (共有 bytes) の 2 層**。 ref は持ち主 module が
  責任を持つ handle (= core/ffi の ephemeral `value_refs`、 api の persistent `api_files`)、
  blob は project-wide で dedup される file 本体 (`value_blobs`)。 run/delegation と同じ
  「永続 record + 解放可能な実体」 の構図
- **file は API の子** (= API が upload を blob 化 + `api_files` record を作る persistent value)。
  v0.1.0 は API 所有で十分 (= 専用 FILE module は作らない)。 file は run の引数として ref で
  渡るだけで、 language からの「名前で file を引く」 capability は無い。 将来要るなら FILE
  module 新設の選択肢あり
- snapshot を識別子に含むのは CORE / FFI のみ (= code が版依存)。 ENV は版非依存
- かつての「ENV/API は global」 「CORE/FFI は per-snapshot」 は不正確。 正しくは
  **全 module 一様に per-project、 snapshot は (CORE/FFI の) def registry の key**

## 3. Option A: 単一 machine + agent_def テーブル

snapshot を「isolation boundary」 ではなく「code version」 とする (= Option A)。

- **snapshot = code version**。 deploy すると各 module の def registry (CORE / FFI / ...) に
  その版の agent_def が分配される。 snapshot 自体は def の集合の identifier
- **instance (shard) = 状態を持つ実行中の agent**。 `currentSnapshot` pointer を持つ
- **migration** (v0.2+) = instance の currentSnapshot pointer を新版に付け替え

v0.1.0 では migration は実装しないが、 `currentSnapshot` pointer を持つ構造にして
おくことで v0.2 で無理なく追加できる。 v0.1.0 の挙動: snapshot 削除 = その snapshot を
currentSnapshot に持つ instance を terminate (= migration せず終了)。 結果として
user 視点は Option B 風 (snapshot 削除 = instance 死亡) に見えるが、 内部 architecture は
Option A。

## 5. 言語仕様への影響 (意外と小さい)

- `file` primitive 型を追加 (`integer` / `string` / `boolean` / `number` と同列)
- `string == string` (content) と `file == file` (identity) の prim 分岐
- `file_to_string` / `string_to_file` / `persist` 等の変換 prim
- それ以外の syntax / semantic type / typechecker は **無変更**

user が書くコードは現状とほぼ同じ。 値が ref になりうるための特別な syntax / 型注釈はゼロ。
内部だけが対応する。 LSP 可視化 / hover は v0.1.0 ではやらない (= 将来課題)。

## 6. スコープと工期

v0.1.0 は **値モデル (blob) + actor/shard refactor + DB 再設計** に絞る。 observable
streaming は [v0.2-streaming](2026-05-30-v0.2-streaming.md) に分離 (D34)。 詳細は
[implementation-plan](2026-05-30-implementation-plan.md)。

これは「streaming / blob 機能追加」 ではなく **runtime / api-server の全体リファクタ**。
engine の host 層を tick-based から actor-based に作り直し、 DB を階層に合わせて再設計し、
値モデルを ref (complete blob) ベースに置き換える。 現状 user 0 なので migration は
考えない (= DB を一から設計してよい)。
