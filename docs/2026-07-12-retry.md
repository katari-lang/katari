# `prelude.retry` — failure-recovery providers (exponential, forever, attended)

2026-07-12. `prelude.time` (same day) deliberately shipped `watch` with **no built-in retry**: a `deliver_to`
that throws or panics kills the watch, and resilience is "composed at the call site, never baked in". This
phase supplies what composes it — `prelude.retry`, three `use` providers that wrap the rest of a block and
**re-run it on failure**. Pure Katari over durable `time.sleep` and the two error channels; no new reactor.

```katari
use retry.exponential(initial_delay_milliseconds = 100.0, factor = 2.0, max_attempts = 5.0)
<the rest of the block — re-run with backoff until it succeeds or the budget is spent>
```

A provider is an ordinary agent whose last parameter is `continuation`; `use p(args…)` makes the rest of the
enclosing block that continuation (the application form from `2026-07-03`). A retry provider is the first code
anywhere that **re-invokes a continuation** — calls it more than once — so this phase also proves that the
runtime handles re-invocation (it does, unchanged) and closes a one-line gap in generic inference (below).

## The surface

```katari
data thrown[Error](error: Error)          // a caught throw, its payload preserved
data panicked(message: string)            // a caught panic, its message
type failure[Error] = thrown[Error] | panicked

data succeeded[R](value: R)
data failed[Error](failure: failure[Error])
type outcome[R, Error] = succeeded[R] | failed[Error]

agent attempt[R, Error, effect E](
  run: agent (value: null) -> R with {...E, prelude.throw[Error]},
) -> outcome[R, Error] with E

agent exponential[R, Error, effect E](
  initial_delay_milliseconds: number, factor: number, max_attempts: number,
  continuation: agent (value: null) -> R with {...E, prelude.throw[Error]},
) -> R with E | io | prelude.throw[Error]

agent forever[R, Error, effect E](
  initial_delay_milliseconds: number, factor: number, max_delay_milliseconds: number,
  continuation: agent (value: null) -> R with {...E, prelude.throw[Error]},
) -> R with E | io

request attention[Error](failure: failure[Error]) -> null

agent attended[R, Error, effect E](
  continuation: agent (value: null) -> R with {...E, prelude.throw[Error]},
) -> R with E | attention[Error]
```

The user writes only the pinned surface — `use retry.exponential(initial_delay_milliseconds = …, factor = …,
max_attempts = …)`, `use retry.forever(…)`, `use retry.attended()`. `R` / `Error` / `E` are inferred from the
continuation; `continuation` is appended by `use`.

## The one normalization boundary, and why the throw stays typed

Katari has two failure channels — a typed `throw[Error]` a program raises, and a `panic` the runtime raises —
and a retry must treat both as "the attempt failed". `attempt` is the **single** place both are caught and
folded into one `outcome` sum. Every provider is then nothing but a `match` over `succeeded | failed`: there
is no test on a counter anywhere, no `if (n < max)`. Special-casing lives once, in the shape of the data.

The catch discharges `throw[Error]` for a **generic** `Error` (not an erased `throw[unknown]`), so a caught
throw reifies as `thrown[Error]` and a caller keeps its typed error end to end — `use retry.exponential(…)`
over an `http.fetch` still re-raises `throw[http.fetch_error]`, catchable by payload exactly as without the
retry. Erasing to `throw[unknown]` was rejected: it would collapse Katari's typed-error composition at every
retry boundary (a caller could no longer catch the specific exhausted error).

`run`'s effect is the **overwrite** row `{...E, prelude.throw[Error]}`, not the union `E | throw[Error]`,
precisely because `attempt` *discharges* the throw with a handler — the discord-provider shape (a handler-
discharged request splits itself out of the tail `E`). The union form is for a *propagated* marker (mcp's
`scope[URL]`, which a tool call raises and `provide` discharges from its own row); a handler discharge needs
`E` to exclude the request it eats.

## Policy core, or three providers? Three, sharing one helper.

The brief asked for a policy-based core if the type system let it stay clean, else "three self-contained
providers sharing a private helper". **Three providers won**, sharing the one `attempt` normalization helper,
for two reasons the type system forced:

1. **The wait action's effect differs and cannot be a uniform policy value.** `exponential` / `forever` wait
   with `time.sleep` (`io`); `attended` waits with the `attention` request (`attention[Error]`, which escalates
   or is intercepted). A single "policy agent" threaded through one mechanism would have to abstract over that
   effect *and* the `Error` payload — the row-payload threading gets ugly fast, exactly the escape hatch the
   brief allowed.
2. **Exhaustion cannot be a policy decision, because a caught panic cannot be re-raised.** A program can never
   perform `panic`, so a provider that *held* a failure could not re-panic it on exhaustion. So exhaustion is
   not "the policy returns stop"; it is a structural split (below). That does not fit a policy-returns-decision
   mechanism at all.

What the three share is the sum `attempt` mints — one boundary, one `failure` normalization — which is the
part of "a policy-based core" that stays clean. Each provider's loop is three or four lines of `match`.

## Exhaustion re-raises by running the last attempt uncaught

`exponential` runs its catchable attempts as a `for` over `array.range(1, max_attempts)` (so `max_attempts - 1`
of them), threading the growing delay as loop state; a success `break`s out with the value, a failure durably
sleeps and continues. When the loop is exhausted, its `then` clause runs `continuation(value = null)` **outside
`attempt`** — a bare, uncaught call — so whatever it raises (throw or panic) propagates **unchanged**, the
self-catch rule proxying it past the provider to the caller's handlers.

This is the whole reason there is no "is this the last attempt?" branch: the `for` body is the catchable
attempts and `then` is the final uncaught one — a structural split, not a comparison on `n`. It also re-raises
both channels uniformly and *typed*: the runtime carries the real payload, and because `Error` is inferred,
the static type the caller sees is the concrete throw, not `unknown`. At least one attempt always runs (a
`max_attempts` below 1 yields an empty range, so `then` still runs once).

`forever` has no last attempt: every failure is caught and retried, capped at `max_delay_milliseconds`
(`math.min`), so it never re-raises and never returns on its own — the daemon shape. It is a self-recursion
whose **depth is the number of failures, not the number of ticks**, so a healthy daemon accumulates nothing,
and each retry is a fresh durable delegation that recovers cleanly across a restart. The headline composition:

```katari
agent daemon() -> never with io {
  use retry.forever(initial_delay_milliseconds = 100.0, factor = 2.0, max_delay_milliseconds = 5000.0)
  time.watch(schedule = time.interval(milliseconds = 1000), deliver_to = flaky_deliver)
}
```

A `deliver_to` failure kills `time.watch` (by `time`'s design); `forever` catches it, backs off, and re-runs
`watch`, which re-arms from its persisted next occurrence — resilience composed *around* the watch, exactly as
`2026-07-12-time.md` promised.

## `attended` — the human in the loop, and interceptability

`attended` waits not by sleeping but by performing `attention(failure = …)`, carrying the normalized (typed)
failure. **Unhandled**, `attention` is an ordinary user-facing request: it escalates to a durable open
question that parks the run until `katari answer`, then re-runs the block. This is the re-auth story — a
revoked token throws, `attended` turns it into a notification, the human re-authorizes and answers `null`, and
the rest of the block re-executes against the restored credential.

Because `attention` is a plain request, an application **can intercept it** with its own handler installed
above the `use retry.attended()` — e.g. post to Discord and wait on a reaction — then `next null` to trigger
the retry (or `break` to give up with a value). The default (escalate-and-park) and the interception (handle-
and-wait-elsewhere) are the same mechanism, chosen by whether a handler is in scope; nothing in `attended`
branches on it.

## The one compiler change: request-row payload inference

`Error` is named only inside the continuation's effect row (`throw[Error]`), never in a value position, so
before this phase it was un-inferrable (K3016) — generics were inferred from arguments and data-type args, but
not from a request row's payload. `collectEffectConstraints` bounded only the flexible effect *tails* (which
requests a continuation adds to `E`), never the payload of a concrete entry.

The fix is one addition to `collectConstraints.goFunction` (`Typechecker/Inference.hs`), symmetric with the
existing `goData` for data-type arguments: a request shared by the actual and parameter rows relates its args
at the request's row variance, so `throw[Error]` against `throw[http.fetch_error]` proposes `Error ≥
http.fetch_error`. "Inference proposes, checking disposes" — the dispose pass re-verifies, so this only makes
more programs' generics *solvable*; it changes nothing already solved (the full compiler suite is unchanged,
plus two new `InferenceSpec` cases pin the retry shape). This is what lets the whole surface stay typed with no
type annotations at the `use` site.

## What re-invocation revealed (nothing broke)

A retry provider is the first code to invoke a continuation more than once, so this was unproven. A probe drove
the compiled `retry_demo` through the in-memory `ProjectActor` with a `ManualClock`: the continuation is
re-entered fresh each attempt, an ambient `warmup` counter installed *outside* the provider persists across
the re-invocations (reaching the success on attempt 3), the durable sleeps fire between attempts, and
exhaustion re-raises the typed `not_ready` unchanged. State duplication, return-target corruption, and handler
re-entry did **not** materialize — the engine handles re-invocation unchanged, so no engine change was needed.
The e2e `retry_demo` case is the standing net; the probe was throwaway.

## What was deliberately not built

- **No "retryable vs fatal" predicate / error filter.** Every failure retries; a caller that must not retry
  some errors catches those *inside* the block (a `use handler` closer than the provider) so they never reach
  it. A filter knob would be an `if` on a payload — the special-case-as-branch this codebase refuses.
- **No jitter, no max-total-elapsed, no dead-letter sink.** Each is a separate policy; composing them would be
  the おせっかい trap `time.watch` already refused. `exponential` is exact backoff; anything else is a different
  provider, added when a real program shows its shape.
- **No `attended` timeout.** Parking on a human is the point; a bounded wait is `exponential`/`forever` around
  the same block, or an app-side timeout in an intercepting handler.
- **No generic "policy" value type.** See above — the two failure channels' asymmetry (a panic cannot be
  re-raised) and the wait actions' differing effects make a single mechanism uglier than three small ones.
- **`max_attempts` stays `number`** (the pinned surface), floored once with `math.floor` where the counted
  loop needs an integer — no separate integer parameter, no deviation in the name or type a caller writes.
