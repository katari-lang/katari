# IR `drop` — provably-dead 変数の早期解放

## 背景

`let x = expr` は lowering で複数の compiler-internal temporary を生む。例えば `let x = f(a)` は

```
makeRecord {output: v1}          -- 引数レコード
delegate   {argument: v1, output: v2}
bindPattern {source: v2, pattern: variable v3}
```

に落ち、v1(引数レコード)と v2(delegate 出力)は bindPattern の直後から死んでいるのに、
scope の寿命いっぱい束縛されたままだった。scope は `values` マップ全体が JSONB 行として
**毎ターン再永続化される**(turn boundary で instance の所有 scope を wholesale に dirty マーク
— `ResourcePool.markOwnedDirty`)ので、死んだ束縛はターンごとに実書き込み量として課金され続ける。
AI ループのような長寿命 agent body ほど temporary が積もり、行が太る。

## 決定

1. **IR に `drop` operation を追加**: `{ kind: "drop", variables: [VariableId] }`
   (Haskell `OperationDrop` / TS `DropOperation`)。ランタイムは各束縛を実行スレッドの
   **local scope からだけ** delete する(`dropVariable` — `writeVariable` の鏡像。
   コンパイラは「同じ sequence が書いた変数」しか drop しないので、束縛は構築上 local)。
   同期 op であり cursor はそのまま進む。
2. **挿入は post-lowering pass**(`Katari.Lowering.Drop`、`lowerModule` の最後に適用)。
   liveness 規則は保守的:
   - sequence S の operation が **書いた** 変数 v は、「S 内の最後の言及」の直後で死ぬ。
     言及が write 自身でもよい(未読 output は書いた直後に drop)。
   - ただし v が module 内の他のどこか — 他 block の operations / 変数フィールド、
     どの block の `parameters`、S 自身の `result` — に一度でも現れたら候補から外す
     (nested block が scope chain 越しに読む変数、closure が捕獲する変数はこれで残る)。
   - 健全性の前提(コードで検証済み): sequence は straight-line で cursor は各 op を高々一度
     通過する(制御フローは別 block = 別 thread)、VariableId は module 全体で一意
     (lowering の単一カウンタ)、他 module からこの module の VariableId は参照不能
     (module 間は引数レコードの値渡し。closure も自 module の block に解決される)。
   - `exit` / `continue` の後には挿入しない(fall through しないので到達不能な死重)。
   - 空の drop は出さない。同じ op で複数死ぬ場合は 1 つの drop にまとめ、id 順 sort で決定的。
3. **`schemaVersion` を 1 → 2 に bump**。wire が「古いランタイムには実行できない operation」を
   得たので、この定数がその契約をそのまま文書化する。
4. **scope GC の健全性は不変**(gc.ts の mark-and-sweep)。旧前提「束縛は single-assignment で
   消えない」は「消えるのは compiler が unreadable と証明した束縛だけ」に弱まるが、
   reachability は縮む方向にしか動かないので、quiesced turn boundary で unreachable な scope が
   後から reachable に戻ることはない。

## 帰結

- 非厳密でよい: 証明できないものは黙って残し、scope-level GC(turn boundary の
  `unreachableOwnedScopes`)が従来どおり backstop になる。取りこぼしは「節約が減る」だけで
  正しさに影響しない。唯一のハード要件は「まだ読まれ得る変数を絶対に drop しない」こと。
- miscompile した drop は、次の read の `requireVariable` が
  `variable N is unbound in scope M` を投げて即座に顕在化する(silent null 化はしない —
  `result` は drop 対象外なので `readResult` の null fallback は踏まない)。
- 走査は Block / Operation / Pattern の全コンストラクタを total-case で書いてあるので、
  変数を持つフィールドが将来増えたら黙って漏れるのではなくコンパイルが落ちる。
