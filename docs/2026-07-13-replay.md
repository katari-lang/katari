# `prelude.replay` — the retry MECHANISM, split from the failure POLICY

2026-07-13. A redesign of the day-old `prelude.retry` (2026-07-12). The old providers baked "what counts as
a retriable failure" INTO the retry mechanism: an internal `attempt` caught BOTH a typed `throw` and a
`panic` and folded them into one `failure` sum, so `retry.forever` over a block retried on **every** failure
indiscriminately. `replay` keeps the same one-line ergonomics and the same flat-footprint `forever { … }`
loop, but moves the failure decision OUT of the mechanism and into user code, across the `use` boundary.

```katari
use replay.exponential(initial_delay_milliseconds = 1000, factor = 2, max_attempts = 5)
use handler {
  // POLICY: a user-written converter. It decides which failures become a replay.
  request prelude.throw(error: fetch_error | auth_error) -> never {
    match (error) {
      case fetch_error(_) -> { replay.interrupted(failure = error) }  // transient -> replay
      case auth_error(_)  -> { prelude.throw(error = error) }         // fatal     -> rethrow (proxies up)
    }
  }
}
<body>   // performs interrupted (via the converter); the provider sleeps and re-runs it
```

## Why the split — three review findings it resolves

A reviewer proved three problems with folding the failure decision into the mechanism:

1. **No selective retry.** A revoked-credential `auth_error` (non-transient) spun forever under
   `retry.forever` exactly like a transient `fetch_error`, because the generic catch swallowed every throw.
   There was no way to say "retry fetch errors but escalate auth errors". Under `replay` that is an ordinary
   `match` at ONE boundary in the converter (above): the transient arm signals `replay.interrupted`, the
   fatal arm rethrows and leaves the loop.
2. **Panic swallowing was invisible.** `retry.forever` silently retried deterministic `panic`s — a broken
   invariant re-run forever with no source that said so. Under `replay` retrying a panic is an **explicit
   opt-in** the converter writes: `request panic(msg) { replay.interrupted(failure = …) }`. A converter with
   no panic clause does not retry panics; the choice is legible in the source.
3. **A caught panic could not be re-raised.** The old `exponential` could not "hold the failure and re-raise
   it on exhaustion", because a program can never perform `panic`; it had to run its last attempt uncaught so
   the panic propagated. Under `replay` that whole problem **dissolves**: by the time control reaches a
   provider, every failure is already a `replay.interrupted[F]` whose `F` the converter chose as a plain
   value. Exhaustion re-raises a **typed `throw[F]`** — never a re-panic.

## The seam: one signal request

```katari
request interrupted[F](failure: F) -> never
```

This is the entire interface between policy and mechanism, and the **only performable request the module
exports**. A converter performs `interrupted`; a provider catches it. `-> never`: performing it transfers
control to the provider and does not return. A provider reasons only about `interrupted` — it never mentions
`throw` or `panic`. The name is the control-flow fact both sides agree on (the block's run was *interrupted*
and control handed to the mechanism), rather than a judgment (`retriable`) or a command (`again`) only one
side makes.

Because `interrupted` is a `-> never` control channel like `throw` / `panic`, an *orphan* `interrupted`
performed with no provider in scope **fails the run** — it never opens an un-answerable escalation at the run
root (its answer type is `never`). The runtime's failure-request set lists it alongside `throw` and `panic`.

`F` is the converter's own choice of failure representation: often the throw's payload directly
(`interrupted(failure = error)`), or — when a converter retries on BOTH channels (a typed throw AND a panic)
— its OWN two-constructor sum declared locally, so the two settle to one type. There is **no** `replay.failure`
/ `thrown` / `panicked` fold in the prelude: a convenience sum shipped by the mechanism was exactly the
"policy baked into the mechanism" smell this redesign removes. A converter brings its own `F`.

## Three mechanism providers, one loop

All three are ordinary `forever { … }` loops (2026-07-13-forever-loop.md), so durable state stays **flat** no
matter how many times the block has replayed: each finished iteration's frames are reclaimed, and what
persists is the loop thread plus its `var` state. The loop's evolving state — the attempt count and backoff
delay — is the **`forever`'s own `var`** (one owner); a `next … with (…)` advances it, and a `break` carries
the success value out. A single shared helper `run_once` runs the continuation once and reifies which
happened — `ran` with the value, or `signalled` with the failure `interrupted` carried (the one boundary that
catches the replay signal and turns it back into ordinary control flow). The loop then `match`es that sum:
the `ran` arm `break`s with the value, the `signalled` arm applies the policy's delay and `next`s.

```katari
agent immediate[R, F, effect E](continuation) -> R with E                              // unbounded, no delay
agent forever[R, F, effect E](initial, factor, max_delay, continuation) -> R with E | io  // unbounded, capped backoff
agent exponential[R, F, effect E](initial, factor, max_attempts, continuation)         // bounded, then re-raise
  -> R with E | io | prelude.throw[F]
```

- **`immediate`** re-runs at once — no artificial delay, no `io`, no `var`. The cadence is whatever the
  converter does *before* it signals (e.g. parking on a human, below).
- **`forever`** carries a `delay` var, backs off exponentially capped at `max_delay_milliseconds` (`math.min`),
  and never re-raises — the daemon shape.
- **`exponential`** carries `attempt` and `delay` vars, backs off for a bounded budget, then re-raises.
  Exhaustion is not a raw counter test: the `signalled` arm mints a `replay_after | give_up` verdict at ONE
  comparison of the count to the budget, then `match`es it — `replay_after` sleeps `delay` and `next`s with
  `attempt + 1`, `give_up` performs `prelude.throw(error = failure)`. The failure the converter chose is
  re-raised **typed as `throw[F]`**, which a caller catches by payload exactly as without the replay.

`F` appears in BOTH the continuation's request-row payload (`interrupted[F]`) AND `exponential`'s result-row
throw (`throw[F]`). That dual-position generic — a generic named only inside effect rows, solved from the
continuation and threaded into the re-raise — is exactly the inference the request-row-payload addition from
2026-07-12 (commit 441aee3) was built for; it carried the whole surface with **no further compiler change**.

`run_once` and the `ran` / `signalled` / `replay_after` / `give_up` sums are internal loop machinery — an
agent and data, **none of them performable**. They are module members only because Katari has no
module-private declarations. There is **no** `done` / `backoff` / `advance` request any more: loop exit is a
lexical `break`, and the loop state is the `forever`'s `var` — so no request a block could perform can hijack
the loop. `interrupted` is the module's sole performable request.

## The converter, verbatim — the forms that compile

A converter is an ordinary `use handler` installed BETWEEN the provider and the body. The design rests on
three behaviours, all verified over compiled IR:

- **A `throw` handler performs `interrupted`.** A `request prelude.throw(…) -> never` handler may perform
  `replay.interrupted(…)` (also `-> never`) as its divergent tail instead of `break`/`next`:

  ```katari
  request prelude.throw(error: warming_up | unauthorized) -> never {
    match (error) {
      case warming_up(_)  -> { replay.interrupted(failure = error) }  // retry this one
      case unauthorized(_) -> { prelude.throw(error = error) }        // rethrow: leaves the loop
    }
  }
  ```

  The rethrow arm re-performs `throw` from inside the `throw` handler; the **self-catch rule** (a request
  from a handler's own body escapes past that handle) proxies it up to an outer handler rather than looping
  on itself. The `interrupted` it performs is a *different* request, so it too proxies up — past the
  converter, to the provider's inner handler.

- **A panic converter is explicit.** Retrying a panic is opt-in, in source. `panic` is undeclared, so its
  parameter name is wired in as `msg` (the message) — a bare panic handler whose parameter is named anything
  else is rejected with a targeted **K3023** ("a `panic` handler's parameter must be named `msg`"), not a
  cryptic subtype mismatch. The converter folds the panic into its own `F`:

  ```katari
  request panic(msg: string) { replay.interrupted(failure = crashed(message = msg)) }
  ```

- **A human-in-the-loop escalation is the converter's OWN request.** A converter that parks for a human
  performs an escalation IT declares (`-> null`) — nothing replay-specific about it — and, on the answer,
  signals:

  ```katari
  request needs_reauth(detail: session_expired) -> null   // the converter's module declares this

  request prelude.throw(error: session_expired) -> never {
    needs_reauth(detail = error)          // unhandled: escalates to an open question; resolves null on answer
    replay.interrupted(failure = error)   // then re-run the block against the restored credential
  }
  ```

## `attended` dissolves into `immediate` + a user escalation

There is no `attended` provider, and no `replay.attention` request. The attended behaviour — failure →
escalate to a human → park → re-run on the answer — is exactly `use replay.immediate()` plus a converter that
performs an ordinary escalation the application declares itself (e.g. `needs_reauth`, above). Unhandled, that
request escalates to a durable open question that parks the run until `katari answer`; an application may
instead install its OWN handler for it above the provider (post to a chat, wait on a reaction) and `next null`
to trigger the replay, or `break` to give up with a value. Default (escalate-and-park) and interception
(handle-in-process) are the same request, chosen by whether a handler is in scope — nothing branches on it,
and none of it is coupled to `replay`. The re-auth story: a revoked token throws `session_expired`, the
converter turns it into a `needs_reauth` notification, the human re-authorizes and answers, and `immediate`
re-runs the block against the fresh token.

## The flagship composition — a resilient daemon

```katari
agent daemon() -> never with io {
  use replay.forever(initial_delay_milliseconds = 100, factor = 2, max_delay_milliseconds = 5000)
  use handler {
    request prelude.throw(error: delivery_failed) -> never { replay.interrupted(failure = error) }
  }
  time.watch(schedule = time.interval(milliseconds = 1000), deliver_to = flaky_deliver)
}
```

`time.watch` ships with no built-in retry (2026-07-12-time.md): a `deliver_to` that throws kills the watch.
The converter turns that typed failure into `replay.interrupted`, and `replay.forever` catches it, backs off,
and re-runs `watch`, which re-arms from its persisted next occurrence — resilience composed *around* the
watch, and flat in durable footprint however many deliveries have failed.

## The surface

Nine module members, of which exactly **one is a performable request** (`interrupted`):

- `interrupted` — the seam (the only performable request).
- `immediate` / `forever` / `exponential` — the three providers.
- `run_once` (agent) + `ran` / `signalled` (data) — the one justified abstraction: the boundary that catches
  `interrupted` and reifies it into ordinary control flow the loop can `match` on.
- `replay_after` / `give_up` (data) — `exponential`'s exhaustion verdict, a local sum matched once.

`run_once` / `ran` / `signalled` / `replay_after` / `give_up` are internal machinery, public only because
Katari has no module-private declarations; none is performable (an agent and data). The loop state is the
`forever`'s `var` and loop exit is `break`, so there is no internal loop-control request to leak.

## What was deliberately not built

- **No catch-all "convert every failure" converter in the prelude.** The whole point is that conversion is
  explicit user code; a shipped catch-all would re-bake the exact policy-in-mechanism coupling this redesign
  removed. The pattern is documented (above), not provided.
- **No `attended` provider, no `attention` request, no `attempt` / `outcome` / `succeeded` / `failed` /
  `failure` / `thrown` / `panicked` public types.** The old throw+panic normalization and the convenience
  fold are gone from the surface entirely; a converter that retries on both channels declares its own
  two-constructor sum. The human-in-the-loop escalation is the application's own request, not a `replay` one.
- **No jitter, max-total-elapsed, dead-letter sink, or attended timeout.** Each is a separate policy;
  composing them in would be the おせっかい trap `time.watch` already refused. A bounded human wait is
  `exponential` / `forever` around the same block, or an app-side timeout in an intercepting handler for the
  converter's own escalation request.
