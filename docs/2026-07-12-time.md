# `prelude.time` ‚Äî durable wall-clock time (a clock reading, durable sleep, and schedule watching)

2026-07-12. Katari had no notion of time: no clock primitive, no sleep, no timers. This phase adds
`prelude.time` ‚Äî `now`, `sleep`, `sleep_until`, and `watch` ‚Äî as a new in-runtime reactor (`time`), the
sibling of `http` / `webhook` / `mcp`. The whole point is DURABILITY: a runtime that persists execution and
recovers from failures cannot treat time as an ambient side effect, because recovery replays work, and a
naive clock disagrees with itself on replay. Every surface here is designed around that one constraint.

## Why time is a reactor, not a prim (the replay problem)

A prim runs as pure computation *inside a turn*. The substrate folds consecutive turns into a batch and
commits the batch atomically; on a commit failure or a restart **the whole batch re-runs from its durable
inputs** (`substrate.ts`). So a prim that read `Date.now()` would compute one instant the first time and a
*different* instant on replay ‚Äî a recovered run would believe time jumped. Worse, any value derived from it
(a deadline, a branch on "is it past 5pm") would silently diverge from what already happened.

The fix is the same one `http.fetch` uses for non-idempotency: route through a reactor. A reactor call
resolves with a `delegateAck`, and that ack's value is **journaled in the durable outbox**. Replay reads the
recorded value from the outbox row; it does not re-read the clock. So:

- **`time.now()` is durable precisely because it is a reactor call.** The instant is observed once, recorded
  on the ack, and every later replay sees that recorded instant. A prim `now` could not be durable at all ‚Äî
  this is not an optimization, it is the reason `now` is external.

The same reactor owns the timers `sleep` and `watch` wait on, so their deadlines are persisted state that
re-arms across a restart rather than in-memory `setTimeout`s that vanish with the process.

## The surface

Four externals and a two-variant schedule sum. Deliberately small ‚Äî see "what was not built".

```katari
external agent now() -> number from "time"
external agent sleep(milliseconds: number) -> null from "time"
external agent sleep_until(time: number) -> null from "time"

data interval(milliseconds: number)                        // fires every `milliseconds`
data cron(expression: string, timezone: string)            // fires at the cron occurrences, in the IANA zone
type schedule = interval | cron

external agent watch[effect E](
  schedule: schedule,
  deliver_to: agent (time: number) -> null with E,
) -> never with E | io from "time"
```

All times are **epoch milliseconds** (whole ms since the Unix epoch), a plain `number`. External calls carry
the un-dischargeable `io` effect implicitly (the impure, non-lifting call path, like `http.fetch`), so `now`
/ `sleep` / `sleep_until` need not spell it; `watch` writes `with E | io` because `E` (the `deliver_to`
callee's effects) must ride out to the caller's handlers, exactly as `webhook.inbound` propagates its
callables' effects.

- **`now`** resolves with the current instant. A whole-millisecond value decodes as `integer` on the wire,
  which conforms to the declared `number` (integer ‚äÜ number) ‚Äî no special decoding.
- **`sleep` and `sleep_until` are one operation.** The reactor collapses both to a single `sleep` variant
  carrying an ABSOLUTE deadline, decided once at the call's entry: `sleep(ms)` becomes `now + ms`,
  `sleep_until(t)` becomes `t`. There is no relative/absolute flag downstream ‚Äî the special case is resolved
  at the data's boundary into one shape, so recovery re-arms the exact same instant either way. A deadline
  already in the past resolves immediately.
- **`cron`'s timezone is required, with no default.** "Every day at 09:00" is a different instant in each
  zone; a durable scheduler must not guess. The reactor uses `cron-parser` (v5), whose luxon-backed `tz`
  option gives correct IANA-zone occurrences and supports both 5-field and 6-field (leading-seconds) cron.
  A well-maintained dependency was chosen over hand-rolling because timezone/DST arithmetic is exactly the
  kind of thing a hand-rolled parser gets subtly wrong.

## Durability semantics

The reactor is an `ExternalCallReactor` (the shared callee-call lifecycle behind `ffi` / `http` / `webhook` /
`mcp`), so it inherits the durable call envelope, the inner-delegation bridge, and the graceful-cancel
barrier. Its own durable state is one jsonb `operation` column on `time_instances` ‚Äî a tagged union
(`now | sleep{deadline} | watch{schedule, deliverTo, nextTick} | invalid{message}`), like
`core_instances.target`. There is no capability token to look up (unlike `webhook`/`mcp`), so no split
columns or subtype tables are warranted; the column is sealed at rest because a `watch`'s `deliver_to` may
close over a secret.

**A time call survives a restart completely** (the webhook recovery story, not the ffi/http one): there is no
external process to reconcile, so on reload the operation is read back and its timer re-armed. Concretely
(`TimeReactor.recover`):

- a `now` re-resolves with the current instant (no result was committed, or the call would be gone);
- a `sleep` re-arms its persisted deadline ‚Äî **a deadline that passed while the runtime was down arms with
  delay 0 and fires at once**;
- a `watch` re-arms its persisted next occurrence, unless a tick delivery is still in flight (then that
  delivery's durable core work resumes on its own and re-arms the next occurrence when it settles).

### `watch`: at-least-once ticks, serialized, with a single restart catch-up

`watch` computes occurrences and calls `deliver_to` once per occurrence via an inner delegation carrying the
occurrence's scheduled epoch ms ‚Äî the `discord_watch` shape. The design decisions:

- **The `nextTick` cursor is the single source of truth.** It is the epoch ms of the next occurrence to fire,
  advanced past *every* missed occurrence as ticks are delivered. This is what makes recovery cheap and
  correct.
- **Deliveries are serialized.** The next occurrence's timer is armed only when the current delivery
  *settles*, never before. A `deliver_to` slower than the interval therefore *rate-limits* the ticks (one
  in flight at a time) rather than queueing an unbounded backlog. This is a deliberate at-most-one-in-flight
  policy, not incidental.
- **A restart across missed occurrences fires EXACTLY ONE catch-up, then continues on schedule.** If the
  runtime is down across ten interval boundaries, recovery arms the persisted `nextTick` (the *earliest*
  missed occurrence), which is now in the past, so it fires immediately ‚Äî once. Firing it advances the
  cursor to the next occurrence strictly after *now* (skipping the other nine), so the schedule resumes on
  its original phase. No backfill of the missed occurrences: a durable scheduler that replayed every missed
  tick after an outage would stampede its downstream.
- **Ticks are AT-LEAST-ONCE, not exactly-once.** The tick delivery is a durable inner delegation (it resumes
  after a restart), but the boundary between "delivered" and "cursor advanced" is a commit, and a crash in
  that window re-delivers the same occurrence on recovery. So `deliver_to` must tolerate a repeated
  scheduled instant. The scheduled epoch ms is passed precisely so a handler can dedupe on it if it must.
  (Exactly-once would require two-phase coordination with `deliver_to`'s own effects, which the runtime
  cannot provide generically ‚Äî the same reason ordinary delegations are at-least-once under recovery.)

### Why `watch` has no built-in retry (composition, not configuration)

A `deliver_to` that throws or panics is **not retried**. The failure is fed back as the whole call's
outcome ‚Äî a throw re-raises typed to the caller, a panic escalates ‚Äî so the watch dies exactly as an
uncaught failure in any callee does. This is a firm boundary, not an oversight:

- Resilience is a SEPARATE concern with its own policy (how many retries, what backoff, which errors are
  retryable, a dead-letter sink). Baking one policy into `watch` would force it on every caller and still
  satisfy none precisely ‚Äî the „Åä„Åõ„Å£„Åã„ÅÑ trap.
- Katari already composes this: wrap `deliver_to` in a **retry provider** (built in parallel), and pass the
  wrapped agent. `watch` fires occurrences; the provider owns failure. Two small pieces that compose beat
  one big piece with a retry knob.

The playground's `every_second` shows the dual composition for *termination*: `watch` never returns, so a
stateful handler counts occurrences and throws to stop after three ‚Äî resilience and termination are both
composed *around* `watch`, never inside it.

### A malformed schedule is a panic, not a throw

`watch` declares no throw effect, so an unusable schedule (a bad cron expression / timezone, a non-positive
interval) cannot be a typed error ‚Äî a correct program supplies a valid schedule, so a bad one means the
program is broken. The reactor decides this once at entry: the bad schedule becomes an `invalid` operation
variant (hoisted out of `watch`, so a live `watch` always carries a valid armed `nextTick` ‚Äî no nullable
"maybe-armed" state), which dispatch turns into a panic that fails the run. This follows the throw/panic
dividing line from `2026-07-03-throw-error-model.md`: could a correct program meet this and continue? No ‚Üí
panic.

## What was deliberately not built

- **No recurring `sleep` / `interval` sugar beyond `watch`.** `watch` with an `interval` schedule already is
  the recurring primitive; a separate `every` would be a second way to say it.
- **No cron "run at most N times" / end-date / jitter knobs.** Those are caller policy, composed around
  `watch` (the termination pattern above), not scheduler configuration.
- **No monotonic clock / high-resolution timers.** `now` is wall-clock epoch ms ‚Äî the durable, replay-stable
  notion. Sub-millisecond timing is not an orchestration concern and would not be durable anyway.
- **No built-in retry, backoff, or dead-lettering in `watch`** (see above) ‚Äî the retry provider owns it.
- **No timezone default for `cron`** (see above) ‚Äî required and explicit.
- **No exactly-once tick delivery** (see above) ‚Äî at-least-once, with the scheduled instant for deduping.

## Runtime shape (for the reader who wants the mechanism)

- `TimeReactor extends ExternalCallReactor<TimePayload>` (`typescript/runtime/src/runtime/actor/time-reactor.ts`).
  `openPayload` decides the `TimeOperation` sum once from the compiled external key; every lifecycle method
  (`dispatch` / `recover` / `abort` / `onTimer` / `deliverInnerOutcome`) dispatches that sum structurally.
- The wall clock and one-shot timers are an injected `Clock` (`external/clock.ts`): `SystemClock`
  (`Date.now` + `setTimeout`) in production, `ManualClock` in tests (advanced by hand ‚Äî durable time is
  tested deterministically with no real waits). This is the same dependency-injection seam as the http / ffi
  / mcp transports.
- A fired timer re-enters the serial loop through the substrate's `submit` (like webhook's post-commit work),
  so a `sleep` resolve or a `watch` tick commits with a real turn.
- Durable state: `time_instances` (migration `0007_time_instances`), reloaded by `TimeReactor.recover` on
  boot the same way webhook re-registers its endpoints.
