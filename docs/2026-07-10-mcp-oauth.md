# MCP outbound auth: `auth` sum + `katari mcp login` (v0.1.0, scrap-and-build)

ゴール: **MCP サーバーへの認証を、ヘッダ方式(anonymous / bearer key / 任意スキーム)と
OAuth 2.1 の両方で、プログラムにトークン素材を一切書かずに合成できること**。

```katari
// ヘッダ方式(anonymous は空 record、bearer key は secret を値に)
let tools = mcp.tools(url = url, auth = mcp.headers(values = record.empty()))
// OAuth: out-of-band で確立した named credential を参照するだけ
let tools = mcp.tools(url = url, auth = mcp.oauth(name = "github"))
```

このドキュメントは、`headers` パラメータを `auth` 直和に置き換えた表面の変更、credential の
保存契約(`mcp.oauth.<name>` の env secret)、`katari mcp login` の分業、そして `auth_error` の
エラー契約を記録する。

## 1. 表面: `auth` は data 直和(`prelude/mcp.ktr`)

- `data headers(values: record[string of private])` — 毎リクエストに乗るヘッダ。anonymous も
  bearer key も custom スキームもこの 1 コンストラクタで一様に表せる。値は secret 可
  (privacy marker が mint 後の tool 値まで生きるのは従来どおり)。
- `data oauth(name: string)` — `katari mcp login` で out-of-band に確立してサーバー側に保存した
  OAuth credential の**名前**。名前は secret ではない — トークン素材はプログラムに一切現れない。
- `type auth = headers | oauth`、
  `external agent tools(url: string, auth: auth) -> toolbox with prelude.throw[server_error | auth_error] from "mcp"`。
- data 値はワイヤ上 `$constructor` 判別子(`Katari.Schema` の判別キー)で運ばれ、ランタイムは
  **transport 境界の 1 箇所だけ**でそのタグをディスパッチする(reactor は auth の中身を見ない)。

## 2. ランタイム: 記述子は `{url, auth}`、credential store は env secret

- tool の記述子(mint された tool 値の context)は `{url, auth}` になった。クライアントキャッシュの
  キーは url + **auth の同一性** — ヘッダ map、または oauth credential の**名前**。トークン素材は
  リフレッシュで回転するのでキーには決して入れない。
- `oauth` 変種は SDK transport の `authProvider`(`OAuthClientProvider`)で認証する。実装は
  `mcp-oauth.ts` の `StoredMcpOAuthProvider`: store から読んだ tokens を差し、リフレッシュされた
  tokens を**同じ store に書き戻す**(refresh token は single-use が普通なので、書き戻さないと
  再起動ごとに credential が死ぬ)。対話的ステップ(`redirectToAuthorization`)に到達したら
  credential はリフレッシュ不能 — typed `auth_error` で拒否する。
- credential store は小さな port(`McpCredentialStore` — `load` / `save`)で、facade が
  `McpBlobProducer` と同じ形で project ごとに `SdkMcpTransport` へ配線する。実体は env entries
  テーブルの予約キー **`mcp.oauth.<name>`**(secret: AES-GCM at rest、admin API では write-only)。
  つまり `katari mcp login` が書く場所・ランタイムが読む場所・refresh が書き戻す場所は同一の
  env secret 1 個。通常の env ツーリング(`katari ls env` / `env unset`)でそのまま一覧・削除できる。

## 3. `katari mcp login` — フローは node helper、保存は CLI

OAuth 2.1 の authorization-code + PKCE + dynamic client registration は
`@katari-lang/mcp`(bin `katari-mcp`)が実行する — `katari-bundle` と同じ配置の node helper
(解決順も同じ: `KATARI_MCP_BIN` → `node_modules/.bin` walk-up → PATH。npm shim の
`katari.mjs` は両方の override を export する)。

```
katari mcp login --url https://mcp.example.com/mcp --name github [--scope "..."]
```

1. helper が loopback の ephemeral port で redirect listener を立て、SDK の `auth(...)` で
   discovery + dynamic registration + authorization URL 生成まで進める。
2. authorization URL は **stderr** に印字(ローカルブラウザ起動は best-effort の
   `xdg-open` / `open`)。ユーザーがブラウザで承認すると loopback に code + state が戻る
   (state 不一致は拒否)。
3. code を交換し、credential JSON `{ tokens, clientInformation, resourceUrl }` を **stdout** に出す。
   helper はフローの実行と JSON の emit だけ — 保存の知識を持たない。
4. Haskell 側(`Command/Mcp.hs`)は stdout を捕捉(stderr は inherit なので URL はユーザーに見える)、
   形を一度だけ検査し、**verbatim のまま** runtime env API で `mcp.oauth.<name>` の secret として
   PUT する。確認メッセージが参照方法(`auth = mcp.oauth(name = "...")`)を印字する。

この分業で helper は runtime API を知らず、Haskell CLI は OAuth を知らない。

## 4. エラー契約: `auth_error` は `server_error` と質的に別

- `server_error` の契約は「retry すれば再接続する」— 接続断・サーバー拒否・restart 中断。
- `auth_error` の契約は「**retry では直らない**。人間が `katari mcp login` を再実行する」—
  named credential の不在、保存 blob の読解不能、リフレッシュ不能な期限切れ / 失効
  (SDK の `UnauthorizedError` を含む)。tool 呼び出しも同条件を踏み得るので、`tool` 型の
  エフェクト行にも `auth_error` が入る。
- 判別は transport の catch で**クラスにより**行う(`McpAuthError` / `UnauthorizedError` →
  `prelude.mcp.auth_error`)。メッセージ文字列の sniff は存在しない。
- ワイヤ形の malformed(auth が未知の constructor 等)は wire drift であり従来どおり typed
  `server_error`(catch 可能、panic にしない)。

留意(スコープ外として記録): login フローの対話部分(実 IdP + ブラウザ)は自動テスト対象外
(helper の引数解析と callback 解析、runtime 側の provider / store / 両変種の live loopback パスは
テスト済み)。`mcp.oauth.*` は予約 namespace だが env API 上の強制はしていない — 同名キーを手で
書けば login と同じ効果になる(それ自体は正しい: 保存契約が env secret だから)。
