# `finally` — instance 終了時に走る finalizer

## 背景

長寿命の agent instance には「正常終了でもキャンセルでも、最後に必ず走らせたい後始末」がある
(確保したリソースの解放、外部への「もう使わない」通知など)。従来これを表現する構文はなく、
`use handler` の `then` 節や手書きの分岐で近似するしかなかった — キャンセル経路では走らない、
複数の後始末を積むと入れ子が深くなる、といった穴が残っていた。

Go の `defer` に相当する専用の**文**として `finally { ... }` を導入する。

## 決定

1. **`finally { <block> }` は文**(式ではない)。評価すると、その block を現在の instance の
   **finalizer スタックに積む**(arming)。値は返さない — 効果は arming のみ。body は
   囲みスコープを通常の parent chain 越しに読むので、パラメータを持たない。

2. **発火のタイミングと順序**。armed finalizer は、instance が自分の terminal を ack する直前に
   **arming の逆順**で走る。同じ `finally` を二度通れば(ループ body など)二度積まれる —
   これはスタック規律であって不具合ではない。

   | 状況 | finalizer は走るか |
   | ---- | ------------------ |
   | 正常完了(`delegateAck` の直前) | 走る |
   | キャンセル到着(`cancelAck` の直前) | 走る |
   | panic(instance 異常終了) | **走らない** |
   | ハンドラ待ちで停止中(キャンセルがまだ来ていない) | 走らない(キャンセルが実際に到着してはじめて走る) |

   正常完了の ack もキャンセルの ack も、どちらも「terminal を確定させる一手前」で同じ
   finalizer スタックを消化する。panic は後始末の前提(スコープが健全)を壊しているので、
   意図的に走らせない。

3. **原子性**。finalizer 実行の途中でキャンセルが到着しても、実行中の turn は割り込まれず、
   その turn は 1 つの atomic commit として畳まれる(既存の turn batching に乗る)。到着した
   キャンセルは実行後まで保留され、最終的な ack は `delegateAck` から `cancelAck` に変換される
   — finalizer を二重に走らせない。

4. **io 限定規則(静的検査、新エラー K3021)**。finalizer body の**残余 effect row は `io` の
   範囲内**でなければならない。finalizer は、親がすでに instance のキャンセルを待っている最中に
   走り得る。`io` は sibling reactor に流れて親を経由しないが、**request(escalation)は親を
   経由して proxy される** — その親が自分のキャンセル ack を待っているなら、finalizer の
   escalation はその待ちとデッドロックし得る。これは構造的なデッドロック級のバグなので、
   finalizer からの escalation を**コンパイル時に禁じる**。
   - body 内で `use handler` により**ローカルに処理された** request は、通常の effect 推論が
     row から discharge するので残余には現れない — よって OK。io 限定検査は既存の推論が計算する
     **残余 row にだけ** `⊆ io` を課す(finally 文で 1 つ足すだけ)。
   - 制御 escape(`return` / `break` / `next`)も終了時には行き先が無いので、残余に残れば同じく
     K3021 で弾かれる。
   - エラーコードは K3021(K3019 / K3020 に続く構造化スタイル)。メッセージは理由を述べる:
     finalizer は自分のキャンセルをすでに待っているかもしれない親を通して escalate できない。

5. **囲みの effect row への寄与**。finally 文それ自体が「別枠の遅延効果」なのではなく、
   **instance は終了時に実際にその io を行う**。よって原則に忠実に、finalizer body の effect は
   通常の文と同じく囲みの row に join する(`enclosing ⊔ body`、ただし `body ⊆ io`)。実装上は
   body が io を行うときだけ `io` を囲みに emit する — 正しい body は io の範囲内なので、
   寄与し得るのは io のみ。これにより「finalizer で io する agent」はその row に io を持つ。

6. **panic in finalizer = instance panic**。finalizer 自身が panic したら、それは instance の
   panic として扱う(後始末の失敗は握り潰さない)。

7. **lowering**。body は他の nested block(match arm など)と同じ機構で、パラメータ無しの独立した
   `Sequence` block に落とし、文の位置に `OperationDefer { block }` を emit する。`defer` は変数を
   参照も書き込みもせず、制御を移譲しない(rest はそのまま続く)。`schemaVersion` を 2 → 3 に
   bump — wire が古いランタイムには実行できない operation(`defer`)を得た契約。

## 帰結

- finalizer は「正常 / キャンセルのどちらの terminal でも走る」「panic では走らない」という
  一点で、`then` 節や手書き分岐が取りこぼしていたキャンセル経路の後始末を漏れなく表現できる。
- io 限定規則により、finalizer 由来の escalation が親のキャンセル待ちと組む
  デッドロック級のバグは、実行時ではなくコンパイル時に排除される(K3021)。
- 既知のトレードオフ: finalizer が `io`(外部呼び出し)でハングした場合、その finalizer は
  **割り込めない** — instance の terminal はその io の完了を待つ。無限に応答しない外部先を
  finalizer から叩くと instance の終了もそこで止まる。長時間の外部処理は finalizer ではなく
  通常の body 側に置くべき、という運用上の制約が残る。
