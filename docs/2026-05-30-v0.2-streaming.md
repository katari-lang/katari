# v0.2: observable data streaming と reachability (将来計画)

> v0.1.0 では **observable streaming は実装しない**。 値は常に complete な ref
> ([value-and-streaming](2026-05-30-value-and-streaming.md))。 このドキュメントは、
> 2026-05-30 の議論で到達した「言語から観測できる streaming」 の設計と、 その正しい
> 基盤 (reachability) を v0.2 のために記録したもの。

## なぜ v0.1.0 で見送るか

CORE が stream の完了を待つ (= α / blocking) なら、 観測上「complete な値を返す
非ストリーミング呼び出し」 と区別がつかない。 streaming の本体 (mid-stream consume、
live 出力、 低 TTFT) は blocking では死ぬ。 一方、 non-blocking にすると escaping-ref
問題 (下記) が生じ、 正しく解くには reachability GC が要る。 これは v0.1.0 に重すぎる。

v0.1.0 は「値が ref になりうる」 だけ (= bloat 解決)。 mid-stream consume は v0.2。
Discord BOT の live 出力は **FFI-internal streaming** (= handler が LLM から受けながら
Discord に post) で賄えるので、 言語観測 streaming は v0.1.0 に不要。

## observable streaming が要求すること

agent が building ref を mid-stream で別 consumer に wire する (= live forward) には、
`let x = f()` が **non-blocking** (= building ref を即 bind して継続) でなければならない。

### producer の寿命を delegation tree に縛ると壊れる

non-blocking にすると、 producer を「いつ止めるか」 が問題になる。 delegation tree に
縛る案 (= producer を tree の child にして terminate cascade で止める) は 2 つの反例で破綻:

```katari
// 反例 1: break で止めたい (ref は escape しない)
handle { request done(r) { break r } }
let a = llm(prompt)        // a は使われない。 break で scope 破棄 → a の producer は止めたい

// 反例 2: return で escape する (止めてはいけない)
let ref = f(0)
if (true) { return ref }    // return は大域脱出 = delegation cancel を伝播するが、
                            // ref は呼び出し元が consume する。 producer を殺すと壊れる
```

tree-binding だと反例 2 で escape する ref の producer を殺してしまう。 「delegation の生死と
stream の生死を同期させる」 のが根本的に誤り。

### 正しいモデル: reachability

producer の寿命 = **ref が reachable な間**:

| ケース                          | reachability               | 正しい挙動            |
| ------------------------------- | -------------------------- | --------------------- |
| `return ref` (escape)           | 呼び出し元 scope に移動 → reachable | producer 生存 ✅ |
| `break r` (ref escape せず)      | scope 破棄で消失 → unreachable | producer cancel (AI 停止) ✅ |
| ref を drop / 上書き              | unreachable                | cancel ✅              |
| consumer に wire 中               | consumer が hold → reachable | 生存 ✅                |

reachability は break 懸念 (止めたい) と return 懸念 (escape を殺すな) を**同時に**解く
唯一のモデル。 これは元々 D10 で提案された「ref_count で building cancel」 の正しい形。

## v0.2 実装方式の候補

### 候補 A: explicit channel (筋が良さそう)

```katari
parallel {
  produce(out = X)    // producer が ref X に書く
  consume(in = X)     // consumer が ref X を読む (live)
}
```

- producer / consumer を `parallel` で**両方 in-tree** に置く。 ref X は両者の data 導管
- CORE は parallel join で block するが、 X 経由で FFI 間が live stream
- cancel は両端 in-tree なので **通常の terminate cascade で済む** (= reachability GC 不要)
- X は parallel の間だけ building、 両端 in-tree なので escaping-ref 問題が起きない
  (= parallel を抜けた後は X は complete)
- 新しい言語構文 (channel/pipe) が要る

### 候補 B: detach + reachability GC (完全 dataflow)

- producer を detach (tree から外す)。 寿命 = ref reachability
- CORE state walk で reachable building ref を mark、 unreachable な producer を cancel
  (= detect-on-push で LLM abort)
- run-cancel は prompt、 break/drop は GC lag (best-effort)
- cross-module の consumer hold まで数える必要があり、 最も重い
- 「全部 stream、 katari は配線」 の完全形

## producer への cancel 伝達 (両候補共通)

detach した (or building 中の) producer は別 process (sidecar) にいる。 cancel 伝達:

- **detect-on-push**: producer の次の `writer.push()` が「cancelled」 を返す → ループ break
  → for-await-of の cleanup が LLM fetch を abort。 active stream なら near-immediate
- stalled producer の prompt abort には control push (ipcCancelProducer 的) を追加可。
  v0.2 で検討

## building chunk の持ち方

mid-subscribe (= 途中購読で「これまでの全 chunk + 今後」) を成立させるには、 building 中の
chunk を all-so-far 保持する必要がある:

- in-memory transient buffer (host process)、 close で durable blob 化して free
- DB には persist しない (= per-chunk write を避ける、 crash で building は errored)
- 100MB cap、 超過で error

## v0.2 のスコープ (概算)

- non-blocking delegate (候補 B) or channel 構文 (候補 A)
- building state の value model 拡張 (`BytesRep` に building variant)
- data plane に subscribe (chunk SSE) / await (terminal SSE) endpoint
- reachability GC (候補 B) or in-tree channel (候補 A)
- producer cancel 伝達 (detect-on-push)

これらは v0.1.0 の complete-ref モデルの上に**追加**で乗る。 v0.1.0 の値モデル (ref /
content-addressing / dedup / file vs string) が streaming の foundation。

## multi-server と blob GC

v0.1.0 の blob GC (= ephemeral ref の reachability traversal) は **single-runtime 前提**
(= project が 1 server に丸ごと = single activation だから、 CORE state を traverse すれば
その project の reachable hash が完全に取れる)。

- **multi-server でも「project = single activation」 を保つ限り traversal は成立** (= 各
  project が自 runtime で自 ref を GC、 server をまたぐのは別 project)
- **persistent file (`api_files`) は traversal 不要** (= explicit な record、 行を数えるだけ)
  なので常に multi-server 安全
- 崩れるのは **1 project を複数 server に分割 (within-project sharding、 v0.3+)** とき。
  単一 traversal が全 shard を見られないので、 ephemeral ref の GC を **lease**
  (= holder が定期更新、 切れたら expire) or **incremental refcount** (= bind/drop で inc/dec)
  に移行する。 traversal-bound なのは ephemeral ref のみ
