# MCP inbound: `mcp.serve` — capability URL でエージェントを公開する (v0.1.0, scrap-and-build)

ゴール: **プログラムが動いている間だけ、選んだ agent 群を MCP サーバーとして外部に公開できること** —
`webhook.inbound` の MCP 版。認可は capability URL(推測不能なトークンを含む URL の所持)そのもの。

```katari
let outcome = mcp.serve(
  tools = { add = my_add, search = my_search },
  subscriber = agent (url: string) -> ... { ... },
)
```

このドキュメントは、`serve` の表面(`toolbox` が TOP 型として二役を果たすこと)、capability URL の
認可契約、エンドポイントのライフサイクル(subscriber と同寿命・再起動を跨いで生存)、そして
inbound 側 OAuth を**意図的に見送った**判断を記録する。

## 1. 表面: `external agent serve`(`prelude/mcp.ktr`)

```katari
external agent serve[R, effect E](
  tools: toolbox,
  subscriber: agent (url: string) -> R with E,
) -> R with E from "mcp"
```

- record のキーが公開ツール名、値の agent の宣言シグネチャが advertise されるスキーマ。
  `reflection.get_metadata` と**同じ** `callableMetadata` で解決するので、AI ループに見せる
  メタデータと MCP listing が乖離することはない。
- パラメータ型は既存の `toolbox = record[tool]` を再利用する。
  `tool = agent never -> unknown with io | prelude.throw[server_error | auth_error]` は
  「自己完結な callable」の **TOP 型**: 入力は反変で `never` が bottom、出力は共変で `unknown` が
  top、エフェクト行は部分集合方向 — つまり**エフェクトを handle し終えた任意の agent がそのまま
  coerce する**(playground の `mcp_serve_demo.ktr` で `{ double = double }` がコンパイルされる
  ことをもって方向を検証済み)。ユーザー側に儀式は要らない。
- 全体の呼び出しは subscriber の settle で settle する(結果 = `serve` の結果)。run の
  cancel は subscriber を cancel し、同じ経路で URL を無効化する。

## 2. 認可: capability URL が鍵そのもの

- ランタイムは呼び出しごとに 24 バイト(base64url)のトークンを mint し、
  `KATARI_PUBLIC_URL` 由来の 1 つの公開ベース(webhook と同じ knob)の下で
  `<base>/mcp/<token>` を subscriber に渡す。
- ルートは bearer 認証の `/api` の**外**(`/inbound` の隣)に mount する — 外部の MCP クライアント
  はランタイムの API key を提示できないし、させない。URL の所持が唯一の credential。
- 秘匿値は**この境界で redact** される(webhook の応答経路と同じ `valueToJson(…, "redact")`)。
  tool の結果に secret が混ざっても素材は外に出ない。

## 3. ランタイム: `mcp` reactor の `serve` variant(webhook をミラー)

- `McpPayload` に `serve` を追加(`openPayload` で `prelude.mcp.serve` キーを 1 箇所比較 —
  既存 `prelude.mcp.tools` と同じ流儀。以降は payload の直和 dispatch で、キー sniff はしない)。
- webhook-reactor と同型: token registry(token → delegation)、subscriber の一回きり
  dispatch(`dispatchCallable` + inner delegation)、`tools/call` ごとの inner delegation +
  waiter、`abort` / `onDropCall` での token 解放、subscriber の outcome を completion に
  合成して全体を settle。
- 永続化: `mcp_instances` に nullable な serve 列(`snapshot_id` / `serve_token`(unique)/
  `serve_tools`(sealed jsonb)/ `relays` / `inner_calls`)を追加(migration 0004)。transport 系
  (listTools / callTool)は従来どおり status のみ・at-most-once。serve は webhook と同じく
  **再起動を完全に生き延びる**: reload が token を再登録し、subscriber は durable な core work
  として自力で再開する。facade の `deliverMcp` は token 1 本で project を引く
  (`webhook_instances.token` と同じ単一クエリパターン)。

## 4. プロトコル: stateless streamable-HTTP を手書き

SDK の `StreamableHTTPServerTransport` は node の生 req/res と独自のライフサイクルを要求する。
このサーバーは Hono 上にあり、tool call はどのみち project actor の serial turn を通るので、
リクエストごとに `McpServer` を包み直すのは不釣り合い — 必要なメソッドは
`initialize` / `notifications/*`(202)/ `ping` / `tools/list` / `tools/call` の 5 つだけなので、
**JSON-RPC を手書き**した(`modules/mcp/mcp-serve.ts`)。stateless(セッション無し・POST-only の
JSON 応答・GET / DELETE は 405)なので再起動でセッションを失う心配が構造的に無い。ワイヤ互換は
**本物の SDK クライアント**(`StreamableHTTPClientTransport`)を loopback サーバーに繋ぐテストで
固定している。

Outcome → wire の対応:

- 返り値: bare string は text content、object は `structuredContent` + text fallback。
  spec 上 `structuredContent` は **object 型**なので、object 以外の JSON は text のみ。同じ理由で
  `outputSchema` は object 形のときだけ advertise する(SDK クライアントは schema を宣言した
  tool に structured content を強制するため)。
- typed throw: MCP tool error(`isError: true`、error JSON を text で)。ただし
  `reflection.call_error`(dispatch 境界のスキーマ違反 — agent は走っていない)は JSON-RPC の
  invalid params(-32602)。SDK サーバー自身の引数検証と同じ扱い。
- panic: internal error(詳細はサーバー側に留める)。
- 死んだ token: 404(不明)/ 410(巻き取り中)— cancel が URL を無効化する、の観測面。

## 5. inbound 側 OAuth を見送った判断

outbound(`mcp.tools`)には `auth` 直和 + `katari mcp login` があるが、inbound の `serve` には
OAuth リソースサーバー(RFC 9728 の protected resource metadata、トークン検証、issuer 設定)を
**載せない**。理由: capability URL が今の要件(プロジェクトオーナーが sanction した鍵認可)を
すでに満たしており、URL は呼び出しごとに新しく・推測不能・subscriber と同寿命 — 漏れたら
cancel すれば即失効する。OAuth を足しても「URL を知っている者が呼べる」以上の保証には、身元の
概念(誰のトークンか)をプログラム表面に持ち込む設計が先に要る。将来 per-caller の身元や
失効粒度が必要になったら、`serve` に auth パラメータの直和(webhook 側と揃えた形)を足すのが
自然な拡張点、とだけ記録しておく。

留意(スコープ外として記録): 通知(`notifications/*`)は 202 で受けて捨てる — stateless に
進める session が無い。`tools/list` の pagination は未対応(served record は小さい)。served
agent の**エフェクト行**は `tool` 型で「自己完結」に制限される — 外へ escalate する request を
持つ agent を公開したければ、まず handle してから record に入れる。
