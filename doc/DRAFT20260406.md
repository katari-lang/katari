# KATALI

Qatali を大幅に簡略化したいです。 (まず名前を KATALI にする)
AI オーケストレーターにここまでの機能はいらないと判断しました。逆に、他のもっとネイティブに近いプログラミング言語を同時に作り、そちらにこれらの機能を入れることにしました。そちらの言語は私がつくるので、あなたは Katali を改善する必要があります。

1. 新しい言語の言語仕様を細かく作ってファイルに保存してください。分割しても良いです。overview は次の節で述べます。
2. 既存のコードベースを全て消してください。バックアップはしてあるので、次に改めてプロジェクトを作るときはそちらを参考にできます。
3. 新しい言語を言語仕様に基づいて実装してください。

## 言語仕様

### 基本的な理念

オーケストレーションに必要な機能のみを付けます。具体的には、型システムの複雑さを下げます。effect system に似た言語仕様は持ちますが、基本的には server side のイベントを listen / 親コンテキストの関数を呼び出すための物です。
以下の機能は消します。

- Generics
  - type, data の定義, fn, val の定義から消す (完全に消す)
  - array だけ generics できるようにする　構文 → `array[T]`
    - array を construct するときは、それぞれの要素の型のunionのarrayとなる
  - result, option は effect system によって再現する。
  - たとえば、result は

```
effect throw(e: string) -> never

fn maybeThrow (...) -> T with throw
```

- 型推論
  - generics と型推論を消すことで、型チェックは全て決定的になり、楽
- Multi shot effect
- 効率化された IR
  - オーケストレーションに効率はいらず、効率がいる処理は別の言語に委譲するべきです。
- data type
  - 判別可能 union で代用します。result や option は effect system で記述可能

以下の機能を新たに追加します。

- Record 型 (object)
- もっと単純な IR
- external fn とその呼び出し
- 柔軟な template literal

次の機能は続投します。

- match
- Effect System (one shot)
- literal 型
- 部分型
- union, intersection
  - 続投しますが、union と intersection はその場で計算されます。判別可能 union はまた別です。後述。
- module システム、pub。
  - これらは同一のものを搭載してください。
  - コードベースを消す前にドキュメント化して保管してください。

## type system

重要:
基本的に現在使用可能な effect ops を保持して起き、effect op を呼び出した際はそれが op <: ops かを判定します。

### 型階層

never <: any type
any type <: unknown

配列と object に関係はありません。length などもありません。
また、タプル状の配列もありません。配列は常に 1 つの型の配列です。

他の点は ts と同じ。

### union / intersection のその場での計算

基本的に subtyping check は effect / fn への引数受け渡しと、let への代入で行います。
それぞれ左辺と右辺が渡されますが、両辺とも正規化します。ここで重要なのが union も問答無用で正規化することです。

```
{type: 0, x: integer} | {type: 1, y: string} -> {type: 0 | 1} // x, y は消える。
{type: 0, x: integer} & {type: 1, y: string} -> {type: 0 & 1, x: integer, y: string}
array[integer] | array[string] -> array[integer | string] // integer と string のどちらか、という情報は消える
```

これによって、全ての type 構文は number 階層, string, boolean, array, object の階層の union (すなわちそれぞれ交叉の無い union) になります。 それぞれ高々一つです (複数の object の union はすべてひとつに正規化されることに注意) (なお、これに加えて unknown が存在。ここは直和で表現する)

こうすれば、それぞれの階層で subtyping check するだけですが、それは容易に行えます。

例外は判別可能 union です。後述。

### 判別可能 union

判別可能 union は、特殊な object の union です。前述した case が鍵です。case が付けられるプロパティは **それぞれの object 型について高々一つ** で、それが discriminator となります。内部では特殊なオブジェクトとして保持します。

```
{case type: 0, x: integer} | {case type: 1, y: string} → DISC { discriminator: "type", mapping : { 0 : {type: 0, x: integer}, 1 : {type: 1, y: string}}}
```

このように変換できる条件は以下です。

1. すべての要素が object の union である。
2. それぞれ case プロパティを持つ。
3. それらが同一のプロパティである。
4. それらの case プロパティの値は全てリテラルである。
5. それらの case プロパティのリテラル値は全て異なる。

それ以外の場合、case は無視され通常の union として計算されます (つまり情報が落ちる)

判別可能 union とその他の型の subtyping 計算は次のように行う。

```
DISC { discriminator: d, mapping : { l1 : t1, l2 : t2,...}} <: T // T は DISC 以外
→ t1 <: T && t2 <: T && ...

T <: DISC { discriminator: d, mapping : { l1 : t1, l2 : t2,...}} // T は DISC 以外
→ T <: t1 || T <: t2 || ...

DISC { discriminator: d, mapping : { l1 : t1, l2 : t2,...}} <: DISC { discriminator: d, mapping : { l1' : t1', l2' : t2',...}}
→ discriminator が同一である必要がある。l1, l2,... は l1', l2',... に含まれる必要がある。ここで l1 == l1',... だったとし、さらに、t1 <: t1', t2 <: t2',... である必要がある。
それらが満たされなかった場合、左辺を普通の union t1 | t2 |... に変換。再び正規化し、右辺と比較する。(右辺は正規化しちゃだめ)
```

### optional field の扱い

runtime では、存在しないフィールドにアクセスすると null が帰る。したがって、x?: T にアクセスすると T | null になる。

Normalize 時は T | null と optional field は区別して考えられる。
union 計算時 → どちらか一方で optional であった場合、optional property になる。右辺はそれぞれの object の type の union になる。
intersection 計算時 → どちらも optional であった場合、optional property になる。右辺はそれぞれの object の type の intersection になる。

subtyping 計算時は、optional field は単に null を含む普通の field として扱う。つまり、x?: T は x: T | null と同じように扱って subtyping の計算をする。

JSON Schema 生成時は optional field として扱われる。

match は基本的に、各 literal pattern, 及び判別可能 union を分解できる。

### Semantic annotation

semantic annotation は JSON Schema の生成に利用される。description として入れる。
JSON Schema は以下について生成される。

## runtime (katali protocol)

runtime は基本的に、katali protocol によって動く

katali protocol に従って、それぞれのサーバーは以下のエンドポイントを何かしらの base URL 以下に提供する。これは runtime も例外ではない。
この意味で、runtime は katali protocol を実装したサーバーの一つであると言える。(他には、AI Server, discord wrapper server, cron server, sandbox server 等がある。)

意味

task ... それぞれのサーバーが提供する機能。effect が発生する可能性もある。
task execution ... task の実行インスタンス。handler を提供する実態でもある。また、待機状態には二種類あり、1. 子からの return / delegate を待機。2. 親からの continue / break を待機。これらは同時には起こらない。詳細は後述する。
delegate ... 子 execution が親 execution に対して effect の handle を要請すること。
effect ... effect 定義それ自体。これはただの型情報と effect の一意な識別のために存在する。

execution に関しては、次の 3 つの制御を投げられる。
/return ... 実行した子 execution からの結果を受け取る
/continue ... 実行した handler delegation からの結果を受け取る
/break ... 実行した handler delegation からの break を受け取る

重要な点 : それぞれ id と URL (どのサーバーに位置するか) で管理される。

- GET /effect
  - req: なし
  - res: `array[{
effect_id: string, // effect の一意な識別子
effect_where: URL, // effect 定義が位置する URL。
name: string, // effect の名前
description: string, // semantic annotation から生成された説明
arg_types: array[JSON Schema], // 引数の型の JSON Schema の配列
return_type: JSON Schema, // 返り値の型の JSON Schema
}]`
  - 提供する effect 定義を返す。(基本的には、main.ktl の pub effect 定義を返す。)
  - たとえば、cron サーバーは cron 関連の effect を返す。
- GET /task
  - req: なし (将来的に検索する可能性あり)
  - res: `array[{
task_id: string, // task の一意な識別子。なお、main.ktl に pub external fn があった場合、その id は **新たに発行される** (id は全サーバーで一意でなければいけない。)
task_where: URL, // task が位置する URL。task を実行する際にここに POST する。
name: string, // fn の名前
description: string, // semantic annotation から生成された説明
type: "fn" | "effect",
arg_types: array[JSON Schema], // 引数の型の JSON Schema の配列
return_type: JSON Schema, // 返り値の型の JSON Schema
// fn が呼ぶ可能性のある effect。これらの handler を提供する必要がある。
effects: array[
  {
    effect_id: string, // effect の一意な識別子
    effect_where: URL, // effect 定義が位置する URL。
    name: string, // effect の名前
    arg_types: array[JSON Schema], // 引数の型の JSON Schema の配列
    return_type: JSON Schema, // 返り値の型の JSON Schema
    description: string, // semantic annotation から生成された説明
  }],
}]`
  - runtime は main.ktl の pub fn を全て返す。
  - たとえば cron サーバーは cron 関連の task を返す。
- POST /execution
  - req: `{
  task_id: string, // 実行したい task の識別子
  args: array[any], // 引数の配列
  from_execution_id: string, // 親の execution の識別子。
  from_execution_where: URL, // 結果を返すべき場所。概ね、親の execution が位置する場所。
  // 提供する handler の列
  handlers: array[{
    execution_id: string, // handler を提供する execution の識別子。
    execution_where: URL, // handler を提供する execution が位置する URL。
    effects: array[{
      effect_id: string, // effect の一意な識別子
      effect_where: URL, // effect 定義が位置する URL。
      name: string, // effect の名前
      arg_types: array[JSON Schema], // 引数の型の JSON Schema の配列
      return_type: JSON Schema, // 返り値の型の JSON Schema
      description: string, // semantic annotation から生成された説明
    }],
  }]
}`
  - res: `{
  execution_id: string, // 子プロセスとして実行された execution の URL
  execution_where: URL, // execution が位置する URL。
}`
  - 親→子。**子プロセスとして**、fn を呼び出す。呼ばれた側は execution_id を生成して返す。execution を実行し始める。
- POST /execution/delegate
  - req: `{
    execution_id: string, // execution_id の識別子
    effect_id: string, // 呼び出された effect の識別子
    args: array[any], // 引数の配列
    from_execution_id: string, // handler delegation を呼び出した execution の識別 id
    from_execution_where: URL, // 結果を返すべき場所。概ね、handler delegation を呼び出した execution が位置する場所。
}`
  - res: `{
    success: boolean,
}`
  - 子→親。親が提供する handler を呼び出す
    - たとえば execution A (effect X に対する handler を提供) -> execution B と階層がある。この時 B が実行中で A は停止中である。
    - B が X を呼び出したいとき、与えられた handler の中から X を探す。A が提供しているので、A に POST /delegate する。B は停止し、A からの continue / break を待つ。
- POST /execution/continue
  - req `{
  result: any, // handler delegation が continue されたときの結果
  from_exectuion_id: string, // 親 execution の識別 id
  from_execution_where: URL, // 親 execution が位置している
  execution_id: string, // handler delegation を呼び出した execution の識別 id
}`
  - res: {
    success: boolean,
    }
  - 親→子。handler delegation の実行が終わり、その結果が continue されたとき、呼び出した execution に結果を返す。
- POST /execution/break
  - req `{
  from_exectuion_id: string, // 親 execution の識別 id
  from_execution_where: URL, // 親 execution が位置している
  execution_id: string, // handler delegation を呼び出した execution の識別 id
}`
  - res: {
    success: boolean,
    }
  - 親→子。handler delegation の実行が終わり、その結果が break されたとき。呼び出した execution に break を伝える。
- POST /execution/return
  - req `{
  result: any, // 子 execution が return したときの結果
  from_execution_id: string, // 子 execution の識別子
  from_execution_where: URL, // 子 execution が位置していた場所
  execution_id: string, // 子 execution を呼び出した execution の識別 id
}`
  - res: {
    success: boolean,
    }
  - 子→親。子 execution の実行が終わり、その結果が return されたとき、呼び出した execution に結果を返す。
- POST /execution/error
  - req `{
  error: string, // エラーメッセージ
  from_execution_id: string, // エラーが起きた execution の識別子
  from_execution_where: URL, // エラーが起きた execution が位置していた場所
  execution_id: string, // 子 execution を呼び出した execution の識別 id
}`
  - res: {
    success: boolean,
    }
  - 子→親。子 execution の実行がエラーになったとき、呼び出した execution にエラーを伝える。

基本的な動き

1. 各サーバーは POST /execution を待機。
2. 受け取ったとき、execution を実行して、その id を返す。
3. 内部で別の execution を実行したいときは、他のサーバー (若しくは自分自身のサーバー) の POST /execution に POST する。このとき、handler を提供することもできる。
   handler としては、上層から受け取った handler + 自身で提供する handler を提供。execution を実行したら、(並列実行出ない限り) 停止し、return / delegate POST を待つ。以下の3つを待機する。
   - POST /delegate を待機。呼び出されたら handler を実行。その中で、さらに execution を呼び出したかったら、同様に POST /execution。この時 from_execution_id は **handler が位置する execution** の id になることに注意。また、提供する handler も上記と同様のルール。
     - handler の実行が終わったら、結果を continue するか、break するかを選ぶ。continue する場合は POST /execution/continue に、break する場合は POST /execution/break に POST する。
   - POST /execution/return を待機。呼び出されたら、結果を受け取って、実行を再開する。
   - POST /execution/error を待機。呼び出されたら、エラーを受け取って、実行を停止する。親にエラーを伝えるために、POST /execution/error に POST する。
4. 内部で effect を呼び出したいときは、提供された handler に該当 effect がある場合は、それを提供する execution_id と URL に対して /delegate を呼び出す。ない場合はエラー。(プロトコルに従い親にエラーを伝播。)。
   - continue されたときの動作 … POST /execution/continue を待機。呼び出されたら、結果を受け取って、実行を再開する。
   - break されたときの動作 … POST /execution/break を待機。呼び出されたら、実行を停止する。これが子プロセスからの delegation の最中であった場合は、その子プロセスに対しても break を即座に伝える。
     - 例:
       - execution A (effect X に対する handler を提供) -> execution B (effect Y に対する handler を提供) -> execution C と階層がある。この時 C が動いていて、A, B は停止している。C が B に対して effect Y の実行を delegation し、また、B はその中で A に対して effect X の実行を delegation したとする。このとき、B, C は親からの continue / break 待機中で、A は delegation を実行中である。A が break を発行した場合、B はそれを受け取るが、B も C からの delegation 中であるため、B は C にも break を投げる。こうしてすべてのプロセスが停止する。
5. 実行が終わったら、結果を親に伝えるために POST /execution/return に POST する。エラーになった場合は、POST /execution/error に POST する。

cron.ktl

```
@"notify effect。現在時刻をパラメータとして持つ。"
external effect notify(time: string) -> null from "cron_server:notify"

@"一定間隔でnotify effectを発行するタスク。cron 形式の文字列を引数に取る。"
pub external fn schedule(cron: string) -> null with notify from "cron_server:schedule"
```

main.ktl

```
import lib.cron

@"1 日毎に log に現在時刻を出力 30 日で終了。"
pub fn daialy_time_log() -> null {
  handle {
    var count = 0
    case cron.notify(time: string) => {
      log(time)
      if (count >= 30) {
        break null
      } else {
        continue null with {
          count = count + 1
        }
      }
    }
  }
  cron.schedule("0 0 * * *") // これは毎日 0 時に実行される cron 形式の文字列
}
```

### 並列実行

並列実行のための構文を用意。

expression の一種。戻り値は、それぞれの戻り値の union の配列。parallel の右辺は構文であって配列ではない。

```
parallel [
  {
    s
    ...
    e,
  },
  {
    s
    ...
    e,
  },
  ...
]
```

parallel 実行時の動作。まず、内部的にはそれぞれの block は独立した task (environment を変数として受け取る) として扱われる。

parallel が実行されると、それぞれの task に対して同時に POST /execution する。これらは全て子プロセスとして実行される。
