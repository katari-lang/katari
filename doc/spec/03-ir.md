## 1. 概要

Katari IR v0.2 は **Thread** を基本実行単位とする中間表現である。ソース言語の `handle` / `for` / `par` / `block` 式を全て Thread 分解によって統一的に扱う。

### 設計目標

| 目標 | 説明 |
|------|------|
| **永続化** | Thread の yield 点 (`ICall`, `IPar`, `IRequest`, `IHandle`, `IFor`) で実行状態をスナップショットし、外部ストレージに保存できる。サーバー再起動後に状態を復元し実行を再開可能。 |
| **分散実行** | yield 点がプロトコル境界と対応するため、ネットワーク越しの中断・再開が容易。agent 呼び出しは独立したコンテキストで実行されるため、異なるノードに分散配置できる。 |
| **単純さ** | 最適化なし。基本ブロック (basic block) への分割なし。Thread 内の命令列はフラットに連続し、制御フローは絶対オフセットジャンプで表現する。 |

### v0.1 からの主な変更

旧 IR (v0.1) では handle body が暗黙的 (`IHandleBegin`/`IHandleEnd` マーカー間) であり、Lowering に先読みが必要だった。par は擬似 agent + captured vars リストで表現されアドホックだった。v0.2 ではこれらを全て Thread 分解に統一し、以下の構造変更を行った。

- `IHandleBegin` / `IHandleEnd` を廃止し `IHandle` + `IRHandleDef` に統合
- `IForBegin` / `IForEnd` を廃止し `IFor` + `IRForDef` に統合
- `IPar` の引数を `[(AgentId, [VarId])]` から `[ThreadId]` に変更 (captured vars 不要)
- Thread 正常完了を示す `IComplete` を新設
- `IReturn` の意味を変更 (ソースの `return` 文に対応し FN_BODY まで巻き上げ)
- `IContinue` / `IHandleBreak` / `IForContinue` / `IForBreak` から HandlerId/ForId/slot index を除去 (構造的に一意に決定されるため)

## 2. Thread モデル

### 概念

Thread は独立した命令列であり、IR の基本実行単位である。1 つのモジュールは複数の Thread を持つ。各 Thread は `ThreadKind` によって用途と境界の意味が定義される。

Thread は自身で命令列を最後まで実行するか、Signal を発行して完了する。Thread 内部にはループ構造は存在せず、ループやハンドラのディスパッチはランタイムが Thread を繰り返し起動することで実現する。

### ThreadKind (7 種)

| Kind | 用途 | params | 終端命令 |
|------|------|--------|----------|
| `FN_BODY` | agent エントリポイント。agent が呼び出されると、このスレッドが起動される。 | agent の仮引数 (`[VarId]`) | `IComplete` または `IReturn` |
| `BLOCK` | `par` の各ブランチ。独立した並列実行単位。 | なし (`[]`) | `IComplete` または `IReturn` |
| `HANDLER_TARGET` | `handle` の本体。`handle` 文以降の残り文がここに切り出される。 | なし (`[]`) | `IComplete`, `IReturn`, `IHandleBreak` |
| `REQUEST_HANDLER` | request case ハンドラ。request が到着したときにランタイムが起動する。 | request の引数 (`[VarId]`) | `IComplete` (Normal で返す), `IContinue`, `IHandleBreak`, `IReturn` |
| `HANDLE_THEN` | handle の then 節。target が Normal 完了したとき実行される。 | 入力値 1 つ (`[VarId]`) | `IComplete` または `IReturn` |
| `FOR_BODY` | for ループ本体。各イテレーションでランタイムが起動する。 | element 変数 (`[VarId]`) | `IForContinue`, `IForBreak`, `IReturn` |
| `FOR_THEN` | for の then 節。全イテレーション完了後に実行される。 | なし (`[]`) | `IComplete` または `IReturn` |

### データ構造

```
IRThread {
  itId:     ThreadId    -- モジュール内でユニークな ID
  itKind:   ThreadKind  -- 上記 7 種のいずれか
  itParams: [VarId]     -- パラメータ変数 (ランタイムが値を充填する)
  itBody:   [Instruction]  -- フラットな命令列
}
```

**対応する Haskell 定義** (`Katari.IR`):

```haskell
data ThreadKind
  = TkFnBody
  | TkBlock
  | TkHandlerTarget
  | TkRequestHandler
  | TkHandleThen
  | TkForBody
  | TkForThen

data IRThread = IRThread
  { itId     :: ThreadId
  , itKind   :: ThreadKind
  , itParams :: [VarId]
  , itBody   :: [Instruction]
  }
```

## 3. 変数コンテキスト (Scope Chain)

### VarId のユニーク性

`VarId` (`Word32`) はモジュール全体でグローバルにユニークである。同じ `VarId` が異なる Thread に出現した場合、それは同一の変数を指す。コンパイラは `freshVar` で連番を割り当てるため、ID の衝突は発生しない。

### Scope Chain による参照

子 Thread は親の変数コンテキストを scope chain で暗黙的に参照できる。Thread は独自のパラメータ変数と、Thread 内で新たに割り当てた変数を持つ。それに加えて、親スレッドの変数に対してはそのまま `VarId` で読み取りアクセスが可能である。

```
FN_BODY (thread_0):        vars = {v0, v1, v2}
  └─ HANDLER_TARGET (thread_1): vars = {v3, v4}
       ├── v3 は thread_1 のローカル変数
       ├── v0, v1, v2 は scope chain で参照可能
       └── v0 を使う命令: IMove v3 v0  (合法)
```

ランタイムは Thread の起動時に、親の変数コンテキストのスナップショットまたは参照を子に引き継ぐ必要がある。具体的な実装手法 (フラット化、リンクリスト等) はランタイムの裁量に委ねる。

### Agent 呼び出しとコンテキスト境界

`ICall` による agent 呼び出しはコンテキスト境界を形成する。呼び出された agent は完全に独立したフレッシュな変数コンテキストで実行される。親の変数を scope chain で参照することはできない。agent 間の値の受け渡しは引数 (`ICall dst agentId [args]`) と返り値 (`IComplete val` / `IReturn val`) のみで行われる。

```
FN_BODY (agent A):  vars = {v0, v1}
  │
  ICall v2, agentB, [v0]
  │
  └─ FN_BODY (agent B): vars = {v3}  ← v3 = v0 の値のコピー
       v1 は参照不可 (コンテキスト境界)
```

### Mutation と Signal Return Map

Thread 内で変数を直接書き換えることは可能だが、親の変数を変更するメカニズムは Signal の mutation map に限定される。

- `IContinue val [(stateVar, newVal)]` — request handler から handle への state 更新
- `IForContinue [(stateVar, newVal)]` — for body からのループ state 更新

これらの mutation map `[(VarId, VarId)]` は「state 変数の VarId」と「新しい値を保持する VarId」のペアである。ランタイムは Signal を受け取った時点で、指定された state 変数の値を更新する。

## 4. Signal システム

Thread の完了方法は Signal によって分類される。Signal は Thread がどのように終了し、制御をどこに返すかを定義する。全 6 種。

### Signal 一覧

| Signal | 発行命令 | 伝搬先 | セマンティクス |
|--------|----------|--------|---------------|
| **Normal(value)** | `IComplete val` | 直接の親 | Thread が正常に完了した。`val` が結果値として親に返される。 |
| **FnReturn(value)** | `IReturn val` | FN_BODY まで巻き上げ | ソースの `return` 文に対応。現在の Thread を含む、FN_BODY に到達するまでの全ての中間 Thread を打ち切り、agent の返り値として `val` を返す。 |
| **HandleBreak(value)** | `IHandleBreak val` | handle scope を脱出 | `handle` のスコープ全体を脱出する。`val` が `IHandle dst` の `dst` に格納される。HANDLER_TARGET 内および REQUEST_HANDLER 内から発行可能。 |
| **Continue(value, mutations)** | `IContinue val [(stateVar, newVal)]` | request handler --> handle | request handler が request に応答する。`val` は request 発行元の `IRequest dst` の `dst` に格納される。`mutations` で handle の state 変数を更新する。その後 HANDLER_TARGET の実行を再開する。 |
| **ForBreak(value)** | `IForBreak val` | for loop を脱出 | for ループ全体を脱出する。`val` が `IFor dst` の `dst` に格納される。then 節は実行されない。 |
| **ForContinue(mutations)** | `IForContinue [(stateVar, newVal)]` | for body --> for loop | 現在のイテレーションを終了し、`mutations` で state 変数を更新した上で次のイテレーションに進む。 |

### Signal の伝搬規則

**Normal**: 直接の親のみに伝搬する。handle の HANDLER_TARGET が Normal 完了した場合、then 節があれば then 節が実行される。then 節がなければ HANDLER_TARGET の結果値がそのまま handle の結果になる。

**FnReturn**: 現在の Thread から FN_BODY に到達するまで、全ての中間構造 (handle scope, for loop 等) をスキップして巻き上げる。FN_BODY の結果値として扱われる。ランタイムは巻き上げの過程で、途中の handle scope 内で生成された子 agent に terminate を送信する必要がある。

**HandleBreak**: `handle` スコープから脱出する。HANDLER_TARGET 内の `break` 文、および REQUEST_HANDLER 内の `break` 文から発行される。handle の `IHandle dst` の `dst` に `val` が格納される。then 節は実行されない。HANDLER_TARGET が待機中 (`ICall`/`IPar`/`IRequest` で suspend) であった場合、ランタイムはスコープ内の子 agent に terminate を送信する。

**Continue**: REQUEST_HANDLER 専用。request 発行元 (HANDLER_TARGET 内で `IRequest` を実行した地点) への reply として機能する。`val` が `IRequest dst` の `dst` に格納される。`mutations` で state 変数が更新された後、HANDLER_TARGET は中断地点から実行を再開する。

**ForBreak**: for ループ全体から脱出する。FOR_BODY 内で発行される。`val` が `IFor dst` の `dst` に格納される。then 節は実行されない。

**ForContinue**: FOR_BODY 専用。現在のイテレーションの完了を示す。`mutations` で state 変数を更新し、ランタイムは次の要素で FOR_BODY を再起動する。全要素を処理し終えた場合、then 節があれば実行する。

## 5. データ構造

以下のデータ構造は `Katari.IR` モジュールに正確に対応する。

### 識別子型

全ての識別子は `Word32` (unsigned 32-bit integer) である。

| 型名 | 用途 | スコープ |
|------|------|---------|
| `VarId` | 変数 | モジュール全体でユニーク |
| `AgentId` | agent 定義 | モジュール全体でユニーク |
| `RequestId` | request 定義 | モジュール全体でユニーク |
| `ConstId` | 定数プールのインデックス | モジュールローカル |
| `HandlerId` | handle 定義 | モジュール全体でユニーク |
| `ForId` | for 定義 | モジュール全体でユニーク |
| `ThreadId` | thread | モジュール全体でユニーク |

### IRModule

モジュール全体を表すトップレベル構造。コンパイラは全ソースモジュールを 1 つの `IRModule` にマージして出力する。

```
IRModule {
  irmName:      Text            -- モジュール名
  irmNameTable: NameTable       -- デバッグ用名前テーブル
  irmConsts:    [ConstVal]      -- 定数プール (インデックス = ConstId)
  irmRequests:  [IRRequestDef]  -- request 定義一覧
  irmThreads:   [IRThread]      -- 全 Thread
  irmHandles:   [IRHandleDef]   -- handle 定義一覧
  irmFors:      [IRForDef]      -- for 定義一覧
  irmAgents:    [IRAgentDef]    -- agent 定義一覧
}
```

### IRThread

```
IRThread {
  itId:     ThreadId
  itKind:   ThreadKind
  itParams: [VarId]
  itBody:   [Instruction]
}
```

### IRHandleDef

handle 式のコンパイル結果。1 つのソース `handle` 文に対して 1 つの `IRHandleDef` が生成される。

```
IRHandleDef {
  ihdId:         HandlerId          -- この handle 定義のユニーク ID
  ihdStateVars:  [VarId]            -- state 変数の VarId リスト
  ihdStateInits: [VarId]            -- 初期値を保持する VarId リスト (親 Thread で計算済み)
  ihdBody:       ThreadId           -- HANDLER_TARGET thread
  ihdReqCases:   [(RequestId, ThreadId)]  -- (request ID, REQUEST_HANDLER thread) のリスト
  ihdThen:       Maybe ThreadId     -- HANDLE_THEN thread (なければ Nothing)
}
```

**補足**:
- `ihdStateVars` と `ihdStateInits` は同じ長さ。`ihdStateInits[i]` の値で `ihdStateVars[i]` を初期化する。
- `ihdStateInits` の各 VarId は、`IHandle` を実行する親 Thread 内で既に値が計算されている変数である。

### IRForDef

for 式のコンパイル結果。1 つのソース `for` 式に対して 1 つの `IRForDef` が生成される。

```
IRForDef {
  ifdId:         ForId              -- この for 定義のユニーク ID
  ifdIterVars:   [VarId]            -- element 変数 (let x of arr の x)
  ifdArrays:     [VarId]            -- イテレーション対象の配列を保持する VarId
  ifdStateVars:  [VarId]            -- state 変数の VarId リスト
  ifdStateInits: [VarId]            -- 初期値を保持する VarId リスト
  ifdBody:       ThreadId           -- FOR_BODY thread
  ifdThen:       Maybe ThreadId     -- FOR_THEN thread (なければ Nothing)
}
```

**補足**:
- `ifdIterVars` と `ifdArrays` は同じ長さ。`ifdArrays[i]` の配列の各要素が `ifdIterVars[i]` に充填される。
- ループ制御 (index 管理、配列長比較、element 抽出) は全てランタイムが担当する。IR にはループ命令は出現しない。

### IRAgentDef

agent の定義。ソースの `agent` 宣言に対応する。

```
IRAgentDef {
  iadId:    AgentId    -- この agent のユニーク ID
  iadName:  Text       -- 修飾名 (例: "main.myAgent")
  iadEntry: ThreadId   -- FN_BODY thread の ID
}
```

### IRRequestDef

request の定義。ソースの `request` 宣言および `external request` 宣言に対応する。

```
IRRequestDef {
  irReqId:   RequestId    -- この request のユニーク ID
  irReqName: Text         -- 修飾名 (例: "main.get_value")
  irReqFrom: Maybe Text   -- external の場合は提供元 (例: "server:name")、ローカルは Nothing
}
```

### NameTable

デバッグ・永続化用の名前テーブル。ランタイムの動作には必須ではないが、デバッグ出力やシリアライズ時の可読性向上に使用する。

```
NameTable {
  ntVars:     Map VarId Text       -- 変数名
  ntAgents:   Map AgentId Text     -- agent の修飾名
  ntRequests: Map RequestId Text   -- request の修飾名
}
```

### ConstVal

定数プールに格納される値。5 種。

```
ConstVal
  = CVNull                -- null
  | CVBool Bool           -- true / false
  | CVInt Integer         -- 任意精度整数
  | CVNum Double          -- IEEE 754 倍精度浮動小数点
  | CVStr Text            -- UTF-8 文字列
```

## 6. Handle 分解

### 概念

ソースの `handle` 文は、コンパイラによって以下の要素に分解される:

1. 親 Thread 内の state 初期化命令 + `IHandle` 命令
2. `HANDLER_TARGET` thread — handle 文以降の残り文
3. 各 request case に対応する `REQUEST_HANDLER` thread
4. (任意) `HANDLE_THEN` thread — then 節

### ソース例

```katari
agent example() -> string {
  let a = 0
  handle (counter: integer = 0) {
    request get_count() => {
      continue counter with { counter = counter + 1 }
    }
  } then (result) {
    f"done: ${result}, count=${counter}"
  }
  let c1 = get_count()
  let c2 = get_count()
  c2
}
```

### IR 出力

```
// 定数プール
const[0] = CVInt 0

// thread_0: FN_BODY [params: v0 (なし — example は引数なし)]
//   v1 = 0               (let a = 0)
//   v2 = 0               (counter の初期値)
//   v3 = IHandle hnd_0   (handle 実行 → 結果が v3 に入る)
//   IComplete v3
thread_0 (FN_BODY, params=[]):
  ILoadConst  v1, const[0]       // a = 0
  ILoadConst  v2, const[0]       // counter_init = 0
  IHandle     v3, hnd_0          // handle 実行
  IComplete   v3                 // agent 終了

// thread_1: HANDLER_TARGET [params: なし]
//   handle body = handle 文以降の残り文
thread_1 (HANDLER_TARGET, params=[]):
  IRequest    v4, req_get_count, []   // c1 = get_count()
  IRequest    v5, req_get_count, []   // c2 = get_count()
  IComplete   v5                      // 結果を返す

// thread_2: REQUEST_HANDLER [params: なし — get_count は引数なし]
//   request get_count() => { ... }
thread_2 (REQUEST_HANDLER, params=[]):
  IMove       v6, v_counter           // counter の現在値を参照
  ILoadConst  v7, const[0]            // 1 (実際には別の const)
  IAdd        v8, v_counter, v7       // counter + 1
  IContinue   v6, [(v_counter, v8)]   // continue counter with { counter = counter + 1 }

// thread_3: HANDLE_THEN [params: v9 (= result)]
//   then (result) { ... }
thread_3 (HANDLE_THEN, params=[v9]):
  ...                                 // f"done: ${result}, count=${counter}" の命令列
  IComplete   v10                     // then 節の結果

// handle_def[hnd_0]:
//   stateVars  = [v_counter]
//   stateInits = [v2]               // v2 は thread_0 で計算済み
//   body       = thread_1
//   reqCases   = [(req_get_count, thread_2)]
//   then       = Just thread_3
```

### ランタイム実行セマンティクス

ランタイムが `IHandle dst hid` を実行したとき、以下の手順で処理する:

1. **State 初期化**: `IRHandleDef.ihdStateInits` の各 VarId から値を読み取り、対応する `IRHandleDef.ihdStateVars` に設定する。

2. **HANDLER_TARGET 実行**: `IRHandleDef.ihdBody` で指定された Thread を起動する。

3. **Request ディスパッチ**: HANDLER_TARGET の実行中に `IRequest` が到達すると、ランタイムは `ihdReqCases` から対応する `RequestId` を検索し、対応する REQUEST_HANDLER thread を起動する。request の引数は Thread の params に充填する。

4. **Signal 処理**:
   - **Continue(val, mutations)**: `val` を `IRequest` の `dst` に格納する。`mutations` に基づき state 変数を更新する。HANDLER_TARGET の実行を `IRequest` の次の命令から再開する。
   - **HandleBreak(val)**: `val` を `IHandle` の `dst` に格納する。then 節は実行しない。handle を完了させる。
   - **Normal(val)** (HANDLER_TARGET が `IComplete` に到達):
     - then 節あり: `val` を HANDLE_THEN の params に充填して実行する。HANDLE_THEN の結果が `IHandle` の `dst` に格納される。
     - then 節なし: `val` をそのまま `IHandle` の `dst` に格納する。
   - **FnReturn(val)**: handle scope を含む全てを打ち切り、FN_BODY まで巻き上げる。

5. **結果格納**: 最終的に得られた値が `IHandle` の `dst` に格納され、親 Thread の次の命令に進む。

### 重要な注意事項

- REQUEST_HANDLER thread から `IComplete` (Normal signal) が発行された場合、それは `IContinue` と同等に扱い、state 更新なしで HANDLER_TARGET を再開する。ただしこの動作はソース言語の仕様上は直接到達しない (request handler は必ず `continue` または `break` で終了する)。
- HANDLER_TARGET 内で `IHandleBreak` が発行された場合も有効であり、handle body 内からの `break` に対応する。
- state 変数は scope chain で REQUEST_HANDLER と HANDLE_THEN の両方から参照可能である。

## 7. For 分解

### 概念

ソースの `for` 式は、コンパイラによって以下の要素に分解される:

1. 親 Thread 内の配列評価 + state 初期化 + `IFor` 命令
2. `FOR_BODY` thread — ループ本体
3. (任意) `FOR_THEN` thread — then 節

**IR にはループ命令は一切存在しない**。index 管理、配列長比較、element 抽出は全てランタイムが担当する。

### ソース例

```katari
agent sum_array() -> integer {
  for (let x of [1, 2, 3, 4, 5], var acc: integer = 0) {
    if x > 2 {
      continue with { acc = acc + x }
    }
    continue with { acc = acc }
  } then {
    acc
  }
}
```

### IR 出力

```
// 定数プール
const[0] = CVInt 1
const[1] = CVInt 2
const[2] = CVInt 3
const[3] = CVInt 4
const[4] = CVInt 5
const[5] = CVInt 0

// thread_0: FN_BODY
thread_0 (FN_BODY, params=[]):
  INewArray    v_arr, [v1, v2, v3, v4, v5]   // 配列 [1,2,3,4,5] を構築
  ILoadConst   v_acc_init, const[5]           // acc の初期値 = 0
  IFor         v_dst, for_0                   // for ループ実行
  IComplete    v_dst

// thread_1: FOR_BODY [params: v_x]
//   ランタイムが v_x に現在の要素値を充填
thread_1 (FOR_BODY, params=[v_x]):
  ILoadConst   v6, const[1]                  // 2
  ICmpGt       v7, v_x, v6                   // x > 2
  IBranch      v7, @then, @else
@then:
  IAdd         v8, v_acc, v_x                // acc + x
  IForContinue [(v_acc, v8)]                 // state 更新して次へ
@else:
  IForContinue [(v_acc, v_acc)]              // state 据え置き

// thread_2: FOR_THEN [params: なし]
thread_2 (FOR_THEN, params=[]):
  IMove        v9, v_acc                     // acc の最終値
  IComplete    v9

// for_def[for_0]:
//   iterVars   = [v_x]
//   arrays     = [v_arr]
//   stateVars  = [v_acc]
//   stateInits = [v_acc_init]
//   body       = thread_1
//   then       = Just thread_2
```

### ランタイム実行セマンティクス

ランタイムが `IFor dst fid` を実行したとき:

1. **State 初期化**: `IRForDef.ifdStateInits` から値を読み取り、`ifdStateVars` に設定する。

2. **イテレーション制御** (ランタイムの責務):
   - `ifdArrays` の配列を取得し、長さを確認する。
   - index を 0 から開始し、各イテレーションで:
     a. 全配列の `index` 番目の要素を取得し、対応する `ifdIterVars` に充填する。
     b. FOR_BODY thread を起動する。

3. **Signal 処理**:
   - **ForContinue(mutations)**: `mutations` で state 変数を更新し、index を進めて次のイテレーションを実行する。
   - **ForBreak(val)**: ループを即座に終了し、`val` を `IFor` の `dst` に格納する。then 節は実行しない。
   - **FnReturn(val)**: for ループを含む全てを打ち切り、FN_BODY まで巻き上げる。

4. **全要素処理完了後**:
   - then 節あり: FOR_THEN thread を起動する。FOR_THEN は scope chain で state 変数の最終値を参照可能。FOR_THEN の結果が `IFor` の `dst` に格納される。
   - then 節なし: `null` を `IFor` の `dst` に格納する。

5. **複数配列のイテレーション**: `ifdArrays` に複数の配列が指定されている場合、全配列を同時にイテレートする (zip 動作)。イテレーション回数は最短の配列の長さで決定される。

## 8. Par 分解

### 概念

ソースの `par` 式は、複数の `BLOCK` thread に分解される。各ブランチは独立した Thread として表現され、親は `IPar dst [ThreadId]` で全 Thread の並列実行と完了待機を行う。

旧 IR (v0.1) では各 par ブランチが擬似 agent として表現され、captured vars を明示的に引数として渡す必要があったが、v0.2 では scope chain による参照があるため captured vars リストは不要になった。

### ソース例

```katari
agent parallel_work() -> array[string] {
  let prefix = "result"
  par [
    {
      let x = 10
      f"${prefix}: ${x}"
    },
    {
      let y = 20
      f"${prefix}: ${y}"
    }
  ]
}
```

### IR 出力

```
// thread_0: FN_BODY
thread_0 (FN_BODY, params=[]):
  ILoadConst   v_prefix, const[...]          // "result"
  IPar         v_dst, [thread_1, thread_2]   // 並列実行
  IComplete    v_dst

// thread_1: BLOCK [params: なし]
thread_1 (BLOCK, params=[]):
  ILoadConst   v_x, const[...]               // 10
  ...                                         // f"${prefix}: ${x}" の命令列
  // v_prefix は scope chain で参照 (親 thread_0 の変数)
  IComplete    v_result1

// thread_2: BLOCK [params: なし]
thread_2 (BLOCK, params=[]):
  ILoadConst   v_y, const[...]               // 20
  ...                                         // f"${prefix}: ${y}" の命令列
  // v_prefix は scope chain で参照
  IComplete    v_result2
```

### ランタイム実行セマンティクス

ランタイムが `IPar dst [tid1, tid2, ...]` を実行したとき:

1. 指定された全 Thread を並列に起動する。各 Thread は親の変数コンテキストを scope chain で参照可能。

2. 全 Thread の完了を待機する。各 Thread は `IComplete` で Normal signal を返す。

3. 各 Thread の結果値を、Thread リストの順序で配列に格納し、`dst` に設定する。

4. いずれかの Thread が `IReturn` (FnReturn signal) を発行した場合、他の全 Thread に terminate を送信し、FN_BODY まで巻き上げる。

## 9. Match (パターンマッチ)

### 概念

`match` 式は **Thread 分解を行わない**。現在の Thread 内でインラインに展開される。パターンの検査には `ICmpEq`, `IGetField`, `ITypeOf`, `IArrGet` 等を使用し、分岐には `IBranch` / `IJump` を使用する。

### ソース例

```katari
agent classify(x: integer | string | { kind: string, value: integer }) -> string {
  match x {
    case integer(n) => { f"int: ${n}" }
    case string(s) => { s }
    case { kind = "circle", value = v } => { f"circle: ${v}" }
    case _ => { "unknown" }
  }
}
```

### IR 出力 (概略)

```
thread_0 (FN_BODY, params=[v_x]):
  // ── case 1: integer(n) ──
  ITypeOf      v1, v_x                       // typeof(x)
  ILoadConst   v2, const["integer"]
  ICmpEq       v3, v1, v2                    // typeof(x) == "integer" ?
  IBranch      v3, @match1, @next1

@match1:
  // n = x (型タグが一致したので v_x をそのまま n として使う)
  ...                                         // f"int: ${n}" の命令列
  IMove        v_dst, v_result1
  IJump        @end

@next1:
  // ── case 2: string(s) ──
  ITypeOf      v4, v_x
  ILoadConst   v5, const["string"]
  ICmpEq       v6, v4, v5
  IBranch      v6, @match2, @next2

@match2:
  // s = x
  IMove        v_dst, v_x
  IJump        @end

@next2:
  // ── case 3: { kind = "circle", value = v } ──
  IGetField    v7, v_x, const["kind"]        // x.kind
  ILoadConst   v8, const["circle"]
  ICmpEq       v9, v7, v8                    // x.kind == "circle" ?
  IBranch      v9, @bind3, @next3

@bind3:
  IGetField    v_v, v_x, const["value"]      // v = x.value
  ...                                         // f"circle: ${v}" の命令列
  IMove        v_dst, v_result3
  IJump        @end

@next3:
  // ── default case ──
  ILoadConst   v_dst, const["unknown"]
  IJump        @end

@end:
  IComplete    v_dst
```

### パターン種別と IR 変換

| パターン | IR での検査方法 |
|----------|----------------|
| `PVar n` | 無条件マッチ。`n` に scrutinee をバインド。`IJump @match` |
| `PTyped n ty` | `PVar` と同様 (型検査はコンパイル時に完了済み) |
| `PLit lit` | `ILoadConst` + `ICmpEq` + `IBranch` |
| `PTag tag n` | `ITypeOf` + `ILoadConst` + `ICmpEq` + `IBranch`。タグ文字列は `"boolean"`, `"integer"`, `"number"`, `"string"` |
| `PObj fields` | discriminator フィールドがあれば先に `IGetField` + `ICmpEq` で判定。その後各フィールドを `IGetField` でバインド |
| `PArr pats` | 各要素を `ILoadConst`(index) + `IArrGet` で取得し、再帰的にパターン検査 |

## 10. 命令セット (全 40 命令)

全命令のリファレンス。引数の型は `V` = `VarId`, `C` = `ConstId`, `T` = `ThreadId`, `A` = `AgentId`, `R` = `RequestId`, `H` = `HandlerId`, `F` = `ForId`, `u32` = `Word32` を表す。

### 10.1 定数・移動 (3 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `ILoadConst` | `dst:V, cid:C` | 定数プール `consts[cid]` の値を `dst` にロードする。 |
| `ILoadNull` | `dst:V` | `null` を `dst` にロードする。 |
| `IMove` | `dst:V, src:V` | `src` の値を `dst` にコピーする。 |

### 10.2 Object (4 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `INewObject` | `dst:V, fields:[(C, V)]` | 新しいオブジェクトを構築する。`fields` は `(フィールド名の ConstId, 値の VarId)` のペアリスト。 |
| `IGetField` | `dst:V, obj:V, field:C` | `obj` のフィールド `field` (ConstId で参照される文字列名) の値を `dst` に取得する。 |
| `ISetField` | `newObj:V, obj:V, field:C, val:V` | `obj` のフィールド `field` を `val` で更新した**新しいオブジェクト**を `newObj` に格納する。元の `obj` は変更されない (純粋操作)。 |
| `IHasField` | `dst:V, obj:V, field:C` | `obj` にフィールド `field` が存在するかを boolean として `dst` に格納する。 |

### 10.3 Array (5 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `INewArray` | `dst:V, elems:[V]` | `elems` の各 VarId の値から新しい配列を構築し `dst` に格納する。 |
| `IArrGet` | `dst:V, arr:V, idx:V` | `arr[idx]` の値を `dst` に格納する。`idx` は integer 値。 |
| `IArrLen` | `dst:V, arr:V` | `arr` の長さ (integer) を `dst` に格納する。 |
| `IArrPush` | `dst:V, arr:V, elem:V` | `arr` の末尾に `elem` を追加した**新しい配列**を `dst` に格納する。元の `arr` は変更されない。 |
| `IArrSlice` | `dst:V, arr:V, start:V, end:V` | `arr[start..end]` のスライスを**新しい配列**として `dst` に格納する。`start`, `end` は integer 値。 |

### 10.4 算術演算 (6 命令)

ランタイムが両オペランドの実行時型を判定し、integer 同士なら整数演算、それ以外なら number (f64) 演算を行う。静的型情報は IR に含まれない。

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `IAdd` | `dst:V, lhs:V, rhs:V` | 加算。integer + integer = integer, それ以外 = number。 |
| `ISub` | `dst:V, lhs:V, rhs:V` | 減算。同上。 |
| `IMul` | `dst:V, lhs:V, rhs:V` | 乗算。同上。 |
| `IDiv` | `dst:V, lhs:V, rhs:V` | 除算。結果は常に number (f64)。 |
| `IMod` | `dst:V, lhs:V, rhs:V` | 剰余。integer 同士なら integer、それ以外は number。 |
| `INeg` | `dst:V, src:V` | 符号反転。integer → integer, number → number。 |

### 10.5 比較 (6 命令)

結果は常に boolean。

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `ICmpEq` | `dst:V, lhs:V, rhs:V` | `lhs == rhs` (構造的等値比較) |
| `ICmpNe` | `dst:V, lhs:V, rhs:V` | `lhs != rhs` |
| `ICmpLt` | `dst:V, lhs:V, rhs:V` | `lhs < rhs` |
| `ICmpLe` | `dst:V, lhs:V, rhs:V` | `lhs <= rhs` |
| `ICmpGt` | `dst:V, lhs:V, rhs:V` | `lhs > rhs` |
| `ICmpGe` | `dst:V, lhs:V, rhs:V` | `lhs >= rhs` |

### 10.6 論理演算 (3 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `IAnd` | `dst:V, lhs:V, rhs:V` | 論理 AND。両方が truthy なら `rhs` を、そうでなければ falsy な方を返す。 |
| `IOr` | `dst:V, lhs:V, rhs:V` | 論理 OR。`lhs` が truthy なら `lhs` を、そうでなければ `rhs` を返す。 |
| `INot` | `dst:V, src:V` | 論理 NOT。boolean を反転する。 |

### 10.7 文字列・型変換 (3 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `IConcat` | `dst:V, lhs:V, rhs:V` | 結合演算。ランタイムが `lhs` の実行時型で分岐する。string なら文字列結合、array なら配列結合。結果は新しい値 (純粋)。 |
| `IToString` | `dst:V, src:V` | `src` を文字列表現に変換して `dst` に格納する。テンプレートリテラル (`f"..."`) の補間で使用される。 |
| `ITypeOf` | `dst:V, src:V` | `src` のランタイム型タグを文字列として `dst` に格納する。返り値は `"null"`, `"integer"`, `"number"`, `"boolean"`, `"string"`, `"array"`, `"object"` のいずれか。 |

### 10.8 制御フロー (5 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `IJump` | `target:u32` | 命令列内の絶対位置 `target` に無条件ジャンプする。 |
| `IBranch` | `cond:V, trueTarget:u32, falseTarget:u32` | `cond` が truthy なら `trueTarget` に、falsy なら `falseTarget` にジャンプする。 |
| `ISwitch` | `val:V, cases:[(C, u32)], default:u32` | `val` を各 `cases` の定数と比較し、一致する `target` にジャンプする。いずれにも一致しなければ `default` にジャンプする。 |
| `IComplete` | `val:V` | Thread を正常完了させる。**Normal(val)** signal を発行する。全ての Thread は (他の signal で中断されない限り) 最終的に `IComplete` で終了する。 |
| `IReturn` | `val:V` | ソースの `return` 文に対応する。**FnReturn(val)** signal を発行し、FN_BODY まで巻き上げる。 |

### 10.9 Agent 操作 (3 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `ICall` | `dst:V, agentId:A, args:[V]` | agent `agentId` を引数 `args` で呼び出す。子 agent の完了を待機し、結果を `dst` に格納する。**yield 点**。 |
| `IPar` | `dst:V, threads:[T]` | `threads` で指定された全 BLOCK thread を並列に起動し、全完了を待機する。各 Thread の結果値を順序付き配列として `dst` に格納する。**yield 点**。 |
| `IRequest` | `dst:V, reqId:R, args:[V]` | request `reqId` を引数 `args` で発行する。handler からの reply を待機し、reply 値を `dst` に格納する。**yield 点**。 |

### 10.10 Handle (3 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `IHandle` | `dst:V, hid:H` | `IRHandleDef[hid]` に基づいて handle を実行する。最終結果を `dst` に格納する。**yield 点** (内部で `IRequest` 等の yield が発生するため)。 |
| `IContinue` | `val:V, mutations:[(V, V)]` | **Continue(val, mutations)** signal を発行する。REQUEST_HANDLER 内でのみ使用。`val` は request への reply 値。`mutations` は `(stateVarId, newValueVarId)` のペアリスト。 |
| `IHandleBreak` | `val:V` | **HandleBreak(val)** signal を発行する。handle scope を脱出する。HANDLER_TARGET 内および REQUEST_HANDLER 内で使用可能。 |

### 10.11 For (3 命令)

| 命令 | シグネチャ | 説明 |
|------|-----------|------|
| `IFor` | `dst:V, fid:F` | `IRForDef[fid]` に基づいて for ループを実行する。最終結果を `dst` に格納する。 |
| `IForContinue` | `mutations:[(V, V)]` | **ForContinue(mutations)** signal を発行する。FOR_BODY 内でのみ使用。`mutations` は `(stateVarId, newValueVarId)` のペアリスト。 |
| `IForBreak` | `val:V` | **ForBreak(val)** signal を発行する。for ループを脱出する。FOR_BODY 内でのみ使用。 |

### Yield 点のまとめ

以下の命令はランタイムの yield 点 (suspension point) となり、永続化のスナップショット取得が可能:

- `ICall` — 子 agent の完了待機
- `IPar` — 全並列 Thread の完了待機
- `IRequest` — handler からの reply 待機
- `IHandle` — handle scope 全体の実行 (内部で `IRequest` が発生する)
- `IFor` — for ループ全体の実行

## 11. 定数プール

モジュールごとに 1 つの定数プールを持つ。プールのインデックスが `ConstId` である。

### ConstVal 一覧

| バリアント | 値の型 | 説明 |
|-----------|--------|------|
| `CVNull` | -- | `null` 値 |
| `CVBool` | `Bool` | `true` または `false` |
| `CVInt` | `Integer` | 任意精度整数。バイナリでは signed LEB128 でエンコード。 |
| `CVNum` | `Double` | IEEE 754 倍精度浮動小数点数。 |
| `CVStr` | `Text` | UTF-8 文字列。フィールド名、型タグ文字列、リテラル文字列等に使用。 |

### 定数の重複排除

コンパイラは定数プールへの追加時に重複チェックを行う (`Lowering.addConst`)。同一の値が既にプールに存在する場合、新しいエントリを追加せず既存の `ConstId` を返す。したがってプール内に同一値の重複は存在しない。

## 12. ランタイム値

IR レベルでの値は JSON 互換の 7 種類である。**クロージャ (関数値) は存在しない**。匿名関数がソース言語に存在しないため、関数参照は `AgentId` で行う。

| 種類 | 説明 | JSON 対応 |
|------|------|----------|
| **Null** | `null` | `null` |
| **Integer** | 任意精度整数 | `number` (JSON は区別しないが、ランタイムは内部的に区別する) |
| **Number** | IEEE 754 f64 浮動小数点数 | `number` |
| **Boolean** | `true` / `false` | `boolean` |
| **String** | UTF-8 文字列 | `string` |
| **Array** | 値の順序付きリスト | `array` |
| **Object** | string --> 値 の順序付きマップ | `object` |

### 型の動的判定

ランタイムは全ての値に対して型タグを保持する。`ITypeOf` 命令で取得可能な型タグ文字列は以下の 7 種:

```
"null" | "integer" | "number" | "boolean" | "string" | "array" | "object"
```

### 算術演算における型プロモーション

- **integer + integer** = integer
- **integer + number** = number (integer を number にプロモート)
- **number + integer** = number
- **number + number** = number
- **IDiv** の結果は常に number (整数除算は組み込み agent で行う)
- **integer --> number** への暗黙プロモーションは算術演算でのみ発生する。明示的な変換命令は存在しない。

### 等値比較

`ICmpEq` / `ICmpNe` は構造的等値比較を行う:

- 異なる型の値は常に不等 (`integer 1` と `number 1.0` は不等)
- Object は全フィールドが再帰的に等値であれば等値
- Array は全要素が再帰的に等値であれば等値

### Truthiness

`IBranch` および論理演算での truthy/falsy 判定:

- **Falsy**: `null`, `false`, integer `0`, number `0.0`, 空文字列 `""`
- **Truthy**: 上記以外の全ての値

## 付録 A: バイナリフォーマット (KTRI v0.2)

### ヘッダ

```
4b 54 52 49 00 02   "KTRI" + version 0x0002
```

### エンコーディング規則

| 型 | エンコーディング |
|----|-----------------|
| unsigned integer | LEB128 unsigned |
| signed integer | signed LEB128 |
| Text | LEB128(byte_length) + UTF-8 bytes |
| Vec\<T\> | LEB128(count) + items |
| Maybe\<T\> | `0x00` = Nothing, `0x01` + value = Just |

### セクション構成

```
Header:     "KTRI" (4 bytes) + version (2 bytes)
ModuleName: Text
Constants:  Vec<ConstVal>
Requests:   Vec<RequestDef>
Threads:    Vec<Thread>
HandleDefs: Vec<HandleDef>
```
