- Type は部分型を持っています。
  - リテラル型も含む。オブジェクト型は部分型関係があります。
  - union / intersection があります。
  - 型の narrowing はしません (複雑になるので)
  - array, tuple, object, function はそれぞれ独立していて、部分型関係を持ちません。
  - Array<"1"> <: Array<string> <: Array<unknown> <: unknown
  - {a : "1"} <: {a : string} <: {a : unknown} <: {} <: unknown
  - Left<"1"> <: Left<string> <: (Left<string> | Right<int>) = Either<string, int> <: unknown
  - (number, bool) <: (number, unknown) <: (number) <: () <: unknown
  - union, intersection はその場で全て計算します。再帰的 type synonym は一旦なし。
    - たとえば、Left<string> | Left<int> は Left<string | int> に簡約される。(変性が out であるため。in の場合は & になります。)
    - 最終的には、primitive type かそのリテラルたち, object か具体 object である {...}, function か具体 function である (...) => ..., tuple か具体 tuple である (type, type,...), array か具体 Array<T>, そして定義された data、それぞれに対して高々一つだけの union、 に正規化できます。すると一意に部分型関係を定めることが出来る。
    - 正規化方法 → {x : string, y : int} | {x : number, z : bool} は {x : string | number} に簡約 (共通フィールドだけ残す)。{x : string, y : int} & {x : number, z : bool} は {x : string & number, y : int, z : bool} に簡約するが、これはまだ簡約可能で、string と number には共通部分が存在しないから {y : int, z : bool} に簡約される。関数に関しては、((string) => int) | ((number) => bool) は ((string & number) => int | bool) に簡約されるが、これもさらに簡約されて ((never) => int | bool) になる。逆に ((string) => int) & ((number) => bool) は ((string | number) => int & bool) に簡約されるが、これもさらに簡約されて ((string | number) => never) になる。tuple に関しては | は最小の長さをとってそれぞれの要素を union する。& は最大の長さをとってそれぞれの要素を intersection する。
  - 外延的に型を定義していることに注意。今回は type narrowing が無いので、{type: "A", value: string} | {type: "B", error: string} は {type: "A" | "B"} と等しい。(つまり、後者に正規化できる。)。なぜなら、type narrowing がないため、obj.type === "A" であっても value: string であることは保証されないから。
  - また、今回の型システムでは、一応型にないプロパティ、 tuple の要素についてもアクセス可能で、その型は unknown となる。例えば、{a : string}.b の型は unknown になる。(warning は出していいかも)
- 高階 generics もあります。
  - 高階型同士の比較は今回必要ない
  - 型の比較をする必要があるのは、関数適用時、と関数適用結果を変数に代入するとき。後述するが、今回は一階のgenericsしか存在せず、どちらの場合も generics は完全に適用されているので、左右の型は一階の完全な型が得られる (より外部の generics が現れることはあるが。)

- generics 周りの整理
  - 1 階の generics しかありません。したがって、全ての generics は変数名、定義名に紐づいている

    ```
    // この関数の型は f: <T>((x: T) => T) ではない。あくまで、 f が T を要請し、 f の型は (x: T) => T となる、という二つの情報から構成されている。これは f を使用するときに必ず T を指定しなければならない事を意味する。つまり、f のジェネリクスを適用せず、forall のまま別の関数の引数に持っていくような、高階の generics は不可能。
    ```
  - つまり、この時点でジェネリクスは "型定義に紐づいたもの" と "値の定義に紐づいたもの" の二種類が存在することになる。(上記 f は後者)

  ```
  // 型定義に紐づいたジェネリクスの例
  type Singleton<T sub number> = { value: T }

  // 値の定義に紐づいたジェネリクスの例
  let f<T sub number> = fn (x: T) => x

  // 特殊な例 : data
  data Box<out T sub number>(value: T)
  // 仮想的には、以下の二つの定義が同時に存在する
  type Box<T sub number> = ... // 実際には Box<T> 型は (T が number 以下なら) primitive でそれ以上進まない。(後述するが、T が number 以下でないときは never になる)
  let Box<T sub number> = (...) : Box<T>
  ```

  - ここで境界の動作について注意。
    - type に紐づいた generics に境界外の型を渡すと、never になる。
      - たとえば、Singleton<string> = never。型エラーではない。
    - 値に紐づいた generics に境界外の型を渡すと、型エラーになる。
      - たとえば、f<string> は型エラーになる。
    - Box<string> (← この Box は type Box)という型自体を書くことはエラーではないが、それは never になる。Box<string>("hello") (← この Box は let Box) という値を書くことはエラーになる。Box<string> を never にするのは、正規化アルゴリズムの役割。

    ```
    data BoolLitExpr<in out T is bool>(value: bool)
    type LitExpr<T> = IntLitExpr<T> | BoolLitExpr<T>
    ```

- match case 構文と型チェック
  - match case 構文がある

  ```
  match expr {
    case (pattern1) => expr1
    case (pattern2) => expr2
  ...
  }
  ```

  - pattern は次のようなものがある。
    - リテラルパターン : 1, "hello", true, false
    - ワイルドカードパターン : \_
    - 変数パターン : x, y ...
    - オブジェクトパターン : {a = pattern1, b = pattern2,...}
    - タプルパターン : (pattern1, pattern2,...)
    - 配列パターン : [pattern1, pattern2,...]
    - コンストラクタパターン : ConstructorName(pattern1, pattern2,...)
    - 型アノテーションは付与できない。(後述するが、runtime は型情報は持たないため)
    - パターンはネストできる
    - 型チェックは次のように行う。変数型の導出→網羅性チェックの順で行う
      - expr の型を正規化する。例えば Failue<int> | Success<"bar" | "baz"> | {foo : number} であったとする。
      - まず、pattern の中身のそれぞれの変数の型、および、constructor に渡される generics の型を推論する。
        - 例えば、pattern1 が {foo = x} であったとする。このとき、expr の型の正規化結果から、{foo : number} という型を抜き出す。次に x という pattern に number という型を割り当てるが、これは変数パターンなので x : number という型情報を得る。
        - 例えば、pattern2 が Success("bar") であったとする。このとき、expr の型の正規化結果から Success<"bar" | "baz"> という型を抜き出す。ここで Success に渡された generics は "bar" | "baz" であることが分かる。Success data 型を参照すると 第一引数 には Success に渡された型の値が入ることが分かるので(ここでは "bar" | "baz")、次に "somevalue" という pattern に "bar" | "baz" という型を割り当てるが、これは literal パターンなので無視。(変数の型導出には関係ない)
        - たとえば、pattern3 が Success("baz") であったとする。この時、expr の型の正規化結果から Success<"bar" | "baz"> という型を抜き出す。ここで Success に渡された generics は "bar" | "baz" であることが分かる。Success data 型を参照すると 第一引数 には Success に渡された型の値が入ることが分かるので(ここでは "bar" | "baz")、次に "baz" という pattern に "bar" | "baz" という型を割り当てるが、これはliteral パターンなので無視。
        - たとえば、pattern4 が Failue(x) であったとする。この時、expr の型の正規化結果から Failue<int> という型を抜き出す。ここで Failue に渡された generics は int であることが分かる。Failue data 型を参照すると error には Failue に渡された型の値が入ることが分かるので(ここでは int)、次に x という pattern に int という型を割り当てるが、これは変数パターンなので x : int という型情報を得る。
        - さらに、網羅性チェックは、再帰的に外側から行う。まず、pattern1 ~ pattern4 の直和をとり、{foo : number} | Success<"bar" | "baz"> | Success<"bar" | "baz"> | Failue<int> という型を得る。これで一番外側は OK。次にそれぞれの pattern (同じ constructorならまとめる) について、網羅性チェックを行う。ここでは特に Success が重要である。Success は複数あるので、その内側のパターンも重要になる。Success<"bar" | "baz"> の第一引数の型は "bar" | "baz" である。ここで、内側のパターンがこれを網羅しているか確認する (ここでは型 "bar" | "baz" を持つ値を、case "bar" => ... と case "baz" => ... でパターンマッチしている、と見ることができる。)パターンの全ての直和を取った "bar" | "baz" は、"bar" | "baz" より上位であるため、OK
        - このようなチェックは例えば {foo = "bar"} と {foo = "baz"}、 ["bar", x] と ["baz", x] のようなパターンの組み合わせでも同様に行われる。
          - つまり、正規化した時の union 毎

- Effect system が入っています。
  - effect 宣言

  ```
  effect Log<out T>(value: T) => ()
  // Log は内部的には以下の二つを構成する。
  // 1. 値コンストラクタ Log<T> : (value: T) => () with Log<T> = ...
  // 2. エフェクト型コンストラクタ Log<T>
  effect Ask<out T, in R>(question: T) => R
  // 同様に値コンストラクタ Ask<T, R> : (question: T) => R with Ask<T, R> = ... を構成する
  ```

  effect は必ず関数型に紐づく。逆に、関数型は必ず effect を持つ。effect が無い場合、with pure とみなされる。
  - Effect の階層は、pure <: Log<T> <: Log<T> | Ask<T, R> <: impure となる。
  - Effect に対しても、union, intersection の計算ができる。intersection は正規化され、必ず互いに異なる effect の union となる。Log<T> | Log<S> は Log<T | S>。ここら辺の演算は通常の型と同じ。
  - 関数のブロックの中では好きなタイミングで effect 付の関数を呼び出せる。その effect は、最も内側の関数定義にバインドされる。また、generics を必ず指定しなければいけないのも通常の型と同様。

  ```
  let f = fn (x: int) => {
    Log<string>(x) // f は with Log<int> を持つことになる。
    let g = fn (y: int) => {
      let z = Ask<string, int>("What is your age?") // g は with Ask<string, int> を持つことになる。
      return y + z
    }
    return g(x) // g の effect には Ask があるので、f に Ask が付与される。必要な effect はすべて union される。ここでは f は with Log<int> | Ask<string, int> を持つことになる。
  }
  ```

  - effect は変性が out な関数の型パラメータとして扱われる。関数型同士の部分型関係は引数は contravariant, 戻り値は covariant であることに加えて、effect も convariant で判断。(effect か type かなだけで、複雑さはない)
  - handler は次のように記述。

  ```
  let result = hanndle f(1) {
    case Log(value) => {
      console.log(value)
      continue(())
    }
    case Ask(question) => {
      if (question === "What is your age?") {
        continue(42)
      } else {
        continue(0)
      }
    }
    return x => { return x } // return type を変えない場合は省略も可能
  }
  ```

  continue を使って継続を呼び出せる。continue の型は例えば Log エフェクトならその effect 定義の戻り値 () を受け取って、return type を変更したあとの f(1) の型を返す。ここでは int である。さらに、continue には handle case の右辺の中で発生する可能性のある effect 全てを持つ。ここでは consol.log があるので、IO effect を発生させる。最終的に continue: () => int with IO。 handle 自体も effective であるので、キャッチされなかった effect はより外側に伝播する。(これは通常のブロックと同様の挙動。つまり、 fn 内のブロックは、handle 内であっても、その内側で発生した effect は fn に投げられる。) ここでは IO がより上層へと投げられる。

  ```
  // return type を変える場合
  let result = hanndle f(1) {
    case Throw(error) => {
      console.error(error)
      Failure(error)
    }
    return x => {
      Success(x)
    }
  }
  ```

  - handle ... {} 自体の型には effect はつかない (上層の fn に投げられるため。) 同様に、Ask<string, int>("What is your age?") の型は int であり、Ask effect は上層の fn に投げられる。
  - 意味論は一旦飛ばす。あとで。

- spread operator
  - spread operator は、オブジェクトと配列、タプルで使える
    - construct
      - オブジェクト : {...obj1, a: 1, b: 2} は、obj1 に a: 1, b: 2 を上書きしたものを構築する。フィールドが重複している場合は、右側のものが優先される。なお、展開は高々 1 つで、必ず一番左にないといけない。これは意図しない上書きを防ぐためである。runtime に型情報が無い以上、例えば、型が {x: string} であっても、{x: 1, y: 2} が入っているかもしれない。このような object を右側で展開すると、y が上書きされてしまう。
      - 配列 : [a, ...arr1, ...arr2, b] は、arr1 と arr2 を展開して、a と b を両端に追加したものを構築する。配列はオブジェクトと違って、展開の順番は自由で、何個でも展開できる。型は union になる。
      - タプル : (a, ...tuple1, ...tuple2, b) は、tuple1 と tuple2 を展開して、a と b を両端に追加したものを構築する。配列と同様に、展開の順番は自由で、何個でも展開できる。型は tuple を直接に繋いだものとなる。
    - destruct
      - オブジェクト : let {a = a1, b = b1, ...rest} = obj は、obj から a と b を抜き取って、残りを rest に入れる。ここで、obj の型が {a: string, b: number, c: bool} であったとする。このとき、a1 は string 型、b1 は number 型、rest は {c: bool} 型になる。spread は高々一つで、一番右でないといけない。
      - 配列 : let [a, ...rest, b] = arr は、arr から両端の要素を抜き取って、残りを rest に入れる。ここで、arr の型が Array<string> であったとする。このとき、a は string 型、b は string 型、rest は Array<string> 型になる。spread は高々一つ
      - tuple : let (a, ...rest, b) = tuple は、tuple から両端の要素を抜き取って、残りを rest に入れる。ここで、tuple の型が (string, number, bool) であったとする。このとき、a は string 型、b は bool 型、rest は (number) 型になる。spread は高々一つ。

## Plan への指摘点

-- 型パラメータ (generics の一つの引数)
data TypeParam = TypeParam
{ tpName :: !Name
, tpVariance :: !Variance -- type 定義に紐づく場合のみ意味がある
, tpBound :: !Bound
}
deriving (Eq, Ord, Show)

→ tpVariance は、data に紐づけた方がよさそう。それぞれ意味が違います。Generics の TypeParm がしたいのは名づけと bounding です。data の方は名づけは必要ありません。例えば Success という **data** が必要なのは 1. 型引数を 1 つとる。 2. その variance は convariant である。という 2 点のみで、引数の名前は必要ないからです。逆に、TypeParam は、型変数を受け取ったらその場で全て計算して、正規化するので、variance は必要ありません。

| TFun ![FunParam] !Type !EffectRow -- (params) => RetTy with Effects

→ Effect は一つでよいです。(union された effect も一つの effect として表現できるため。) 例えば、with Log<T> | Ask<T, R> は with (Log<T> | Ask<T, R>) と表現できる。これも type の union と同様に、正規化されて、必ず互いに異なる effect の union になる。

| TIntersection !(Set NormalizedComponent) -- A & B & C (正規化済み)

→ そもそも正規化された型と普通の型を分けて考えた方がよさそう。正規化された型や effect にはもはや intersection は必要ありません。(例外があればおしえてください。) union も一重でいいはずで、つまり、union も intersection もない型として記述できます。もっと言うと、正規化された型はそれぞれの独立した型階層ごとに高々一つなので、例えば Array<string> | Foo<number> | {x: bool} の内部表現として `{array: string, data: Map({foo: number}), object: Map({x: bool}), tuple: Nothing, number {- int lit / num lit / int / num の階層 -}: Nothing, string {- string lit, string の階層 -}: Nothing, ...}` のような形で表現できるかもしれません。

TObject fields1 <: TObject fields2 iff fields2 のすべてのキー k に対して fields1[k] <: fields2[k] (fields2 にないキーは無視)
TTuple ts1 <: TTuple ts2 iff length ts1 >= length ts2 かつ各 i で ts1[i] <: ts2[i]
TArray a <: TArray b iff a <: b (covariant)
TFun params1 ret1 eff1 <: TFun params2 ret2 eff2 iff params contravariant, ret covariant, eff covariant
TData name args1 <: TData name args2 iff variance に従って各引数を比較
TUnion comps <: T iff 全 comp に対して comp <: T
T <: TUnion comps iff ある comp に対して T <: comp

→ いくつか指摘。Subtype の計算は正規化された型同士だけでやれば十分です。したがって、Union とかの比較は必要ない。
正規化された union 同士の比較は、それぞれの独立した型階層についてチェックすれば大丈夫です。例えば、`Array<"hello"> | Foo<int> | {x: bool} と Array<string> | Foo<number> | {x: unknown}` の比較は、Array<"hello"> と Array<string> の比較、Foo<int> と Foo<number> の比較、{x: bool} と {x: unknown} の比較をそれぞれ行えば十分です。(組み合わせ爆発を起こさない。)
あと重要なのがジェネリクスで、ジェネリクスが表に出てくるまで計算を進めます。最終、Generics T <: U の比較になったら、T の上界すべての intersection と、 U の下界すべての union をそれぞれまた正規化して比較します。最終、Generics が消えるか、両辺が同じ Generics T <: T みたいななるまで繰り返します。(無限ループするかな、これ？)

-- パターン (match/case + destructuring)
data Pat ann
= PVar !ann !Name
| PLit !ann !Literal
| PWild !ann
| PCon !ann !QualifiedName ![Pat ann] -- Con(p1, p2)
| PObject !ann ![(Name, Pat ann)] !(Maybe (Pat ann)) -- {a = p1, ...rest}
| PTuple !ann ![TuplePat ann] -- (p1, ...rest, p2)
| PArray !ann ![ArrayPat ann] -- [p1, ...rest, p2]

-- Spread 対応のタプル/配列要素
data TupleElem ann = TElem !(Expr ann) | TSpread !(Expr ann)
data ArrayElem ann = AElem !(Expr ann) | ASpread !(Expr ann)
data TuplePat ann = TPElem !(Pat ann) | TPSpread !(Pat ann)
data ArrayPat ann = APElem !(Pat ann) | APSpread !(Pat ann)
