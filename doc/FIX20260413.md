## 概念の整理

現在の Katari に導入されていない機能については★を付け明示する。これらの実装も併せて行う。

### Katari

Katari とは、分散した Agent を実行するための AI オーケストレーションシステムである。テキストベースの Katari 言語で記述されたワークフローと、Katari Protocol を元に、エージェントを非同期・並列的に実行し、オーケストレーションを達成する。
Effect System Like な機能をもっており、親子間の通信をそれによって表現する。

Katari は、様々なツールで構成されるツールチェーンの総称であるが、以下の 2 つに大別される。

- Katari Language
  - ワークフローを記述するためのドメイン特化言語、およびそのエコシステム。変数の説明である Semantic Annotation や、デフォルト非同期、後述する handle、および常駐 Runtime などが特徴である。
- Katari Protocol
  - Katari Protocol はインターネット上に構築された分散システムの通信規格である。Katari Protocol を実装するサーバーは、互いに通信が可能である。Katari Language の Runtime も、これを実装するサーバーの一つである。Effect System Like な Request System という特徴的な機能をもっている。

### Katari Protocol

Katari Protocol は、Katari Runtime をはじめとする Katari Protocol を実装するサーバー同士が通信するためのプロトコルである。HTTP ベースの REST API で定義されていて、Edge Computing を前提とした設計である。(つまり一度リクエストを切って、後続のリクエストで結果を返す、という形。そのためすべてのリクエストはほとんど即座に完了する。)

Katari Protocol は、Agent, Request と呼ばれる概念を中心に構成されている。まず、それらの概念を整理する。

また、Katari Protocol を実装したサーバーの URL を、Katari Endpoint と呼ぶ。これは必ずしもサーバーの Base URL ではない。例えば、サーバーは https://example.com/ で動いているが、Katari Protocol のエンドポイントは https://example.com/katari-protocol/ である、ということもありうる。この場合 /katari-protocol/ 以外の場所には自由な API を置いてよい。

また、Katari Protocol に属する各リソースは、サーバー endopoint url と 一意な id で識別され、それらを～ Ref と呼ぶ。

以下は Katari Protocol に登場する各種リソースである。分散システムなので、特に Single Source of Truth を意識してモデリングする必要がある。

### Agent Definition

Agent Definition とは、後述する Agent の定義である。各 Katari Endopoint は、複数の Agent Definition を持つことができ、各サーバーが割り当てる一意の ID を持つ。つまり、Agent Definition は (Katari Endpoint, Agent Definition ID) で一意に識別される。

Agent Definition がどのように定義されているかはサーバーの実装に依存するが、少なくとも以下の情報を持つ必要がある。これらは全て immutable であるべきである。

- id
  - サーバー内で一意な ID。UUID などの形式であることが望ましい。
- endpoint
  - この Agent Definition を提供する Katari Endpoint の URL。
- name
  - 人間が識別するための名前。必ずしも一意でないことに注意 (たとえばバージョン違いなどがあり得る。一意にする努力はするべきだが、必須ではない)
- description (optional)
- input schema
  - Agent に渡されるべき引数のスキーマ。JSON Schema に準拠する。(関数の引数等と違い、要素としては一つである。)
- output schema
  - Agent が返すべき結果のスキーマ。JSON Schema に準拠する。
- template refs (optional)
  - この Agent Definition が発行する可能性がある Escalation の Template Ref の配列への参照。Template については後述する。

### Agent (Instance)

Agent Instance (または、単純に Agent) とは、タスクの実行主体であり、通常は Agent Definition に引数を与えることで実行される。Agent は、Katari protocol を通して他サーバーの Agent Definition から Agent を起動することができる。Agent Definition 以外から Agent を起動することもできるが、それは Katari Protocol の範囲外である。Katari Protocol の範囲外で Agent を作ることは、許可されている。

Agent はその中で別の Agent を、Katari Protocol に従って起動したとき、それを子 Agent とする。逆に、Root 以外は親 Agent を持つ。
すなわち、Agent は木構造をなす。親 Agent は子 Agent の起動/終了の権限を持ち、子 Agent は親 Agent に対して結果を返す、または Escalation (後述) を発行することができる。

Agent の実装はサーバーによって異なるが、少なくとも以下の情報を持つ必要がある。基本的には、Agent とは、生存中の物を指す。Agent 実行履歴は各サーバーで保持しても良いが、それは Katari Protocol の必須要件ではない。

Agent は、以下の情報を持つ。これらは全て immutable であるべきである。

- id
  - サーバー内で一意な ID。UUID などの形式であることが望ましい。
- endpoint
  - この Agent を実行している Katari Endpoint の URL。
- input
  - Agent に渡された引数。JSON オブジェクト。
- definition ref
  - この Agent が起動された Agent Definition の ID。
- delegation ref (optional)
  - この Agent を起動した親 Agent が発行した Delegation への参照。トップレベルの Agent では null にする。
- status
  - RUNNING | TERMINATEING
  - Agent の状態。RUNNING は実行中、TERMINATING は終了処理中であることを表す。TERMINATING になった Agent は、終了処理が完了し、親に terminate_ack、もしくは delegate_ack を送るまで、他のイベントを実行しない。

### Delegation

親 Agent が子 Agent を起動したという証跡を表す。親 Agent 側のサーバーが管理する。

- id
  - サーバー内で一意な ID。UUID などの形式であることが望ましい。
- endpoint
  - この Delegation を管理する Katari Endpoint の URL。
- agent def ref
  - この delegation によって起動された、agent def への ref
- input
  - この delegation によって起動された、agent への input
- capability refs
  - この delegation で渡された capability への ref の配列

### Template

子 Agent の Escalation の雛形を表す。Algebraic Effects における Effect 定義

- id
  - サーバー内で一意な ID。UUID などの形式であることが望ましい。
- endpoint
  - この Template を管理する Katari Endpoint の URL。
- input schema
  - Escalation に渡される引数のスキーマ。JSON Schema に準拠する。
- output schema
  - Escalation が返すべき結果のスキーマ。JSON Schema に準拠する。
- name
  - template の名前。
- description (optional)

### Capability

Agent が持つ、子 Agent からの Escalation を処理する能力を表す。Algebraic Effects における Handler。

- id
  - サーバー内で一意な ID。UUID などの形式であることが望ましい。
- endpoint
  - この Capability を管理する Katari Endpoint の URL。
- template ref
  - この Capability が処理できる Escalation の Template への参照
- agent ref
  - この Capability を保持している agent の ref

### Escalation

子 Agent が特定の Capability に処理を依頼した証跡。

**重要: 現在のプロトコルではリレーをしている(親に対して request を投げ、それを親が proxy……)が、新しいプロトコルでは Capability に対して直接 Escalation するので、途中の親子関係をすっ飛ばして先祖 Agent に Escalation できる。**

- id
  - サーバー内で一意な ID。UUID などの形式であることが望ましい。
- endpoint
  - この Escalation を管理する Katari Endpoint の URL。
- capability ref
  - この Escalation が依頼している Capability への参照
- input
  - Escalation に渡された引数。JSON オブジェクト。

### Katari Protocol API

以上のリソースをやり取りするため、Katari Protocol には以下の POST が生えている。

- delegate
- delegate_ack
- escalate
- escalate_ack
- terminate
- terminate_ack

また、リソースの状態確認のため、以下の GET が生えている。

- agent_definitions
- agents
- delegations
- templates
- capabilities
- escalations

すべての katari サーバーはこれらすべての API を実装する必要がある。

以下、詳しく説明する。

また、以下の動作は katari protocol ライブラリで実装されるべきである (DB への保存含めて。) これらのイベントに合わせて追加の処理をしたい場合は、ライブラリ使用者がカスタムした hook を差し込めるようにする。(たとえば、後述するように、Katari Runtime は katari protocol の api call に合わせて、thread への event 発火を行う。)

流れとしては

外部からの API call
↓
--- katari protocol library ---
↓
katari protocol resource の状態変更 (DB への保存など)
↓
--- katari protocol server ---
↓
カスタムイベント
↓
終了

### {base_url}/delegate

このサーバーに対して、何らかのサーバーから Agent を起動するための API。

Request

- Agent Definition Ref
- Input
- (optional) Delegation
  - toplevel delegation の場合はなし
- Capability Refs

Response

- Agent Ref

Delegate を行うときは、親サーバーは新しい Delegation を作成する。その Delegation への参照と、入力値、Agent Definition Ref、Capability Ref の配列を子サーバーに /delegate で渡すことで、子サーバーは Agent を起動する。子サーバーは、その Agent の参照を親サーバーに返す。

Capability Ref の配列には

- 親から与えられた Capability Ref
- 自分の Agent が作った Capability の Ref

の中から選んで入れることができる。

### {base_url}/delegate_ack

このサーバーが、子サーバーに頼んでいた処理の結果を受け取るための API

Request

- Delegation Ref
- Output

Response

- なし

Delegation Ref はこのサーバーが保持するものである (delegate 時に作ったもの)
delegate_ack を受け取ったサーバーは Delegation を消す。
delegate_ack を送信するサーバーは Agent を消してから送信
←このとき、Agent に紐づいている Capability も Cascade で消す。
つまり、親からすると delegate_ack を受け取った時点で agent が消えていることを期待する。

### {base_url}/escalate

子サーバーが、親サーバーに Escalation を発行するための API

Request

- Escalation Ref
- Capability Ref
- Input

Response

- なし

Escalation を行う時、子サーバーは新しい Escalation を作成する。その Escalation への参照と、Capability Ref、入力値を親サーバーに /escalate で渡すことで、親サーバーは Escalation を処理する。

送信先は、capability ref の endopoint。

### {base_url}/escalate_ack

このサーバーが、親サーバーに頼んでいた Escalation の結果を受け取るための API

Request

- Escalation Ref
- Output

Response

- なし

子サーバーは作った Escalation と渡された Esclation Ref の値を比較し、適切な場所から処理を再開する。

再開した Escation は消す。

### {base_url}/terminate

親 Agent が、子 Agent を強制終了するための API。/terminate は子サーバーに生えている。

Request

- Delegation Ref

Resonse

- なし

親サーバーは、子サーバーに /terminate で terminate を送ることで、子サーバーの Agent を強制終了することができる。子サーバーは、/terminate を受け取ったら、該当する Agent の状態を TERMINATEING にする。子 agent に /terminate を伝播。

### {base_url}/terminate_ack

子サーバーが、親サーバーに頼んでいた Agent の強制終了の完了を受け取るための API

Request

- Delegation Ref

Response

- なし

子サーバーは、/terminate を受け取ったら、該当する Agent の状態を TERMINATEING にする。子 agent に /terminate を伝播。すべての子 agent から /terminate_ack が帰ってきたら、親サーバーに /terminate_ack を送る。このとき、 delegation を用いる。

### Katari Runtime

Katari Language を実行するためのランタイム。Katari Protocol を実装するサーバーの一つである。

基本的なシステムとしては

- Katari Runtime は常時起動している。
- Katari Compiler は Language を IR (Binary Code) に変換ーし、それを Runtime にアップロードする。
  - Katari Runtime は過去の IR も合わせてバージョン管理する。
  - またこの時、Agent Def -> Block のマッピング等もアップロード
- Katari Runtime は IR を元に Katari Protocol を提供する。

また、内部では Agent などは DB に保持し、実際の実行単位は Thread である。イベントも独自のものを用いる。ここの境界はあいまいになりがちなので、明確に分ける必要がある。

### Katari Runtime: Block

Block は、Katari Language の各 Block を表す。if/else 等の block もすべて分割されて block として表現される。
Runtime は Agent Def id -> Block id のマッピングを保持している。

基本的に、構文状の 1 block がそのまま block として扱われるが、例外として handle 構文がある

```
{
  // parent block
  handle {
    request f() => {
      // handler body block
    }
  } then {
    // handler then block
  }
  // handler target block ← ここが parent block とは別になる。
}
```

これ以外に例外はない。

### Katari Runtime: Thread

Thread は実行単位である。Block が call event によって実体となったもの。call event については後述する

Katari Runtime は仮想的にイベントドリブンで成り立っている。動作としては、イベントの発火 → イベントに合わせて該当 Thread が処理を実行 → 再びイベントを発火
というプロセスである。
Thread は program counter を保持していて、event に合わせて処理を進める。

**現在の実装では イベントキューがグローバルにあり、それを消費することで動いているが、実はキューはいらないことに気づいたので、ここに記しておく。**

まず、各 thread はあたかも非同期的に同時実行されているように見えるが、その実 event handle → event 発火は IR の処理を数行実行するだけであり、同期的である。その event に反応して次の thread を実行するのも同期的である。

ゆえに、非同期点は外部 API からの呼び出し、および外部 API の call のみになる。したがって、Runtime の動作としては

外部 API からの呼び出し → 内部eventへ変換 → event handling & event 発火の繰り返し (同期的) → 外部 API 呼び出し等を行う (複数回実行されることもある。) → 現在の状態の DB 保存

までを同期的 (DB 保存だけ非同期) に行える。この性質は将来 edge computing 化する時に重要である。

同期的であるので、event 発火関数はその event が完了されるまで待つことができる。例えば

```
complete_handler = () => {
  ...
  // thread 状態変更
  fire_event("...", args) // → 同期的に event handler が実行される
  // ここではもう event handling は完了している。
}
```

一応、 await しても良い (将来実行中に DB いじりたくなった時など。) しかし、これは本質的には一連の流れである。

2つ例外がある。par 境界は内部で発火した request のキューイングを行う。
また、外部からの request を実行中、非同期点 (つまり、別の外部 agent 実行) まで辿りついて待機中、同じ場所から request が届いた際、現在の request 完了まで待機 (これも本質的には par と同じである。)

これらのキューイングの追加/ 消費も外部からの API call が起点で同期的に行われるため、今までの議論は問題ない。したがって、グローバルキューを廃止する。

したがって、Katari runtime は以下の簡単な lambda である。

```
handleExternalApiCall = (apiCall) => {
  // API call を event に変換
  const event = convertApiCallToEvent(apiCall);
  // event handling & event 発火の繰り返し (同期的)
  handleEvent(event);
  // 現在の状態の DB 保存 (非同期)
  saveStateToDB();
}
```

### Katari Runtime: Thread Status

thread の状態は以下のどれかを取る。

- CALLING
  - 子 thread を実行中である。CALLING にはさまざまな種類がある。以下列挙する。
  - kind
    - BLOCK
      - 通常の block。例えば block 式, if の block, match の block などを呼び出す。変数は引き継がれる。
    - AGENT
      - 別で定義された agent を呼び出している。引数コンテキストは引き継がれない。
      - return の境界にもなる。
    - HANDLE_TARGET
      - handle の対象となっている block を呼び出している。引数コンテキストは引き継がれる。
    - HANDLE_BODY
      - handle 対象の block 呼び出し中に、request が呼ばれたときに、該当する handle body を呼び出している。引数コンテキストは引き継がれる。
    - HANDLE_THEN
      - Handle の then 節を呼び出している。引数コンテキストは引き継がれる。
      - handle then はさらに内部状態 next_action を持つ。next_action の意味は次のようである: return, continue, break 構文は一種の大域脱出である。これらの大域脱出は正常系として扱われる必要があるため、handle 構文はその handle target が大域脱出を試みた際、単純に親に素通しするのではなく、then 節を実行してから親に再びイベントを投げる必要がある。そのため then 実行中、その次の action を覚えておく必要がある。
      - next_action: Event を持つ。(持つ可能性のあるイベントは、returned, continue, broken for_continued, for_broken のいずれかである。)
    - FOR_BODY
      - for loop の body を呼び出している。引数コンテキストは引き継がれる。
    - FOR_THEN
      - for loop の then 節を呼び出している。引数コンテキストは引き継がれる。
    - PARALLEL
      - parallel block を実行中である。
    - DELEGATING
      - 外部 agent に delegation をしている。
- REQUESTING
  - 親エージェントに request を発行している (これは katari protocol における escalation である。)
  - from_thread (optional): ThreadRef を持つ。request がどの thread から来たかを表す。実行主体(自身が IRequest で値をなげた)である場合は null。また、外部 agent の場合もある。この場合は……どうしようか
  - previous_state: CallingKind
    - request を発行する前の CALLING の種類を表す。
- CANCELING
  - 処理のキャンセル中。親からの cancel イベントを受け取ってから、すべての子 thread から canceled イベントが返ってくるまでの状態。この間、自身が実行する必要のある event は、canceled 以外すべて無視する。
  - 例外として、par の 1 thread が大域脱出を行った際は、他の thread に cancel イベントを投げる必要がある。その際、canceled イベントが帰ってくるのを待つ必要があるため、CANCELING 状態に入る必要がある。ここで、next_action をセットすることで、par の thread の大域脱出の種類を記録しておく。
  - next_action (optional): Event を持つ。

### Katari Runtime: Event

Katari runtime 内でやりとりされるイベント。Katari protocol と区別をするために、明確に命名を分ける。イベントの命名は、親の立場になり、親から子は通常の動詞、子から親は過去分詞系を使う。

Katari Protocol からの通信は適宜 Katari Runtime 内のイベントに変換される。

- call
  - 親 thread が、block を thread として実体化し、それを子 thread とする。(外部からの delegation の場合、親 thread は存在しない。root とする。)
- cancel
  - 親 thread が、子 thread に対してキャンセルを要求する。子 thread はさらに子 thread にcancel イベントを伝播させ、すべての子から canceled が帰ってきた後、canceled を親 thread に返す。この間の状態は CANCELING である。
  - cancel イベントは全ての状態を強制的に CANCELING にする。さらに、その状態の子 thread (par の場合は複数個、それ以外なら 1 つ) に cancel イベントを投げる。
    - DELEGATING の場合は、外部へ katari-protocol, terminate を投げる。自身を CANCELING にするのは同様。
- completed
  - 子 thread が処理を完了したときに呼ばれる。親は適宜再開する。
  - 親の状態別。親から見た動作:
    - HANDLE_TARGET の場合は、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)
      - この際、next_action には親に対する completed イベントをセットしておく。
    - HANDLE_THEN の場合は、自身を 終了してさらに上へ投げる
      - この時、上に投げるイベントは next_action に準拠する。
    - HANDLE_BODY の場合は、continued と同様のイベントとして扱う。つまり、自身を HANDLE_TARGET にして、continue event を該当の子 thread に投げる。(continued は後述)
    - FOR_BODY の場合は、次のループに行くか、終わりならば FOR_THEN にして then 節の実行
    - FOR_THEN の場合は、適宜再開 (handle と違って、for loop は必ずしも block の最終行ではない)
    - BLOCK の時には、単純に再開
    - REQUESTING の場合は、次の二種のいずれかである。
      - program counter が IPar を指している。これは次の状態であることを意味する: par のブランチが requested を発行し、それを par 境界が proxy 中であるが、その間に他の par のブランチが completed を発行した。
      - program conter が外部 IDelgate を指している。これは次の状態であることを意味する: 外部 agent が escalation し、それが requested イベントに変換されこのスレッドへやってきた。このスレッドは 親へrequestをproxyした。そのイベントが処理中に、再び同じ agent が delegate_ack を発行した。
      - いずれにせよ、thread の event キューに event の内容を溜めておく。このキューは次回 continue イベント実行時に使う。
    - AGENT の場合は、単純に再開
    - PARALLEL の場合は、結果を溜めておく。すべて溜まっていたら配列として再開。
    - DELEGATING の場合は、実行を再開 (期待としては、外部からの delegation_ack が completed イベントに変換された。)
- returned
  - 子 thread が `return` 構文で処理を完了した時に呼ばれる。completed と違い、returned event は状態が CALLING:AGENT である境界まで巻き戻る。(ネストした位置での return)
  - 親の状態別、親から見た動作:
    - HANDLE_TARGET の場合は、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)
      - これ大事。return であっても、then 節は実行する必要がある。その代わり、next_action に return をセットして実行することで、then 節の実行が完了したら、再び AGENT 境界までの巻き戻しが行われる。
      - この際、next_action に 親に対する returned イベントをセットしておく
    - HANDLE_THEN, FOR_BODY, FOR_THEN, BLOCK の場合は、どのような program counter であっても自身を終了してさらに上へ投げる (returned イベント)
    - HANDLE_BODY の場合は、target 側の thread に対して、cancel イベントを投げる。状態を CANCELING にする。CANCELING の next_action に親に対する returned イベントをセットしておく。
    - REQUESTING の場合は、thread の event キューに returned イベントを貯めておく。
    - AGENT の場合、単純に再開
      - この境界まで巻き戻る。
    - PARALLEL の場合は少々複雑。returned を投げてきた子 thread 以外の thread に対して、cancel イベントを投げる。状態を CANCELING にする。CANCELING の next_action に親に対する returned イベントをセットしておく。
    - DELEGATING の場合はありえない筈。(外部から returned が来ることはない)もし来たらwarningを出して無視。
- continue
  - requested event を発火した子に対する thread の再開命令。
  - REQUESTING {from_thread} の場合
    - from_thread はこの request がどこから来たかを指す。実行主体 (自身の pc が IRequest) である場合は false
    - 自身の pc が IRequest (実行主体である) === from_thread が null であるときは、値を代入し処理を再開する。(次の event point まで進める。)
    - 自身の pc が IRequest ではない === from_thread が null でない時は、continue イベントを from_thread に投げ、thread の event キューにたまっているイベントの種類にて次を実行する。
      - event キューの先頭が requested イベントがある → REQUESTING 状態にする、キューから pop して event を投げる。
      - event キューの先頭が returned, continued, broken, for_continued, for_broken イベントがある → IPar である場合は、他のブランチに cancel イベントを投げる。自身は CANCELING 状態にする、それ以外の場合はないはず (IDeligate もキューに貯めるが、外部から returned イベントが来ることはないはず)。CANCELING の next_action に親に対するイベントをセット (event に入っていたイベントの種別を引き継ぐ)
      - event キューになんのイベントもない → 状態を REQUESTING に溜めていた previous_state に戻す。
  - それ以外の状態
    - ありえない。warning を出して無視。
- continued
  - これはややこしいが、子 thread が IContinue を発行したときに発行される。つまり、handle body 内で continue 構文を書いた時の動作である。HANDLE_BODY 境界まで巻き戻る。
  - 親の状態別、親から見た動作:
    - HANDLE_TARGET の場合は、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)
      - この際、next_action に 親に対する continued イベントをセットしておく
    - HANDLE_THEN, FOR_BODY, FOR_THEN, BLOCK の場合は、どのような program counter であっても自身を終了してさらに上へ投げる (continued イベント)
    - HANDLE_BODY の場合は、HANDLE_TARGET に状態を変えて、continue event を子 thread に投げる。
      - これが continued の本命
    - REQUESTING の場合は、thread の event キューに continued イベントを貯めておく。
    - AGENT の場合は warning を出して無視 (agent 内で continue はありえない)
    - PARALLEL の場合は、continued を投げてきた子 thread 以外の thread に対して、cancel イベントを投げる。自身は CANCELING 状態にする。CANCELING の next_action に親に対する continued イベントをセットしておく。
    - DELEGATING
      - warning を出して無視 (外部から continued が来ることはないはず)
- broken
  - HANDLE_TARGET の場合は、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)
    - この際、next_action に 親に対する broken イベントをセットしておく
  - HANDLE_THEN, FOR_BODY, FOR_THEN, BLOCK の場合は、どのような program counter であっても自身を終了してさらに上へ投げる (broken イベント)
  - HANDLE_BODY の場合は、自身を終了して、自身を CANCELING 状態にする。さらに、cancel イベントを子 thread に投げる。CANCELING の next_action に親に対する completed イベントをセットしておく。
    - これが broken の本命
  - REQUESTING の場合は、thread の event キューに broken イベントを貯めておく。
  - AGENT の場合は warning を出して無視 (agent 内で break はありえない)
  - PARALLEL の場合は、broken を投げてきた子 thread 以外の thread に対して、cancel イベントを投げる。自身は CANCELING 状態にする。CANCELING の next_action に親に対する broken イベントをセットしておく。
  - DELEGATING の場合は warning を出して無視 (外部から broken が来ることはないはず)
- for_continued
  - 子 thread が IForContinue を発行したときに発行される。for loop の body 内で continue 構文を書いた時の動作である。FOR_BODY 境界まで巻き戻る。
  - 親の状態別、親から見た動作:
    - HANDLE_TARGET の場合は、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)
      - この際、next_action に 親に対する for_continued イベントをセットしておく
    - HANDLE_THEN, FOR_THEN, BLOCK の場合は、どのような program counter であっても自身を終了してさらに上へ投げる (for_continued イベント)
    - FOR_BODY の場合は、FOR の配列がまだ残っている場合と残っていない場合で場合分け
      - まだ残っている場合は、FOR_BODY に状態を変えて、次のループの body を呼び出す (call イベント)
      - 残っていない場合は、FOR_THEN に状態を変えて、then 節の実行 (call イベント)
      - コレが for_continued の本命
    - HANDLE_BODY の場合は、target 側の thread に対して、cancel イベントを投げる。状態を CANCELING にする。CANCELING の next_action に親に対する for_continued イベントをセットしておく。
    - REQUESTING の場合は、thread の event キューに for_continued イベントを貯めておく。
    - AGENT の場合は warning を出して無視 (agent 内で for continue はありえない)
    - PARALLEL の場合は、for_continued を投げてきた子 thread 以外の thread に対して、cancel イベントを投げる。自身は CANCELING 状態にする。CANCELING の next_action に親に対する for_continued イベントをセットしておく。
    - DELEGATING の場合は warning を出して無視
- for_broken
  - HANDLE_TARGET の場合は、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)
    - この際、next_action に 親に対する for_broken イベントをセットしておく
  - HANDLE_THEN, HANDLE_BODY, FOR_THEN, BLOCK の場合は、どのような program counter であっても自身を終了してさらに上へ投げる (for_broken イベント)
  - FOR_BODY の場合は、自身を CANCELING 状態にする。さらに、cancel イベントを子 thread に投げる。CANCELING の next_action に親に対する completed イベントをセットしておく。
    - これが for_broken の本命
  - REQUESTING の場合は、thread の event キューに for_broken イベントを貯めておく。
  - HANDLE_BODY の場合は、target 側の thread に対して、cancel イベントを投げる。状態を CANCELING にする。CANCELING の next_action に親に対する for_broken イベントをセットしておく。
  - AGENT の場合は warning を出して無視 (agent 内で for break はありえない)
  - PARALLEL の場合は、for_broken を投げてきた子 thread 以外の thread に対して cancel イベントを投げる。自身は CANCELING 状態にする。CANCELING の next_action に親に対する for_broken イベントをセットしておく。
  - DELEGATING の場合は warning を出して無視 (外部から for break が来ることはないはず)
- requested
  - 子 thread が親 thread に request を投げたときの処理。
  - 以下の "さらに上に投げる" について、親が存在しない (= root thread) の場合は、外部に対して katari protocol / escalate を投げる。
  - 親の状態別、親から見た動作:
    - HANDLE_TARGET の場合、request 内容 (template) が handler として用意されているかで場合分け。
      - 用意されている場合、HANDLE_BODY に状態を変えて、handle body を呼び出す (call イベント)
        - これが requested の本命
      - 用意されていない場合、REQUESTED に状態を変えて、さらに上へ requested イベントを投げる
    - HANDLE_THEN, HANDLE_BODY, FOR_BODY, FOR_THEN, BLOCK の場合は、状態を REQUESTING してさらに上へ投げる (requested イベント)
    - REQUESTING の場合は、thread の event キューに requested イベントを貯めておく。
    - AGENT の場合は、状態を REQUESTING にしてさらに上へ投げる (requested イベント)
    - PARALLEL の場合は、状態を REQUESTING にしてさらに上へ投げる (requested イベント)
    - DELEGATING の場合は、子 agent からの escalation が変換された結果である。状態を REQUESTING にしてさらに上へ投げる (requested イベント)
- canceled
  - 子 thread がキャンセル完了したことを親 thread に伝えるイベント。親は CANCELING であることが期待される。

"再開する" とは以下の IR 命令に突き当たるまで IR を実行することを指す。

- IRequest
  - thread が親 thread に request を発行する命令。これが実行されると、親 thread に requested イベントが投げられる。
  - この thread は REQUESTING 状態になる。
  - この後の想定挙動は、requested イベントが伝播していき、HANDLE_TARGET、かつ該当 template が用意された thread にぶつかるまでさかのぼる。その間の thread は REQUESTING 状態になり、さらなる REQUESTING はキューイングされる。HANDLE_TARGET は HANDLE_BODY に状態を変えて、handle body を呼び出す。その後、handle body は completed, continued, broken のいずれかのイベントを発行。completed, continued の場合は HANDLE_TARGET に状態を変えて、今度は子 thread に continue イベントを投げるのが連鎖し、IRequest を発行した thread まで続く。broken の場合は、cancel イベントが伝播し、子が全て終了する (これは IRequest を発行した thread も含まれる。) その後、completed イベントが親 thread に投げられる。
- IPar
  - 複数の子 thread を並列に実行する命令。これが実行されると、複数の子 thread が生成される。この thread は PARALLEL になる。
  - 子 thread の内一つが大域脱出 (returned, continued, broken, for_continued, for_broken) を行った場合、他の thread に cancel イベントを投げる。すべての子 thread から canceled が帰ってきたら、親 thread は PARALLEL でなくなり、さらに上へ大域脱出イベントを投げる。
  - 子 thread が request を投げた場合、それを proxy する。(PARALLEL -> REQUESTING) 親からの continue を待機。この間に再び request が来た場合、キューイングしておく。
  - 子 thread が正常系で終了 (completed) した場合、その結果を貯めておく。すべての子 thread が正常系で終了したら、続きの処理を行う。
    - `let xs = par [...]` の xs に代入して続きを行う、という意味
- IBlock
  - 通常の block を呼び出す命令。これが実行されると、子 thread が生成される。この thread は BLOCK になる。
  - 子 thread が request を投げた場合、それを proxy する。(BLOCK -> REQUESTING) 親からの continue を待機。この間に再び request が来た場合、キューイングしておく。
  - 子 thread が大域脱出 (returned, continued, broken, for_continued, for_broken) を行った場合、親 thread は BLOCK でなくなり、さらに上へ大域脱出イベントを投げる。
  - 子 thread が正常系で終了 (completed) した場合、続きの処理を行う。
- ICall
  - Katari Runtime 内の他の agent を呼び出す命令。これが実行されると、子 thread が生成される。この thread は AGENT になる。
  - 子 thread が request を投げた場合、それを proxy する。(AGENT -> REQUESTING) 親からの continue を待機。この間に再び request が来た場合、キューイングしておく。
  - 子 thread が大域脱出 (returned, continued, broken, for_continued, for_broken) を行った場合、returned については、後続の実行を再開する。その他の大域脱出は型システム上起こりえない。warning を出して無視。
- IFor
  - for loop を実行する命令。これが実行されると、子 thread が生成される。この thread は FOR_BODY になる。
  - この後の流れとしては、子 thread が request を投げた場合、それを proxy する。(FOR_BODY -> REQUESTING) 親からの continue を待機。この間に再び request が来た場合、キューイングしておく。
  - 子 thread が大域脱出 (returned, continued, broken, for_continued, for_broken) を行った場合、親 thread は FOR_BODY でなくなり、さらに上へ大域脱出イベントを投げる。
  - 子 thread が正常系で終了 (completed) した場合、次のループに行くか、終わりならば、FOR_THEN に状態を変えて、then 節の実行 (call イベント)
- IHandle
  - handle 構文を実行する命令。これが実行されると、handle target block をもとに子 thread が生成される。この thread は HANDLE_TARGET になる。
  - この後の流れとしては、子 thread が request を投げた場合、handle body が用意されていれば、HANDLE_BODY に状態を変えて、handle body を呼び出す (call イベント)。用意されていなければ、REQUESTING に状態を変えてさらに上へ requested イベントを投げる。
  - 子 thread が completed, continued, broken のいずれかのイベントを発行した場合、HANDLE_THEN に状態を変えて、then 節の実行 (call イベント)。この際、next_action に親に対するイベントをセットしておく。

### Katari Runtime: Katari Protocol Event <-> Runtime Event マッピング

外部からのイベントは次のように katari runtime 内のイベントに変換される。

- delegate -> call
  - 発行実態は親 agent である。katari protocol ライブラリが作った agent instance に紐づけて、該当 thread を開始する。
- delegate_ack -> completed
  - いずれかの thread が外部への delegation を行い (IDelgate 命令を実行し、状態は DELEGATING)、その agent からの応答を待機している。agent からの delegate_ack はその thread に対して completed イベントを発火する。
- escalate -> requested
  - いずれかの thread が外部への delgation を行い (IDelgate 命令を実行し、状態は DELEGATING)、その agent からの応答を待機している。agent からの escalation はその thread に対して requested イベントを発火する。
- escalate_ack -> continue
  - root まで処理されなかった requested イベントが、先ほど述べた event handling に従い外部に escalate として投げられたあと、その escalate に対する親 agent からの応答 escalate_ack は、該当 root thread に対して continue イベントを発火する。
- terminate -> cancel
  - 親からの terminate 命令は該当 root thread に対して cancel イベントを発火する。
- terminate_ack -> canceled
  - 子からの terminate_ack は該当する DELEGATING thread に対して canceled イベントを発火する。

### Katari Runtime: IR に搭載する各概念

IR 上の id と katari protocol 上の id の関係がややこしいのまとめておく。

IR 上では、すべての block に一意な id を振る。これを block id とする。**また、external agent にも、同様に block id を割り当てる**
function call はその対象によって変換さきの IR が変わる。

- katari 上で定義された agent
  → block id の ICall に変換する。(これは現状の設計と多分違う。現状では、agent は外部呼び出しになっている。)
- katari 上で定義された template request
  → IRequest に変換
- katari 上で定義された external agent
  → ICall に変換 (先ほど割り当てた仮想的な block id)
  - この実行後、call event が発火する。call event は block を thread に変換しようとするが、ここで初めて block が仮想的な external agent であることがわかる。この thread の状態は AGENT とし、外部に投げる。
- katari 上で定義された external template request
  → IRequest に変換

Block ID は Block を表す一意な ID である。 Thread ID は Thread を表す一位な ID である。Block, Thread は名前を持たない。

Agent Def ID は Katari Protocol 上の Agent Definition を表す一意な ID である。Agent Def ID -> Block ID と 名前 -> Agent Ref のマッピングをどこかに用意しておく必要がある。Agent Def ID は Compiler ではなく、ランタイムが生成する。次のようなフローで行う。

コンパイラは、すべてのの agent 定義の "名前" (module.to.path.name) から Block ID へのマッピングをランタイムに送る。また、この際、内部定義か外部定義かを付与しておく。(将来的に内部定義だが非公開もあり得るので、ここは enum)。外部定義 agent については、その agent の alias名:agent_def_name のマッピングも送る。また、 alias -> katari endpoint のマッピングも送る。
ランタイムは各内部定義された Agent について、Agent Def ID を生成。また、外部定義された Agent については、alias -> katari endopoint のマッピングをもとに、agent def を名前で検索。一意に特定できたら、その agent ref をその名前の mapping 先として保存する。
template についても同様に行う。
最終的に以下のような 4 つのマッピングが得られる。

- Agent Def ID -> Block ID
  - 各ないぶ定義 agent について、agent def id から block id へのマッピング。これにより外部からの /delegate に対して、どの block を起動すべきかわかる。
- 名前 (path.to.module.agent) -> Agent Ref
  - 各 agent 定義 (external 含む) に対して、名前から agent ref へのマッピング。この名前と外部 agent def の名前が異なることには注意。(例えば、discordサーバーが "watch_message" という名前で定義されていても、内部で external agent watch from "discord:warch_message" というコードがモジュール path.to.module 内に書かれている場合、"path.to.module.watch_message" という名前から、外部 agent ref へのマッピングが構成される。)
  - これは ref_agent 関数で使用される。
- 名前 (path.to.module.template) -> Template Ref
  - 各 template 定義 (external 含む) に対して、名前から template ref へのマッピング。これも外部 template を含む。

また、runtime で handler 定義をした際は、その handle 対象の template 一つごとに capability を作成し、Capability ID -> Thread ID のマッピングなどを保存。ここらへんはよしなに。

## コード改善

コードの改善を以下の手順で行う。

次の 2 つの軸 (スケール、コード品質) を掛け合わせて評価。例えば、可読性について大スケール～小スケールまで評価する。保守性について大スケール～小スケールまで評価する。など。少なくとも数回コードベース全体を読み、徹底的に評価。改善点をあげる。

- スケール軸の評価
  - 大きなスケールから、小さなスケールまで
  - 例 (↑が大きなスケール、↓が小さなスケール)
    - 言語選定、フレームワーク選定
    - アーキテクチャの選定 (レイヤー構造、モジュール構造)
    - Katari Protocol 仕様
    - Katari 言語仕様
    - データ構造定義
    - 抽象構造の定義
    - モジュール分割、関心の分割
    - 関数の分割
    - 命名等

## publish ready な状態にする

足りていない機能を列挙する。

- DB への状態保存 / 再開
  - 再起動耐性のほか、そもそも最終的に Functions (サーバーレス) として実装するので、状態保存は必須。
    - 一旦 Cloudflare Workers へのデプロイを目指す。
    - DB は PostgreSQL を使いたい。Cloudflare Workers から Neon にアクセスして状態保存。
  - 外部サーバーもすべて Functions として実装可能にする。問題は cron サーバー、sandbox サーバー、discord サーバーあたりの常時接続をしているサーバー。
  - cron サーバー
    - Cloudflare Workers の cron trigger を使用する。
    - これでは動的にスケジューリングできないため。1 分間隔の polling と前回実行などの情報を組み合わせていい感じに katari runtime に request を投げる。
      - この実装がむずそう
  - sandbox サーバー
    - E2B を使用
      - e2b.dev
    - E2B 内で container を立ち上げたり落としたりすることで、sandbox を提供する。これにより、sandbox サーバー自体は stateless になる。
  - discord サーバー
    - コレばっかりはどうしようもない。message は監視したいから、次の二つのうちどちらかの方法で実装。
      - discord サーバーはサーバーレスにする。そのほかに、ECS でも Fly.io でもいいから、常時接続サーバーを用意。このサーバーはメッセージが飛んできたら discord サーバーを起動。discord サーバーは現在実行中の message 監視 agent 一覧を読んで、その親に escalation を飛ばす。
      - discord サーバーだけは常時接続サーバーにする。

- モデル切り替え
  - 単純に、Gemini や Claude という単位の他に、その中のモデルも ask ai with tools で選択できるようにする。モデル名が指定できれば良い。

- エラーハンドリング
  - 次のようにする。
  - Katari Protocol を拡張して、/throw を新たに作る。 /throw は子thread が 親に対して呼ぶものである。入力は delegation と message。/throw を投げることができるのは、子 thread が自身の子 thread すべてを /terminate した後のみ。親 agent は /throw を受け取ったら、その親に対して /throw を投げるなどする。しかしこれは強制ではない。後述するように katari runtime はコレをハンドルできる。
  - Katari Runtime に、特殊な handler throw(message) => ... を用意する。 throw イベントは内部的には throw request へと変換され、katari runtime はそれを effect として処理できる。なお、break を呼ぶしか handle 方法はない。(型的に)
    - break を呼ぶと、terminate を子に対して送る (ここらへんの動作はすべて handler と同じ。) 自身は completed を親に投げる。
    - throw request が root thread まで到達すると、それは再び /throw api call に変換されさらに親へと伝わる。
