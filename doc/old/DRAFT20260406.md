# KATARI

Qatali を大幅に簡略化したいです。 (まず名前を KATARI にする)
AI オーケストレーターにここまでの機能はいらないと判断しました。逆に、他のもっとネイティブに近いプログラミング言語を同時に作り、そちらにこれらの機能を入れることにしました。そちらの言語は私がつくるので、あなたは Katari を改善する必要があります。

1. 新しい言語の言語仕様を細かく作ってファイルに保存してください。分割しても良いです。overview は次の節で述べます。
2. 既存のコードベースを全て消してください。バックアップはしてあるので、次に改めてプロジェクトを作るときはそちらを参考にできます。
3. 新しい言語を言語仕様に基づいて実装してください。

## 言語仕様

### 基本的な理念

オーケストレーションに必要な機能のみを付けます。具体的には、型システムの複雑さを下げます。effect system に似た言語仕様は持ちますが、基本的には server side のイベントを listen / 親コンテキストの関数を呼び出すための物です。
以下の機能は消します。

- Generics
  - type, data の定義, fn, val の定義から消す (完全に消す)
  - array だけ generics できるようにする　構文 → `array[T]`
    - array を construct するときは、それぞれの要素の型のunionのarrayとなる
  - result, option は effect system によって再現する。
  - たとえば、result は

```
effect throw(e: string) -> never

fn maybeThrow (...) -> T with throw
```

- 型推論
  - generics と型推論を消すことで、型チェックは全て決定的になり、楽
- Multi shot effect
- 効率化された IR
  - オーケストレーションに効率はいらず、効率がいる処理は別の言語に委譲するべきです。
- data type
  - 判別可能 union で代用します。result や option は effect system で記述可能

以下の機能を新たに追加します。

- Record 型 (object)
- もっと単純な IR
- external fn とその呼び出し
- 柔軟な template literal

次の機能は続投します。

- match
- Effect System (one shot)
- literal 型
- 部分型
- union, intersection
  - 続投しますが、union と intersection はその場で計算されます。判別可能 union はまた別です。後述。
- module システム
  - これらはほとんど同一のものを搭載してください。
  - コードベースを消す前にドキュメント化して保管してください。
  - なお pub は一旦消します。それぞれの task がどこから見れるかっていうのは複数サーバー出てくるこの構成だとちょっとややこしいので、一旦すべての定義は publish とします。今後また考える。

## type system

重要:
基本的に現在使用可能な effect ops を保持して起き、effect op を呼び出した際はそれが op <: ops かを判定します。

### 型階層

never <: any type
any type <: unknown

配列と object に関係はありません。length などもありません。
また、タプル状の配列もありません。配列は常に 1 つの型の配列です。

他の点は ts と同じ。

### union / intersection のその場での計算

基本的に subtyping check は effect / fn への引数受け渡しと、let への代入で行います。
それぞれ左辺と右辺が渡されますが、両辺とも正規化します。ここで重要なのが union も問答無用で正規化することです。

```
{type: 0, x: integer} | {type: 1, y: string} -> {type: 0 | 1} // x, y は消える。
{type: 0, x: integer} & {type: 1, y: string} -> {type: 0 & 1, x: integer, y: string}
array[integer] | array[string] -> array[integer | string] // integer と string のどちらか、という情報は消える
```

これによって、全ての type 構文は number 階層, string, boolean, array, object の階層の union (すなわちそれぞれ交叉の無い union) になります。 それぞれ高々一つです (複数の object の union はすべてひとつに正規化されることに注意) (なお、これに加えて unknown が存在。ここは直和で表現する)

こうすれば、それぞれの階層で subtyping check するだけですが、それは容易に行えます。

例外は判別可能 union です。後述。

### 判別可能 union

判別可能 union は、特殊な object の union です。前述した uniq が鍵です。uniq が付けられるプロパティは **それぞれの object 型について高々一つ** で、それが discriminator となります。内部では特殊なオブジェクトとして保持します。

```
{uniq type: 0, x: integer} | {uniq type: 1, y: string} → DISC { discriminator: "type", mapping : { 0 : {type: 0, x: integer}, 1 : {type: 1, y: string}}}
```

このように変換できる条件は以下です。

1. すべての要素が object の union である。
2. それぞれ uniq プロパティを持つ。
3. それらが同一のプロパティである。
4. それらの uniq プロパティの値は全てリテラルである。また、optional でない。
5. それらの uniq プロパティのリテラル値は全て異なる。

それ以外の場合、uniq は無視され通常の union として計算されます (つまり情報が落ちる)

判別可能 union とその他の型の subtyping 計算は次のように行う。

```
DISC { discriminator: d, mapping : { l1 : t1, l2 : t2,...}} <: T // T は DISC 以外
→ t1 <: T && t2 <: T && ...

T <: DISC { discriminator: d, mapping : { l1 : t1, l2 : t2,...}} // T は DISC 以外
→ T <: t1 || T <: t2 || ...

DISC { discriminator: d, mapping : { l1 : t1, l2 : t2,...}} <: DISC { discriminator: d, mapping : { l1' : t1', l2' : t2',...}}
→ discriminator が同一である必要がある。l1, l2,... は l1', l2',... に含まれる必要がある。ここで l1 == l1',... だったとし、さらに、t1 <: t1', t2 <: t2',... である必要がある。
それらが満たされなかった場合、左辺を普通の union t1 | t2 |... に変換。再び正規化し、右辺と比較する。(右辺は正規化しちゃだめ)
```

### optional field の扱い

runtime では、存在しないフィールドにアクセスすると null が帰る。したがって、x?: T にアクセスすると T | null になる。

Normalize 時は T | null と optional field は区別して考えられる。
union 計算時 → どちらか一方で optional であった場合、optional property になる。右辺はそれぞれの object の type の union になる。
intersection 計算時 → どちらも optional であった場合、optional property になる。右辺はそれぞれの object の type の intersection になる。

subtyping 計算時は、optional field は単に null を含む普通の field として扱う。つまり、x?: T は x: T | null と同じように扱って subtyping の計算をする。

JSON Schema 生成時は optional field として扱われる。

### match 式の型付け

match は基本的に、各 literal pattern, 及び判別可能 union を分解できる。

match の先頭から見ていき、マッチした分を引く。これを全ての pattern について行う。最後まで行っても残りがある場合はエラー。

例:

```
e : {uniq type: 0, x: integer} | {uniq type: 1, y: string}  // -> 内部的には DISC { discriminator: "type", mapping : { 0 : {type: 0, x: integer}, 1 : {type: 1, y: string}}} として保持されている。

match e {
  case {uniq type = 0, x = x} => ...
  // 引いた後は DISC { discriminator: "type", mapping : { 1 : {type: 1, y: string}}} になる。
  case {uniq type = 1, y = y} => ...
}
```

引き算については、ネストした部分も引き算される。例えば、

```
e : {uniq type: 0, x: integer | string} | {uniq type: 1, y: string} // -> 内部的には DISC { discriminator: "type", mapping : { 0 : {type: 0, x: integer | string}, 1 : {type: 1, y: string}}} として保持されている。
```

に対して、

```
match e {
  case {uniq type = 0, x = integer(x)} => ...
  // 引いた後は DISC { discriminator: "type", mapping : { 0 : {type: 0, x: string}, 1 : {type: 1, y: string}}} になる。
  case {uniq type = 0, x = string(x)} => ...
}
```

引き算について

パターンが受け入れる exact な、正規化された型を求めることが出来る。例えば、pattern 0 なら 0, pattern x なら unknwon, pattern integer(x) なら integer, pattern {foo = x, bar = boolean(y)} なら {foo: unknown, bar: boolean}、pattern {uniq type = 0, x = integer(x)} なら `DISC { discriminator: "type", mapping : { 0 : {type: 0, x: integer}}}` になる。
この例外はないと思う (確認して欲しい)
DISC パターンを作るには、必ず一つのみ uniq が存在し、それがリテラルパターンでなければいけない。すべてのパターンで同じ discriminator である必要はない (後述するパターンの型生成と引き算において、先頭以外の disc は object になっちゃうが……)

そして、正規化された型同士の引き算を行う。一旦 optional と DISC を無視すると、これは以下のように行える。

- 正規化されたそれぞれの階層の union について引き算を行う。両辺に string がある場合は結果に string はなし。左辺のみに string があるなら結果は true

```
Normalized(stringKind = true, ...) - Normalized(stringKind = true, ...) -> Normalized(stringKind = false, ...)
Normalized(stringKind = true, ...) - Normalized(stringKind = false, ...) -> Normalized(stringKind = true, ...)
Normalized(stringKind = false, ...) - Normalized(stringKind = true, ...) -> Normalized(stringKind = false, ...)
...
```

- int / number は階層が被っているが、number - int = number, int - number = never とする。 (安全に倒す)

右辺が unknown の場合は全て引く。

- Object の引き算の場合、ネストした部分をそれぞれ引き算する。この時、左辺にのみあるプロパティはそのまま。右辺にのみあるプロパティは消える。

#### optional

pattern から型を構成する方法より、pattern の型には optional はない (つまり左辺にのみある)

- あるプロパティについて、左辺 optional, 右辺にはない → optional になる
- あるプロパティについて、左辺 optional, 右辺にもある → プロパティは残らない。

#### DISC

DISC - DISC は次のようにする。

- discriminator が同一である必要がある。
  - → 同一でない場合、両辺を普通の object union に変換、再び正規化して obj - obj として行う
- 左辺のそれぞれの mapping について、右辺に存在するなら、その内部の型の引き算を計算する。その時 never になったらその DISC mapping を消去する。

DISC - obj
obj - DISC

それぞれ DISC を普通の object union に変換して再び正規化してから引き算。

### effect system

基本的に先頭から見ていき、

1.それぞれの式にたいして発生する可能性のある effect を集める。2. statement を最後から見ていき それぞれの statement について発生する可能性がある effect について union を取っていく。3. handle があった場合、handle 対象の effect を引き、handle の case 内で発生している effect を足す。4. fn の body 全体で発生する effect と fn 定義の effect annotation を見比べて型チェック。なお、annotation が無ければ推論として、body から発生する effect を採用。effect が task だった場合、発生する可能性のある effect が一つでも存在したらエラー。

### Semantic annotation

semantic annotation は JSON Schema の生成に利用される。description として入れる。
JSON Schema は以下について生成される。

- fn の引数、返り値
- effect の引数、返り値

全てにおいて、正規化後の型に対して生成される。正規化後なので、基本的に union のみ含まれていて、かつ、交叉はない状態で生成されます。
再帰的に内側まで生成されます。DISC は discriminator の情報をちゃんと入れる。

## Katali protocol

結構設計適当なので議論しながら specify していきたい。突っ込んでください。例えば、今の仕様だと delegate を読んだあと、continue / break が帰ってきた場合、どの delegate に対する応答かわかんなくね…？とか

実行時は基本的に、katari protocol によって動く

katari protocol に従って、それぞれのサーバーは以下のエンドポイントを何かしらの katali base URL (これはサーバーの base url ではない) 以下に提供する。これは runtime も例外ではない。
この意味で、runtime は katari protocol を実装したサーバーの一つであると言える。(他には、AI Server, discord wrapper server, cron server, sandbox server 等がある。)

サーバーには以下の種類がある。

- runtime: katali 言語で記述した各 task が実行できるサーバー
- cron: cron task, 及びその他日付に関する task を提供するサーバー
- AI Server: AI に関する task を提供するサーバー
- discord wrapper server: Discord に関する task を提供するサーバー
- sandbox server: セキュアな環境でのコマンド実行に関する task を提供するサーバー
- entry point server:
  - 動作としては、1. ユーザーからの task 実行依頼 (katali protocol 外からの通信) について root execution として entry point server 内で execution を実行し、その子プロセスとして他のプロセスを /POST execution により実行する。(大体の場合 runtime の task を実行することになる)
  - 最終的な return / error 結果が帰ってくるまで待機して、どこかに保持しておく。

これらが katali protocol に従ってやりとりすることで動作する。

意味

task ... それぞれのサーバーが提供する機能。effect が発生する可能性もある。
task execution ... task の実行インスタンス。handler を提供する実態でもある。task execution の中見は特に指定されておらず、サーバーごとに異なっていて良い。runtime では、並列実行もあるので内部で同時に複数の場所を動かしている可能性もあることに注意。仮想的には "動いているそれぞれのエージェント" である。エージェントは子エージェントを召喚したり、子からの delegation を処理したりする。
delegate ... 子 execution が親 execution に対して effect の handle を要請すること。
effect ... effect 定義それ自体。これはただの型情報と effect の一意な識別のために存在する。

execution に関しては、次の 3 つの制御を投げられる。
/return ... 実行した子 execution からの結果を受け取る
/continue ... 実行した handler delegation からの結果を受け取る
/break ... 実行した handler delegation からの break を受け取る

それぞれの control は好きなタイミングで飛んでくる可能性があることに注意。すなわち、子プロセスを実行中、/delegate が実行され、その delegation を実行中に別の /delegate が飛んでくることもある。その場合の処理は各サーバーに任せられる。runtime の場合は、同じ handler に対する delegate の場合は、前の物を待ち、別の handler だった場合は並列に実行する。

重要な点 : それぞれ id と URL (どのサーバーに位置するか) で管理される。

基本的な動き

1. 各サーバーは POST /execution を待機。
2. 受け取ったとき、execution を実行して、その id を返す。
3. 内部で別の execution を実行したいときは、他のサーバー (若しくは自分自身のサーバー) の POST /execution に POST する。このとき、handler を提供することもできる。
   handler としては、上層から受け取った handler + 自身で提供する handler を提供。execution を実行したら、(並列実行出ない限り) 停止し、return / delegate POST を待つ。以下の3つを待機する。
   - POST /delegate を待機。呼び出されたら handler を実行。その中で、さらに execution を呼び出したかったら、同様に POST /execution。この時 from_execution_id は **handler が位置する execution** の id になることに注意。また、提供する handler も上記と同様のルール。
     - handler の実行が終わったら、結果を continue するか、break するかを選ぶ。continue する場合は POST /execution/continue に、break する場合は POST /execution/break に POST する。
   - POST /execution/return を待機。呼び出されたら、結果を受け取って、実行を再開する。
   - POST /execution/error を待機。呼び出されたら、エラーを受け取って、実行を停止する。親にエラーを伝えるために、POST /execution/error に POST する。
4. 内部で effect を呼び出したいときは、提供された handler に該当 effect がある場合は、それを提供する execution_id と URL に対して /delegate を呼び出す。ない場合はエラー。(プロトコルに従い親にエラーを伝播。)。
   - continue されたときの動作 … POST /execution/continue を待機。呼び出されたら、結果を受け取って、実行を再開する。
   - break されたときの動作 … POST /execution/break を待機。呼び出されたら、実行を停止する。これが子プロセスからの delegation の最中であった場合は、その子プロセスに対しても break を即座に伝える。
     - 例:
       - execution A (effect X に対する handler を提供) -> execution B (effect Y に対する handler を提供) -> execution C と階層がある。この時 C が動いていて、A, B は停止している。C が B に対して effect Y の実行を delegation し、また、B はその中で A に対して effect X の実行を delegation したとする。このとき、B, C は親からの continue / break 待機中で、A は delegation を実行中である。A が break を発行した場合、B はそれを受け取るが、B も C からの delegation 中であるため、B は C にも break を投げる。こうしてすべてのプロセスが停止する。
5. 実行が終わったら、結果を親に伝えるために POST /execution/return に POST する。エラーになった場合は、POST /execution/error に POST する。

### 並列実行

並列実行のための構文を用意。

expression の一種。戻り値は、それぞれの戻り値の union の配列。parallel の右辺は構文であって配列ではない。

```
parallel [
  {
    s
    ...
    e,
  },
  {
    s
    ...
    e,
  },
  ...
]
```

parallel 実行時の動作。まず、内部的にはそれぞれの block は独立した task (environment を変数として受け取る) として扱われる。また、parallel 内では return / continue / break の文脈は切られる (そのため単純に分けて保存するだけで良い。) たとえば、parallel 内で return すると fn を抜けるのではなく、そのプロセスが終わるだけ (parallel に渡す block は fn の body と同様の性質を持つと言ってよい。) また、 continue / break を呼び出すとエラーである (型システムも上手く作ってください。)

parallel が実行されると、それぞれの task に対して同時に POST /execution する。これらは全て子プロセスとして実行される。

それぞれの execution が return するのを待ち、全て return したら、結果を配列にして返す

また、execution がエラーになると、他の execution に対して /break を投げ、自身も終了。

execution 内で effect が発生し、いずれかの handler に渡される。ある handler について、delegation 実行中に別の effect 処理の要請が飛んでくることがあるが、その場合は queue に入れて順番に処理する。handler variable は共通のものを更新していくこととする。

### return / continue / break の動作

return は一番内側の fn / 若しくは parallel の 1 block を抜けます
handler の case 内で呼ばれた場合は、break してから return と同等の処理になります。

continue は一番内側の handler / for を抜けて次の処理を開始します。for, handler がネストしていた場合は必ず一番内側です。

break は一番内側の handler / for を抜けます。handler 内で呼ばれた場合は、handler の実行を停止、以降の delegation は無視、また、handler の結果として break に渡された値を返します。for, handler がネストしていた場合は必ず一番内側です。

提案してください。
なお、実行パフォーマンスは一切求めないので tail call とかの最適化はいりません。
あと、他 task 呼び出しは自身のエンドポイントに対して POST /execution する形で行うと良いと思います。effect の呼び出しも同様。
