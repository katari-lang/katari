# KATARI Language Specification - Intermediate Representation (IR)

## 概要

KATARI の IR は、task をコルーチンとして表現する。1 つの task の本体は 1 つのフラットな命令列 (flat sequence) として表現される。基本ブロック (basic block) への分割は行わず、制御フローは絶対オフセットジャンプで表現される。パフォーマンス最適化は行わず、永続化・分散実行を主目的とする。

## 設計目標

1. **1 task = 1 コルーチン**: 1 つの task の本体は単一のフラットな命令列。基本ブロックへの分割なし。
2. **永続化**: プロトコル境界 (call/par/request/reply) で実行状態をスナップショット可能。
3. **分散実行**: コルーチンの yield 点がプロトコル境界と対応するため、ネットワーク越しの中断・再開が容易。
4. **単純さ**: 最適化なし。tail call 最適化なし。

## par ブロックのコンパイル

`par` 式の各ブロックは**独立した task としてコンパイル**される。IR にはクロージャが存在しないため、par ブロックが参照する外側の `let` 変数 (free variables) は、合成 task の明示的な引数として渡される。

```
par [
  { let r = ask(q1); r ++ "!" },
  { transform(q2) }
]
```

上記は概念的に以下のように lower される:

```
// 合成 task #par_block_0(q1: string) -> string
//   let r = ask(q1)
//   r ++ "!"
// 合成 task #par_block_1(q2: string) -> string
//   transform(q2)
//
// 親 task の命令列内:
//   IPar dst, [(par_block_0, [v_q1]), (par_block_1, [v_q2])]
```

このため `par` の各ブロックは IR 上では呼び出し対象の TaskId と引数 VarId 列で表現される (`IPar` 命令)。各 par ブロック task はランタイムによって独立した子エージェントとして起動される。

## プログラム構造

コンパイラは全モジュールを 1 つの `IRModule` にマージして出力する。

```
IRModule
  ├─ ModuleName: string
  ├─ NameTable
  ├─ Constants: [ConstVal]
  ├─ RequestDefs: [RequestDef]
  │    ├─ RequestId: u32
  │    ├─ Name: string
  │    └─ From?: string            // external の場合 "server:name"
  └─ Tasks: [Task]
       ├─ TaskId: u32
       ├─ Name: string             // デバッグ用
       ├─ Params: [VarId]
       ├─ Body: [Instruction]      // 単一のフラット命令列、ブロック分割なし
       └─ HandleBlocks: [HandleBlock]
            ├─ HandlerId: u32
            ├─ StateVars: [VarId]                              // 状態変数の VarId リスト
            ├─ RequestCases: [(RequestId, [VarId], [Instruction])]  // (req, arg_vars, body)
            └─ ReturnCase?: (VarId, [Instruction])             // (input_var, body)
```

1 つの task は複数の HandleBlock を持つことができる (段階的なキャパビリティの追加)。各 HandleBlock は一意の HandlerId を持ち、Body 内の `IHandleBegin` / `IHandleEnd` 命令によってアクティベーション・デアクティベーションされる。

HandleBlock の状態変数の初期化式は、Body 内の `IHandleBegin` より前の位置に通常の命令としてコンパイルされる。RequestCase の各 body は独立した命令列で、arg_vars にランタイムが request の引数を充填する。ReturnCase の body は独立した命令列で、input_var にスコープの結果値が充填される。

## 識別子

| 識別子 | 型 | スコープ |
|--------|-----|---------|
| VarId | u32 | task ローカル |
| TaskId | u32 | グローバル |
| RequestId | u32 | グローバル |
| ConstId | u32 | モジュールローカル |
| HandlerId | u32 | task ローカル |

## NameTable

IR バイナリに同梱される名前テーブル。ランタイムでのデバッグ、永続化に使用。

```
NameTable
  vars:     Map<VarId, string>              // 変数名 (デバッグ用)
  tasks:    Map<TaskId, QualifiedName>      // task の修飾名 (デバッグ用)
  requests: Map<RequestId, QualifiedName>   // request の修飾名
```

## ランタイム値

IR レベルでの値は JSON 互換:

| 種類 | 説明 |
|------|------|
| Null | `null` |
| Integer | 任意精度整数 |
| Number | IEEE 754 f64 |
| Boolean | `true` / `false` |
| String | UTF-8 文字列 |
| Array | 値の配列 |
| Object | string → 値 のマップ |

**クロージャは存在しない** (匿名関数がないため)。関数参照は TaskId で行う。

## 定数プール

モジュールごとにリテラル値の定数プールを持つ。

```
Constant = IntConst(BigInt) | NumConst(f64) | StrConst(string) | BoolConst(bool) | NullConst
```

## 命令セット

命令は単一の命令列内で順次実行される。制御フロー命令も命令列に統合されている (ターミネータという概念はない)。

### 定数・移動

```
ILoadConst   dst: VarId, const: ConstId     // 定数プールから値をロード
ILoadNull    dst: VarId                      // null をロード
IMove        dst: VarId, src: VarId          // 値をコピー
```

### Object 操作

```
INewObject   dst: VarId, fields: [(ConstId, VarId)]   // object 構築 {key: value, ...}
IGetField    dst: VarId, obj: VarId, field: ConstId    // object.field
ISetField    obj: VarId, dst: VarId, field: ConstId, val: VarId  // 新しい object (field を更新)
IHasField    dst: VarId, obj: VarId, field: ConstId    // フィールドの存在チェック (boolean)
```

### Array 操作

```
INewArray    dst: VarId, elems: [VarId]       // 配列構築
IArrGet      dst: VarId, arr: VarId, idx: VarId  // arr[idx]
IArrLen      dst: VarId, arr: VarId           // 配列長
IArrPush     dst: VarId, arr: VarId, elem: VarId  // 新しい配列 (要素追加)
IArrSlice    dst: VarId, arr: VarId, from: VarId, to: VarId  // スライス
```

配列の結合は統合命令 `IConcat` (後述) で行う。ランタイムは lhs の実行時型 (array / string) に応じて分岐する。

### 算術演算 (動的型判定)

IR は `integer` / `number` を区別せず、単一の算術命令を発行する。ランタイムインタプリタが両オペランドの実行時型を見て `integer` 同士なら整数演算、それ以外なら number (f64) 演算を行う。静的型情報は IR には含まれない。

```
IAdd         dst: VarId, lhs: VarId, rhs: VarId   // +  (数値加算)
ISub         dst: VarId, lhs: VarId, rhs: VarId   // -
IMul         dst: VarId, lhs: VarId, rhs: VarId   // *
IDiv         dst: VarId, lhs: VarId, rhs: VarId   // 結果は常に Number (f64)
IMod         dst: VarId, lhs: VarId, rhs: VarId   // %
INeg         dst: VarId, src: VarId               // 単項マイナス
```

`IDiv` は真の除算 (結果は常に Number)。床除算は組み込み task `prim.div` を `ICall` で呼ぶ。

### 比較

```
ICmpEq       dst: VarId, lhs: VarId, rhs: VarId   // ==
ICmpNe       dst: VarId, lhs: VarId, rhs: VarId   // !=
ICmpLt       dst: VarId, lhs: VarId, rhs: VarId   // <
ICmpLe       dst: VarId, lhs: VarId, rhs: VarId   // <=
ICmpGt       dst: VarId, lhs: VarId, rhs: VarId   // >
ICmpGe       dst: VarId, lhs: VarId, rhs: VarId   // >=
```

### 論理演算

```
IAnd         dst: VarId, lhs: VarId, rhs: VarId
IOr          dst: VarId, lhs: VarId, rhs: VarId
INot         dst: VarId, src: VarId
```

### 文字列・配列結合

```
IConcat      dst: VarId, lhs: VarId, rhs: VarId   // 結合 (ランタイムが lhs の型で分岐)
```

`lhs` の実行時型が `string` の場合は文字列結合、`array` の場合は配列結合となる。どちらも新しい値を返す純粋命令。

### 型変換

```
IToString    dst: VarId, src: VarId      // to_string (純粋)
ITypeOf      dst: VarId, src: VarId      // ランタイム型タグを返す
                                          // "null"|"integer"|"number"|"boolean"|"string"|"array"|"object"
```

`integer → number` への明示変換命令は存在しない。ランタイムは必要に応じて動的にプロモートする。

## 制御フロー

基本ブロックが存在しないため、制御フローは命令列に統合されている。オフセットは命令列内の絶対位置を指す。

### 基本制御

```
IJump        target_offset: u32                                         // 無条件ジャンプ
IBranch      cond: VarId, true_offset: u32, false_offset: u32          // 条件分岐
ISwitch      scrut: VarId, cases: [(ConstId, u32)], default_offset: u32 // switch
IReturn      val: VarId                                                 // task から戻る
```

### Agent 操作

```
ICall        dst: VarId, task_id: TaskId, args: [VarId]         // 子 agent を生成し完了を待機 (コルーチンはここで yield)
IPar         dst: VarId, tasks: [(TaskId, [VarId])]             // 複数の子 agent を並行生成し全完了を待機 (array を返す)
IRequest     dst: VarId, request_id: RequestId, args: [VarId]   // request を発行 (コルーチンはここで yield、reply を待機)
```

agent_id は IR レベルでは隠蔽される (ICall は完了結果を、IPar は結果配列を返す)。

`ICall` および `IPar` は同期サスペンションポイントであり、子エージェントの完了を待機してから次の命令に進む。したがって `IReturn` や body 末尾に通常到達した時点では、子エージェントは既に全て join 済みであり、生存している子は存在しない。

例外は **`IBreak` による中断**である。`IBreak` は現在 `WaitingCall` / `WaitingPar` 状態で待機中の body を強制的に中断するため、待機対象となっていた子エージェントが宙に浮く。この場合のみ、ランタイムが該当する handler スコープ内で生成された生存中の子エージェントに terminate を送信する。terminate は IR 命令ではなく**ランタイムが暗黙的に行う**。ランタイムは `AgentState.children` および handler スコープ情報から terminate 対象を判定する。

### Handle Block ライフサイクル

```
IHandleBegin handler_id: HandlerId                                  // handle block をアクティベートする (handle 文の実行到達時)
IHandleEnd   dst: VarId, scope_result: VarId, handler_id: HandlerId // handle block をデアクティベートする (レキシカルスコープ離脱時)
```

これらの命令は Body の命令列内の適切な位置に配置される。`IHandleBegin` により handler が `active_handlers` スタックに追加され、`IHandleEnd` により除去される。`IHandleEnd` は handle スコープの結果値 (`scope_result`) を `dst` にコピーする。

### Handler 内命令

```
IReply       val: VarId, handler_id: HandlerId, var_updates: [(u32, VarId)]  // handler 関数内: request に応答する。handler_id で指定された handler の状態変数を更新
IBreak       val: VarId, handler_id: HandlerId                               // handler 関数内: 指定された handler スコープ全体を中断する
```

### For ループ内命令

```
INext        var_updates: [(u32, VarId)]                   // for ループ body 内: 次のイテレーションへ進む
IForBreak    val: VarId                                    // for ループ body 内: for ループを抜ける
```

## コルーチンモデル

各 agent は 1 つのコルーチンとして実行される。yield 点は以下の命令で発生する:

- **ICall**: 子 agent の完了まで中断
- **IPar**: 全子 agent の完了まで中断
- **IRequest**: handler が reply するか、agent が terminate されるまで中断

yield 点はプロトコル境界と一致するため、ネットワーク越しの中断・再開が自然に行える。

### 協調イベントループ

ランタイムは協調的なイベントループモデルを採用する。suspension point (ICall, IPar) に到達した際に request queue のチェックが行われる。エージェントは**完全に非再入的**であり、同時に実行されるハンドラは常に最大 1 つ。request は FIFO キューで待機し、suspension point で 1 件ずつ逐次処理される。並行実行は par ブロック (独立した子エージェント) によってのみ実現される。

**転送中の非再入性**: エージェントが `IRequest` を発行し、直接の親が handler を持たず転送が発生した場合も、転送完了まで非再入状態が維持される。

### Agent 状態

ランタイムは各 agent の状態を以下の構造で管理する:

```
AgentState = {
  agent_id:            string,
  parent_agent_id:     string,          // 直接の親 agent の識別子
  parent_agent_where:  URL,             // 直接の親 agent の URL
  task_id:             TaskId,
  instruction_pointer: u32,
  vars:                Map<VarId, Value>,
  handler_states:      Map<HandlerId, Map<VarId, Value>>,  // handle block ごとの状態変数
  active_handlers:     Vec<HandlerId>,                      // 現在アクティブな handler のスタック
  request_queue:       Queue<PendingRequest>,
  children:            Set<AgentId>,
  status:              Running
                     | WaitingCall(AgentId)      // 同期タスク呼び出し待機
                     | WaitingPar([AgentId])     // par 完了待機
                     | WaitingReply(RequestId)   // request の reply 待機
                     | Terminated
}
```

## Handler 実行

request が到着した際の処理フロー:

1. agent が suspension point に到達した際、`request_queue` を確認する。
2. request がキューにあれば、`active_handlers` を内側 (最新) から順に検索し、該当する handler を探す。
3. 対応する handler が見つかった場合:
   a. `handler_states[handler_id]` から handler の現在の状態変数を取得する。
   b. 対応する handler 関数 (独立してコンパイルされた命令列) を実行する。
   c. `IReply` の場合: 状態変数を更新し、`from_agent_where` に reply を送信する。
   d. `IBreak` の場合: handler スコープの全子 agent に terminate を送信し、break 値を返す。
4. 対応する handler が見つからなかった場合 (プロキシモード):
   a. agent は `WaitingReply(request_id)` 状態となり、非再入状態になる。
   b. 親 agent (`parent_agent_where`) に `POST /agent/request` を転送する。
   c. 親から `POST /agent/reply` が届いたら、元の request 発行元 (`from_agent_where`) に reply を転送する。

## 永続化

コルーチンの yield 点 (ICall, IPar, IRequest) で実行状態をシリアライズできる。`AgentState` を JSON 形式に変換し、PostgreSQL 等のデータベースに保存する。サーバー再起動後に状態を復元し、実行を再開できる。

## バイナリフォーマット

IR はバイナリ形式でシリアライズされる。

### ヘッダ

```
magic:   4 bytes = "KTRI"
version: 2 bytes = (major, minor)
```

### エンコーディング

- 整数: LEB128 可変長エンコーディング
- 文字列: 長さプレフィックス + UTF-8 バイト列
- 配列: 長さプレフィックス + 要素列

### モジュール構造

```
Header: "KTRI" (4 bytes) + version (2 bytes)
ModuleName: text
Constants: vec<ConstVal>
RequestDefs: vec<RequestDef>
Tasks: vec<Task>
```
