## Syntx.hs

### Module data 型の説明。

以下の定義に置いて、modName と modFile は Syntactic には存在しない (どちらも cli によって解決される。) Syntax モジュールにコレを置くことは許容するが、コメントで構文要素ではないことは明示すべき。

```
data Module = Module
  { modFile :: FilePath,
    modName :: Text, -- e.g. "lib.cron" (dot-separated qualified name)
    modDecls :: [Decl]
  }
  deriving (Show)
```

### type 定義とか request 定義などの、param が抽象化されていない。

```
-- request name(params) -> RetType
data RequestDecl = RequestDecl
  { reqAnnot :: Maybe Text,
    reqName :: Text,
    reqParams :: [(Text, Type, Maybe Text)], -- (name, type, annotation)
    reqRet :: Type
  }
  deriving (Show)
```

tuple がコンテキスト外に飛んで行った時、name, type, annotation という関係がわからなくなる。コレは data として切り出すべき。(type 定義や external task 定義などのp共通したパターンにも同じものが使えそう。)

### 機能変更 (No-return): すべての return / handle break / for break / for next / for finally / for の廃止と機能変更

まず、handle break 以外に言及。

すべての return 文 / for の break / for の next / for の finally は request システムで再現可能なので、廃止します。コレらの control operator は block の分解を複雑にします。また、この変更により祖結合性を保てます。for 文は単にループするだけ。finally も handle の return clause で対応可能です。

さらに、handle の break も廃止して、一点機能変更

- 現在は handler の request clause が終了すると、それは reply として扱われると思います。
- コレを、break として扱うように変更します。処理を返すには明示的な reply 呼び出しが必要とします。
- この変更によって、control operator は reply with のみだけになります。

ついでに reply with の spec を詳細に記述します。

- block には "reply with" が許可される block と許可されない block の二種類のみが存在します。それぞれ以下のように導出されます。
- task 定義の block は許可されない。
- handle の request clause の block 内では、上位 block がなんであっても、許可される。
- handle の return clause の block 内での許可は、上位 block に依存する。handle return calause 内での reply は、その handle ではなく、一層上の handle の request の結果として処理される。
- for の block 内での許可は上位 block に依存する。reply が発火すると、即座に for 文を抜け、値を返す。
- block 式の block 内での許可は上位 block に依存する。
- match / if 式の block 内での許可は上位 block に依存する。
- par block 内での許可は上位 block に依存する。
- その block の top level として reply with が呼び出されている block の値としての型は never とする。

これに伴って、IR と Lower の大幅な簡略化(全面変更)も可能だと思います。以下、この機能変更は No-return としてメンションします。

糖衣構文はそのうち用意すると思いますが、今は一旦なるべく最小にする方針でいきます。
