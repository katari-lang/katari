## 構文変更 (用語統一)

for と handle の用語を揃えます。また、その他色々変更。

まず、task / agent を task → agent, agent → instance に変更。agent 定義とその instance 的な　全ての場所で変更してください。 (protocol, syntax など。)

handle 式の改善

```
handle (var x = init, var y = init2) { // var がないときは丸括弧省略可能
  request foo(..) => {
    ...
    continue e with {
      x = newX,
      y = newY
    }
  }
  request bar(..) => {
    ...
    break
  }
} then (r) {
  ... // 以前の return clause をここに移行
}
```

for 式の改善

```
let z = for (let x of iterable, var state = init, var state2 = init2) {
  ...
  continue with {
    state = newState,
    state2 = newState2
  }
  ...
  break value
} then {
  ...
}
```

↑ほとんど同様の概念なので用語も統一。 break / continue にする。 handle と for の break / continue は scope が分かれているので、これまで通りコンパイル時には handle_break, handle_continue, for_break, for_continue などに分ける必要がある。

また、handle の continue は値を取るが、for の continue はとらない。また、handle の then は値を取るが���for の then は取らない。

## 機能変更 (par の spec)

par 内でも return が書けるようにする。その他も含め、次に control statement と block の組み合わせでどう動作するかを全てまとめる。これらはあくまで semantics であって、IR の実装をどうするかは別途決める必要がある。特に par 内での control statement の実装は難しい。IR にて、現在の instance の id を par のそれぞの task に内部的に渡し、return が発火したら例外的に親の instance から terminate を送る、等が考えられる。

IR と Lowering の実装は大幅に変えて良い。発想を変えて、これらの control operator が **最も自然に** 実装できる方式を発見してください。後方互換性やコードの最小変更は気にしなくていいです。

以下、"block 式" は次の block を指す。これらの block は全て同一の動きをするので、まとめる。
通常の block 式 (let x = {...} といった形式の物), if, match, for の then clause, handle の then clause

使用可能…… OK、使用不可…… NG として、その後に説明を加える。

- return
  return をする block は常に never 型として扱われる。return が属する task 定義の返り値は、その body の最終行の値と、return の右辺の型をそれぞれのコンテキストにおける上位 handler の then clause 全て逆順に適用したもの、を全て union したものである。(この then における型変更の際は、then の引数との型チェックが必要。)
  - block 式
    その block を囲む最も内側の block での return の動作と同様の動作をする必要がある。(たとえば、request clause 内のネストした block 式であったなら、request clause の return と同様の動きをする。)
  - function body
    OK. 現在実行中の位置からもっとも内側の "コード上での" task 定義に対して、その task 定義の結果を return の右辺とするが、handle context がある場合、それら全ての then clause を適用する必要がある。例:
    ```
    task foo() {
      handle {
      } then (r) {
        r + 1
      }
      let x = if (true) {
        // ここは block 式なので function body の動作を受け継ぐ。
        // handle は block に所属する。
        handle {
        } then (r) {
          r * 2
        }
        return 3 // return → 3 を返すが、さらに外側の then も順次適用される。ここでは 3 * 2 + 1 = 7 が最終的な返り値になる。
      } else {
        handle {
        } then (r) {
          r / 4
        }
        return 5
      }
      return x + 1000 // ここは実際には実行されない。 if式の中身が return しているため。
    }
    ```
  - request handler clause
    OK. break とほとんど同じ動きをするが、break はその handle が書かれた block の返値とするのに対し (これは後述)、return は task 定義までさかのぼる。違いは後述する。
  - for の body
    OK. for の body で return が呼ばれた場合、以降の for 実行はすっ飛ばして、task まで巻き戻る。この際 handle の then を適用する。
  - par のそれぞれの block
    OK. par の block で return が呼ばれた場合、以降の block の実行はすっ飛ばして、task まで巻き戻る。この際、handle の then を適用する他、par で起動された他の instance は terminate される。
- handle_continue
  handler_continue が属する block は全て never 型として扱われる。handler_continue に渡された値の型は、その handler_continue が属する request handler clause の返値の型となり、これは request 宣言と照らし合わせて型チェックをする必要がある。
  - block 式
    その block を囲む最も内側の block での handle_continue の動作と同様の動きをする必要がある。
  - function body
    - 許可されない。
  - request handler clause
    OK. この request の応答結果として値を request 元に返す。
  - for の body
    - 許可されない。for の continue で上書きされるので。
  - par のそれぞれの block
    OK/NGは親を引き継ぐ. この par が属する最も内側の request handler clause の返値の型とする。par 内で continue した場合は、par で起動したすべての task は terminate される。
- handle_break
  handle_break が属する block は全て never 型として扱われる。handle_break に渡された値の型は、その handle が属する block の返値の型となる。(同 block に handle が上位にある場合、その then clause の型変更が適用される。この型変更の際は型チェックが必要。)
  - block 式
    その block を囲む最も内側の block での handle_break の動作と同様の動きをする必要がある。
  - function body
    - 許可されない。
  - request handler clause
    OK. この request handler clause が所属する handle が所属する block の返値とする。同じ block のより上位に handle があれば、それも適用する。
    ```
    task foo () {
      handle {
      } then (r) {
        r * 10 // この block の以下の式の結果に対して * 10
      }
      let x = {
        handle {
        } then (r) {
          r + 1 // この block の以下の式の結果に対して + 1
        }
        handle {
          request bar(v) => {
            break v * 3
          }
        } then (r) {
          r * 40 // ここは実行されない。
        }
        bar(100) // bar request を飛ばす → これは上の handler でキャッチされる → break で 100 * 3 = 300 を返す → その上の then clause が適用されて 300 + 1 = 301 が x に入る。
      }
      x + 2 // → 303 となるが、 then clause が適用され 303 * 10 = 3030 となる。
    }
    ```
    こちらは、handle が所属する block の返値とするのみで、return と違い task 定義までさかのぼらないことに注意。次は return の例。
    ```
    task foo () {
      handle {
      } then (r) {
        r * 10 // この block の以下の式の結果に対して * 10
      }
      let x = {
        handle {
        } then (r) {
          r + 1 // この block の以下の式の結果に対して + 1
        }
        handle {
          request bar(v) => {
            return v * 3
          }
        } then (r) {
          r * 40 // ここは実行されない。
        }
        bar(100) // bar request を飛ばす → これは上の handler でキャッチされる → return で 100 * 3 = 300 を返す (ここで return を使用したので、task 定義まで巻き戻ることが期待される。) →　コンテキストに適用された then clause が逆順で実行される → 300 + 1 = 301 → 301 * 10 = 3010 が最終的な返り値になる。
      }
      x + 2 // ここは実行されない。
    }
    ```
  - for の body
    - 許可されない。for の continue で上書きされるため。
  - par のそれぞれの block
    OK/NGは親を引き継ぐ. この par が属する最も内側の request handler clause が属する handle の戻り値となる。break が呼ばれた場合、par で起動したすべての task は terminate される。
- for_continue
  - block 式
    その block を囲む最も内側の block での for_continue の動作と同様の動きをする必要がある。
  - function body
    - 許可されない。
  - request handler clause
    - 許可されない。request handler clause の continue で上書きされるため。
  - for の body
    OK. 次のイテレーションに進む。
  - par のそれぞれの block
    OK/NGは親を引き継ぐ. この par が属する最も内側の for の動作をする。for_continue が呼ばれた場合、par で起動したすべての task は terminate される。
- for_break
  - block 式
    その block を囲む最も内側の block での for_break の動作と同様の動きをする必要がある。
  - function body
    - 許可されない。
  - request handler clause
    - 許可されない。request handler clause の break で上書きされるため。
  - for の body
    Ok. for を終了する。for 式の戻り値として採用される。
  - par のそれぞれの block
    OK/NGは親を引き継ぐ. この par が属する最も内側の for の動作をする。for_break が呼ばれた場合、par で起動したすべての task は terminate される。

すべての block について、control operator が存在しないなら、末尾の値が戻り値となる。ただし、末尾が単体式でない場合 (handle 文や let 代入の場合) は null が補完される。つまり `{handle {...}}` → `{handle {...}; null}`、`{let x = e}` → `{let x = e; null}` として扱う。toplevel に存在するなら never になる。(ネストした位置では変わらず) その型は、その block 内 toplevel に存在するすべての handle の then clause を逆順に適用したものになる。これらの then clause の適用の際は、型チェックが必要。

## Module.hs

### resolveTypeAlias と Types.normalize の重複

`resolveTypeAlias` (L449-502) と `Types.normalize` (Types.hs L181-219) はほぼ同一のロジックを持つ。両方とも `Type → NormalizedType` の変換を行い、`TNull/TBoolean/...` の直接変換、`TArray` の再帰、`TUnion` の `foldr unionNT`、`TInter` の `foldr intersectNT`、`TObj` の `Map.fromList + tryMakeDISC` が全て重複。

唯一の差分は `TAlias` の解決:

- `resolveTypeAlias`: `geTypeEnv` → `geTypes` (未解決) → 再帰解決 (サイクル検出付き)
- `normalize`: `geTypeEnv` (解決済み) のみ参照

`resolveTypeAlias` が存在する理由は、Pass 1 で型エイリアスを `geTypeEnv` に登録する段階で `geTypeEnv` が不完全なため。この設計を見直し、型エイリアスの解決を独立したステップとして先に全て完了させ、`geTypeEnv` 構築後は `normalize` だけを使う方式に変更すべき。

具体的には: Pass 1 を 2 段階に分ける。

1. 全モジュールの全型エイリアスを `geTypes` に登録
2. 型エイリアスを依存順に解決して `geTypeEnv` を完成させる (サイクル検出はここで行う)
3. その後、task/request/val の登録時には `normalize` を使う

これにより `resolveTypeAlias` を廃止でき、unknown alias → never の問題、TObj の同名フィールドマージの問題も `normalize` 側 (Types.hs) で一箇所だけ修正すれば済む。

### ValDecl の normalize が不完全な geTypeEnv を使用 (L354)

上記の重複問題と同根。Pass 1 の `registerDecl` で `DeclVal` の型を `normalize` する際、`geTypeEnv` にはまだ全ての型エイリアスが登録されていない可能性がある。同モジュール内で後に定義された型エイリアスを参照する val は、unknown alias として ntNever になる。上記のパス再設計で解決。

## Lowering.hs

`lowerFinalStmt` で `SHandle` は `ILoadNull` を返すだけで、handle のパラメータやハンドラを一切 lower しない。仕様上、block の末尾が単体式でない場合 (handle 文や let 代入) は null が補完される (`{handle {...}}` → `{handle {...}; null}`)。従って `lowerFinalStmt` ではなく `lowerNonFinalStmt` 相当の処理 + 末尾 null を行うべき。

### 未知変数の lenient 処理 (L473-475)

EVar で環境にない変数を `ILoadNull` でフォールバックしている。型チェッカーが事前にエラーを出すことを前提としているが、コメントの通り lenient な動作。理想的にはエラーにすべきだが、型チェッカーとの二重チェックになるため現状維持でも可。

### ECall の不明な callee の処理 (L531-534)

task でも request でもない名前への呼び出しが `ILoadNull` になる。同上。

### par block の合成タスク名が非一意 (L897)

`__par_` + キャプチャ数で名前を生成しているため、同数のキャプチャを持つ複数の par ブロックで名前が衝突する。カウンタベースのユニーク名にすべき。

### for の break が then (旧 finally) をスキップする (L303-305, L719-720)

現在の実装では `SForBreak` が `lbl_after` にジャンプし、then ブロックをスキップする。仕様上、for の break は then を飛ばすのが正しい動作。現在の実装は正しい。

### 定数プールの重複チェックが O(n) (L1097-1108)

`addConst` で逆順リストを線形走査している。Map による O(log n) ルックアップに変更可能。

### task → agent のリネーム

`lowerTask`, `allocTask`, `lookupTask`, `lsTaskIds`, `lsTasks`, `IRTask` 等、全箇所のリネーム。

### 構文変更に伴う命令生成の変更

`SReply` → handle 用 `continue`、`SNext` → for 用 `continue` への変更に追従。`SBreak` と `SForBreak` の意味は変わらないが、IR 命令名は変わる。

### IR/Lowering の大幅再設計

「IR と Lowering の実装は大幅に変えて良い」に従い、以下を検討:

1. par 内の return/break/continue のサポート
2. then clause のセマンティクス実装 (return 時に then chain を実行)
3. handle/for の制御フロー統一

これらは現在の flat instruction + label jump 方式では実装が困難な可能性がある。structured control flow (基本ブロック + 終端命令) への移行を検討すべき。
